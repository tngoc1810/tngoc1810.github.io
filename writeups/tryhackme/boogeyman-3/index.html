Boogeyman 3: SOC Investigation Writeup

This writeup documents my investigation of the TryHackMe Boogeyman 3 room. I followed the evidence like a SOC analyst: understand the alert, set a hypothesis, pivot across logs, validate each step, and build a complete attack timeline.

ElasticKibanaSysmonPowerShellWinRMDFIRIncident Response

Overview

Quick Logistics LLC had already been targeted by the Boogeyman threat group. In this case, the attacker returned and targeted the CEO with a financial-themed email. The message looked like a normal business request, but the attachment was not a normal document. It used a file name that looked like a PDF, while the real behavior showed a mounted-image and HTA-based execution chain.

The investigation expanded from one user workstation to a wider compromise path: payload execution, local staging, persistence, outbound communication, privilege activity, credential abuse, remote share access, lateral movement to WKSTN-1327, activity on DC01.quicklogistics.org, and final impact preparation.

SOC mindset: I did not start by guessing the whole attack. I started from one reported email, then used logs to prove each next step. The key was to connect parent process, command line, timestamp, user, host, and network fields together.

Methodology

  1. Scope the investigation window. I used the scenario time range to avoid drowning in unrelated logs.
  2. Choose useful fields. I added process name, parent process, command line, event ID, PowerShell script block text, PID, and parent PID.
  3. Start from the known artifact. The suspicious file name became the first search pivot.
  4. Follow parent-child relationships. This showed how the first payload launched later processes.
  5. Pivot into network and identity events. Once execution was confirmed, I searched for outbound connections and credential-related activity.
  6. Build a timeline. I connected each event into one chain instead of treating logs as separate answers.

Step-by-step analysis

1. Phishing email: initial report

The first screenshot shows the reported phishing email. The subject and body used urgency around a financial document. This matters because business-pressure lures are designed to reduce caution. From a SOC perspective, the email gives context, but it does not prove compromise by itself.

What I noticed:
  • The message used an urgent finance theme, which is a common way to pressure executives.
  • The sender appeared to be internal, making the email more believable to the victim.
  • The body was short and generic, which often appears in phishing because the real goal is to make the user open the attachment.
How I used it:
  • I treated this as the starting point, then moved to endpoint logs to prove whether the attachment executed anything.

2. Attachment mismatch: fake PDF behavior

The file in Downloads had a financial PDF-looking name. However, Windows showed it as a disc image. This was the first technical sign that the visible filename was misleading. A legitimate PDF should not mount like an image file.

What I noticed:
  • The filename looked like a PDF report, but the Type field showed a disc image.
  • This mismatch suggested masquerading: the attacker wanted the victim to trust the name, not the file type.
  • The file was in Downloads, matching the scenario that the user opened an email attachment.
Next thought:
  • If this is a mounted image, I need to inspect what file inside the image actually executed.

Inside the mounted image, the file still looked like a PDF by name, but the type was HTML Application. This was the bridge between the email lure and script execution. The attacker used a business filename to hide an executable HTA payload.

What I noticed:
  • The file inside the mounted drive used a double-extension style name.
  • The visible name still looked like a document, but the Type field showed HTML Application.
  • HTA files are script-capable and can be executed by Windows through mshta.exe.
Hypothesis:
  • The user likely opened this fake PDF and triggered mshta-based execution.

3. Scoping the investigation in Kibana

I set the time range to cover August 29 and August 30, 2023. This was important because the scenario already gave the suspected window. A good time filter reduces noise and keeps the investigation focused.

Next, I selected columns that support process-tree analysis. Parent process and command line fields are especially important because many Windows binaries are legitimate by themselves. The suspicious part is often the parent process, argument, file path, or timing.

Why these fields matter:
  • process.name tells me what ran.
  • process.parent.name tells me who launched it.
  • process.command_line tells me the real intent.
  • process.pid and process.parent.pid allow accurate pivoting when multiple processes have the same name.

4. Confirming initial execution

The first clear execution event showed File Explorer launching the HTA handler from the mounted image. This strongly suggested user-triggered execution: the victim opened the fake document through the graphical shell, and Windows used the HTA handler to run it.

What I noticed:
  • The parent process was File Explorer, which fits manual user execution.
  • The child process was the Windows HTA handler.
  • The command line pointed back to the mounted drive and fake PDF/HTA file.
Conclusion:
  • This confirmed that the phishing attachment led to local code execution.

5. Stage 1 payload staging

The next event showed a copy operation from the mounted image to a local user path. This looked like payload staging. The attacker likely wanted the next file to live on the local filesystem instead of only inside the mounted image.

What I noticed:
  • The parent was the HTA execution process, so this copy action belonged to the attack chain.
  • The source was the mounted drive and the destination was a user-writable temporary location.
  • The file name looked harmless, but it was being staged by the payload.
SOC interpretation:
  • This was not normal document behavior. A financial document should not copy payload-like files into Temp.

Shortly after the copy, the staged file was executed through a trusted Windows binary. The file extension looked like data, but the behavior showed it was being used as executable content.

Why this mattered:
  • The staged file was executed through a trusted Windows binary instead of opened as normal data.
  • This showed the file was part of the next payload stage.
  • The parent process still tied back to the original HTA execution.

6. Persistence through scheduled task

The attacker then created a scheduled task named Review. The name matched the document-review theme, which helped it blend in. Functionally, this was persistence: the payload could run again after the original file execution.

The expanded event made the task name and action easier to see. This is why expanding important Kibana events matters: the table view may hide the most useful part of a command line.

What I concluded:
  • The task name was designed to sound normal.
  • The task action pointed back to the staged payload.
  • This moved the incident from one-time execution into persistence.

7. Network pivot after payload execution

Once execution and persistence were confirmed, I pivoted to network fields. The destination IP summary showed one external IP dominating the results. Frequency alone is not proof of malicious activity, but in this timeline it became important because it appeared after the payload execution.

The detailed network view showed repeated outbound HTTP traffic from the victim machine to that endpoint. Because this traffic was tied to suspicious processes, I treated it as likely C2 or staging communication.

SOC reasoning:
  • I did not mark every external IP as malicious.
  • I focused on the destination tied to the process chain and timing.
  • This helped connect local payload execution to outbound infrastructure.

8. UAC bypass pattern and hidden PowerShell

Later, I found hidden PowerShell launched with a suspicious parent process. PowerShell is common in Windows administration, but hidden execution combined with this parent process and the earlier payload chain made it suspicious.

The parent-child relationship made the event more meaningful. The parent process was associated with a known UAC bypass pattern, so I interpreted this as an attempt to run commands with elevated context.

I validated the behavior with quick research. In a real SOC workflow, this is a normal step when a built-in Windows binary is used in an unusual way.

What made this suspicious:
  • The PowerShell window was hidden.
  • The parent process was not a normal admin console.
  • The command appeared after the attacker had already established execution and persistence.

9. Encoded PowerShell stage

The next PowerShell command used encoded content. Encoded PowerShell is not automatically malicious, but here it followed phishing execution, payload staging, persistence, network traffic, and privilege activity. The surrounding context made it highly suspicious.

How I interpreted it:
  • Encoding reduced readability in the log view.
  • The command used hidden/non-interactive style execution.
  • Because it appeared after elevation-related activity, I treated it as a later-stage payload rather than normal administration.

10. Identity-focused activity

After the privilege activity, the attacker downloaded tooling associated with credential access. This changed my investigation focus: I was no longer only tracking execution. I now needed to track user accounts, authentication context, and remote access.

Next question:
  • Did the attacker only download the tool, or did they use it to access another machine?

The following events showed credential material being reused. This was the bridge from credential access to lateral movement. A SOC analyst should always ask: if new credentials were obtained, where were they used next?

11. Remote share access and lateral movement

The attacker accessed a remote share and read an automation script. This was important because automation scripts often expose useful internal information. In this case, it helped explain the next movement step.

After reading the file, the attacker used newly discovered credentials to run a remote command on WKSTN-1327. This confirmed lateral movement attempt and showed the new target host.

On the second machine, the parent process was wsmprovhost.exe. This parent process is a strong clue for PowerShell Remoting / WinRM execution.

Why this confirmed movement:
  • The command targeted another hostname.
  • The target host executed PowerShell under a remoting-related parent process.
  • This connected the discovered credentials to successful remote execution.

12. Second host credential activity

On the second host, the attacker repeated the same pattern: gain access, perform identity-focused activity, and prepare for deeper access. This is a common enterprise intrusion cycle.

13. Domain controller stage

The attacker eventually reached DC01.quicklogistics.org. Activity on a domain controller is critical because it can affect the whole domain, not only one workstation.

Severity note: Once the domain controller is involved, this should be treated as a domain compromise incident.

14. Final impact staging

After the domain controller activity, the attacker downloaded a final binary associated with the impact stage. This was the last step observed in the chain and showed that the attacker was preparing to cause business impact.

Lessons learned

  • Do not trust visible filenames. Always check real file type and execution behavior.
  • Parent-child process relationships are one of the fastest ways to rebuild an attack chain.
  • Suspicious PowerShell becomes stronger evidence when connected to a suspicious parent process and timeline.
  • Remote share access can reveal the next step of lateral movement.
  • Domain controller activity should be treated as critical.

Conclusion

The Boogeyman 3 incident started with a single suspicious email but expanded into a full intrusion chain. By following evidence step by step, I connected the fake attachment, HTA execution, staged payload, persistence, network traffic, account activity, lateral movement, domain controller access, and final impact preparation into one clear timeline.

The main lesson was simple: do not analyze logs in isolation. Connect them, validate each pivot, and explain why every event matters.
