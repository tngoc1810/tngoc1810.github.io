HTB Sherlock: Reaper

A senior-style SOC investigation of a suspicious Windows logon alert where the source workstation name did not match the source IP address. I correlated Security.evtx with ntlmrelay.pcapng to prove NTLM relay behavior, identify the victim user, map hosts to IPs and connect Event ID 4624 with Event ID 5140 using the malicious Logon ID.

Investigation goal: Validate a SIEM alert for a suspicious successful logon where the workstation name and source IP address were inconsistent, then report the real source, victim account, file share access and malicious session details.

Scenario Overview

The lab provides two evidence sources: a Windows Security event log and a packet capture from the same incident window. The SIEM alert is not simply “a user logged in.” The important detail is that the IP address and source workstation name did not match.

That mismatch is a strong sign of authentication abuse. In an NTLM relay scenario, the target host may log the workstation name embedded in NTLM authentication metadata, while the actual TCP connection comes from the attacker-controlled relay device.

Senior SOC Mindset

First thought: Do not trust a single field in Event ID 4624. WorkstationName can describe the claimed workstation in the authentication metadata, while IpAddress shows the host that actually connected to the target. If these do not map to the same system, I need network evidence.

I approached the case as a correlation problem:

  1. Map hostnames to IP addresses from PCAP evidence.
  2. Find the NTLM authentication flow and identify the user whose authentication was captured or relayed.
  3. Locate the successful NTLM network logon in Security.evtx.
  4. Use the source port and Logon ID to connect the Windows log to the PCAP session.
  5. Use Event ID 5140 to confirm what share was accessed under the malicious session.

Host/IP Mapping

The first step was to map the internal hosts. Searching for FORELA-WKSTN001 in the capture showed NetBIOS/NBSTAT activity associated with 172.17.79.129. This became the target workstation in the malicious logon flow.

From the SMB/NTLM conversations, 172.17.79.136 acted as the victim workstation sending NTLM authentication for FORELA\arthur.kyle toward 172.17.79.135. That made 172.17.79.135 the unknown device used by the attacker to intercept or relay credentials.

Victim Share Navigation

The victim workstation tried to access a network share named \\DC01\Trip. In SMB, this appears as a Tree Connect Request. The server later returned STATUS_BAD_NETWORK_NAME, but the key point is that the user navigated to that share path.

Why this matters: \\DC01\Trip is a UNC path. It means “open the share named Trip on host DC01.” In this case, the attempted navigation triggered SMB authentication activity that the attacker could abuse.

Stolen NTLM Authentication

The capture shows 172.17.79.136 connecting to 172.17.79.135:445 and completing an NTLMSSP authentication exchange. The NTLMSSP_AUTH packet identifies the account as FORELA\arthur.kyle.

This explains the “hash stolen” question in the lab. The attacker did not need the plaintext password. The useful artifact was the NTLM authentication material, which could be intercepted or relayed.

Malicious Logon

The main Windows evidence is Event ID 4624, a successful logon. The event shows a network logon using NTLM:

TargetUserName: arthur.kyle
TargetDomainName: FORELA
LogonType: 3
LogonProcessName: NtLmSsp
AuthenticationPackageName: NTLM
WorkstationName: FORELA-WKSTN002
IpAddress: 172.17.79.135
IpPort: 40252
TargetLogonId: 0x64a799

This is the core mismatch: the event claims the workstation name is FORELA-WKSTN002, but the source IP is 172.17.79.135, which is the unknown attacker device, not the legitimate IP of FORELA-WKSTN002.

The PCAP validates the Windows event: 172.17.79.135:40252 connected to 172.17.79.129:445 and used NTLM authentication for FORELA\arthur.kyle.

Logon ID Correlation with Event ID 5140

After identifying the suspicious logon, I pivoted to Event ID 5140, which records network share access. The key field is SubjectLogonId. It matched the malicious 4624 Logon ID:

4624 TargetLogonId  = 0x64a799
5140 SubjectLogonId = 0x64a799

The share accessed as part of the malicious authenticated session was \\*\IPC$. This is why using the PCAP path \\172.17.79.129\IPC$ was not the exact answer for the task; the lab asked for the Windows event field from 5140.

Recommendations

  • Reset or rotate credentials for FORELA\arthur.kyle and invalidate active sessions where possible.
  • Investigate 172.17.79.135 as the attacker-controlled or unauthorized relay device.
  • Hunt for additional Event ID 4624 Logon Type 3 events where WorkstationName does not match the legitimate assigned IP address.
  • Correlate Event ID 4624 and 5140 by Logon ID to identify what resources were accessed after suspicious authentication.
  • Enforce SMB signing where practical and reduce NTLM exposure to limit NTLM relay risk.
  • Review name resolution paths such as NBNS/LLMNR and consider hardening or disabling legacy name resolution if not required.

Lessons Learned

The most important lesson from Reaper is that a successful logon is not automatically legitimate. The analyst must validate whether the source workstation identity and the source IP address describe the same system.

This lab also reinforces a practical SOC habit: correlate host logs with network traffic. Event ID 4624 gave the suspicious session. The PCAP proved where the traffic came from. Event ID 5140 showed what share was accessed under the same Logon ID. Together, they formed a complete and defensible incident story.
