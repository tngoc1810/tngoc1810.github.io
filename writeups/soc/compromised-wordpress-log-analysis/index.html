Compromised WordPress

A log-based investigation of a compromised WordPress site. I followed the Apache access log from normal traffic to reconnaissance, plugin exploitation, web shell upload and confirmed access to fr34k.php.

Compromised WordPress
Platform
Blue Team Labs Online
Category
SOC / Log Analysis
Difficulty
Medium
Artifact
Apache access.log
Focus
WordPress plugin compromise
Investigation goal: reconstruct how the attacker reached the WordPress site, identify attacker tooling, find the vulnerable plugin, locate the uploaded PHP web shell and explain the final attack chain.

Challenge Overview

The scenario starts with a compromised WordPress site. Since the artifact is an Apache access log, I cannot see process execution or filesystem contents directly. I have to infer the attack path from HTTP methods, status codes, request URIs, source IPs and User-Agent strings.

What I thought first: for web logs, the fastest path is usually not reading every line. I want to answer three questions first: who was noisy, what endpoints were touched, and which requests succeeded with 200 or another success-like status code.

Initial Log Review

I started with the beginning and end of the log. The beginning gives a baseline: normal assets, normal browsing and expected WordPress paths. The end often shows late-stage activity or the final successful artifact.

$ head -n 10 access.log # First entries show normal WordPress traffic, static resources and the local gateway/proxy address.
Early observations
  • Activity starts at 12/Jan/2021:15:52:41 +0000.
  • 172.21.0.1 appears heavily, likely because of an internal Docker bridge, reverse proxy or lab gateway.
  • The early User-Agent looks like Linux Firefox 78.
  • Initial requests include normal WordPress CSS, JavaScript and PHP resources.
$ tail -n 20 access.log # Tail entries show probing of plugin/theme paths and a successful request to /YouShouldSeeThis.txt.

The tail showed probing of paths related to contact-form-7, simple-file-list and kadence. I also noticed a successful request to /YouShouldSeeThis.txt. The name looked like a lab marker or post-compromise clue, so I kept it in mind but did not treat it as the initial access vector by itself.

Source IP Analysis

Next, I counted source IPs. This helps separate normal internal proxy traffic from external scanning and exploitation attempts.

$ awk '{print $1}' access.log | sort | uniq -c | sort -nr

The external IP 119.241.22.121 stood out as the most active non-internal source. Later, 103.69.55.212 became important because it interacted with the web shell.

How I treated the IPs: request volume alone is not proof of compromise. I used volume only as a pivot. The real proof came from successful POST requests, plugin upload paths and shell access.

User-Agent Analysis

User-Agent strings showed a mix of browser spoofing and automated tools. This helped me understand attacker behavior: reconnaissance, scanning and exploitation were likely automated.

$ awk -F'"' '{print $6}' access.log | sort | uniq -c | sort -nr WPScan v3.8.10 sqlmap/1.4.11 python-requests/2.24.0 Mozilla/5.0 ... iPhone OS 5_1 ... Opera/9.00 (Windows NT 5.1)
Why this mattered
  • WPScan strongly suggests WordPress vulnerability enumeration.
  • sqlmap suggests SQL injection testing.
  • python-requests/2.24.0 often appears in custom exploit scripts or proof-of-concept tooling.
  • Old browser User-Agents can be spoofed to blend in or test access rules.

Successful POST Requests

For web compromise cases, I pay close attention to successful POST requests. GET requests can be scanning or browsing, but POST requests often carry uploads, login attempts or exploit payloads.

$ grep "POST" access.log | grep -v "403" POST /wp-admin/update.php?action=upload-plugin HTTP/1.1" 200 6434 POST /wp-admin/admin-ajax.php HTTP/1.1" 200 ...

The successful POST to /wp-admin/update.php?action=upload-plugin looked important because WordPress plugin upload can be abused if the attacker has admin access or can bypass upload controls. However, the stronger confirmed web shell path appeared in the Simple File List upload workflow.

What I checked here: I did not want to call every POST malicious. I looked for POST requests with success status codes, unusual upload endpoints and later access to executable PHP files.

Web Shell Discovery

I grouped successful POST requests by source IP and target URI. This exposed repeated POST requests to a PHP file inside the WordPress uploads path.

$ grep "POST" access.log | grep -v "403" | awk '{print $1 " " $7}' | cut -d'?' -f1 | sort | uniq -c | sort -nr 9 103.69.55.212 /wp-content/uploads/simple-file-list/fr34k.php
Confirmed web shell indicator
  • Source IP: 103.69.55.212
  • Web shell path: /wp-content/uploads/simple-file-list/fr34k.php
  • Observed count: 9 POST requests
  • Successful response: 200 OK

The path is highly suspicious. WordPress /uploads/ should normally contain media files, not executable PHP. The filename fr34k.php also looks like a manually chosen shell name rather than a normal plugin asset.

$ grep "103.69.55.212" access.log # At 06:27:04 UTC, the web shell returned 200 OK with a 1295-byte response.

The 1295-byte response was useful because it suggested the shell returned content or interface output, not only a blank hit. In log analysis, response size can be a clue when body content is unavailable.

Vulnerable Plugin Analysis

Simple File List — CVE-2020-8380

The confirmed entry path involved Simple File List. The attacker targeted the plugin upload engine and dropped fr34k.php into the uploads directory.

$ grep "simple-file-list" access.log | grep "POST" | grep -v "403" POST /wp-content/plugins/simple-file-list/lib/ee-upload-engine.php ... 200
  1. 119.241.22.121 sent a POST request to the Simple File List upload engine.
  2. The User-Agent python-requests/2.24.0 suggested an automated Python exploit.
  3. The uploaded PHP file appeared under /wp-content/uploads/simple-file-list/.
  4. 103.69.55.212 accessed fr34k.php shortly afterward.
Why I called this the entry point: it connects an upload-capable vulnerable plugin, a successful POST, an executable PHP file in uploads and later shell interaction. That chain is stronger than only seeing plugin reconnaissance.

Contact Form 7 — CVE-2020-35489

Contact Form 7 was also targeted through the REST API feedback endpoint:

POST /wp-json/contact-form-7/v1/contact-forms/7/feedback

Several requests returned 200 OK, but the logs did not show the same direct shell path as Simple File List. I treated this as a secondary exploitation attempt or alternate path, while Simple File List remained the confirmed successful vector.

Lessons Learned

This lab reinforced that Apache access logs can still tell a full compromise story if I follow the right pivots. The important part was not only finding suspicious tools like WPScan or sqlmap, but connecting them to successful POST requests, vulnerable plugin endpoints and the web shell path.

  • Successful POST requests are often the fastest way to isolate exploit or upload activity.
  • Clean public reputation does not override local evidence. A source IP can be unlisted publicly and still be malicious in this case.
  • PHP files inside WordPress uploads directories are high-confidence web shell indicators.
  • Plugin reconnaissance matters, but the confirmed exploit path needs status code and URI correlation.
  • Response size can provide useful context when the response body is not available.
Final conclusion: the attacker abused the Simple File List plugin upload workflow to place fr34k.php under /wp-content/uploads/simple-file-list/. The shell was then accessed successfully with 200 OK, confirming post-exploitation web shell activity.
