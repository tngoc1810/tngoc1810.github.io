Paper Lantern — CRT-RSA Fault Attack

A beginner-friendly technical thought journal for a pwn challenge where the exploit path is not classic ROP, but protocol analysis, comment parser state corruption, CRT-RSA faulting, RSA private key recovery, and signature forgery.

1. Story and Mental Model

When I first saw this challenge, I expected a normal pwn task: maybe a stack overflow, a format string, a ROP chain, or a heap bug. But the challenge files immediately changed the direction of my thinking. There was a binary named paper_lantern, a Python helper named capsule.py, and a JSON file named public_params.json.

That combination was suspicious. A normal pwn challenge rarely gives public RSA parameters. The presence of SIGN, RUN, and crt-rsa-fdh suggested that the binary was not just executing my input. It was enforcing an authorization rule: a capsule must be signed before it can run.

Beginner mental model: the server is like a guard. A capsule is like a package. The guard only opens packages that have a valid stamp. The unsafe package contains the flag opcode, but the guard refuses to stamp it. So the exploit goal becomes: make a valid stamp ourselves. 
Safe capsule      → server signs it → server runs it
Unsafe capsule    → server refuses to sign it
Exploit strategy  → recover signing key → forge signature → run unsafe capsule

paper_lantern: ELF 64-bit LSB executable, x86-64, ...

It is still a binary challenge, but the helper script suggests the main attack surface is the protocol and internal state, not only saved RIP.
The public parameters looked strange for pwn. I wanted to know exactly what the server exposes. 
cat public_params.json

  "scheme": "crt-rsa-fdh",

The key hints are crt-rsa-fdh and replay_buf = 72. CRT-RSA is vulnerable to fault attacks if one CRT branch is corrupted. A fixed replay buffer size hints at a parser boundary.

3. Learning the Capsule Protocol

I then opened capsule.py. For a newbie, this file looks noisy because it defines many constants and helper functions. The correct approach is to identify the lifecycle of one request.

Newbie StateCommand / InputOutput / Expert Analysis
I saw many frame constants. I needed to translate them into human language. 
sed -n '1,220p' capsule.py
 
FT_HELLO   = 0x10
FT_NEWCAP  = 0x20
FT_APPEND  = 0x21
FT_SIGN    = 0x22
FT_COMMENT = 0x23
FT_RUN     = 0x24

The protocol is: handshake, create a capsule, append records, optionally add comment metadata, ask for a signature, then run with that signature.
I looked at the record helpers to understand what a capsule contains. 
grep -n "REC_\|def rec_" capsule.py
 
REC_TEXT = 0x01
REC_ECHO = 0x02
REC_HALT = 0x03
def rec_raw(kind, payload=b"")

The rec_raw helper is important because it lets me construct arbitrary opcodes, including hidden or unsafe ones.

 
Capsule lifecycle:
HELLO → NEWCAP → APPEND records → SIGN → RUN(signature)

4. First Manual Test: Safe Capsule

Before exploiting anything, I needed to prove that I understood the normal flow. The safest capsule is a harmless one: store a short text or simply halt.

Newbie StateCommand / InputOutput / Expert Analysis
I wrote a tiny test to ask the server to sign a safe capsule. 
payload = rec_text(b"A") + rec_halt()
c.send(FT_NEWCAP)
c.send(FT_APPEND, payload)
c.send(FT_SIGN)
 
[*] getting normal signature
[+] good signature = <64-byte hex signature>

This confirmed that safe capsules are accepted. This signature becomes the good signature for the later CRT fault comparison.
I also tested a visible text output to make sure RUN works. 
payload = rec_text(b"hello") + rec_echo() + rec_halt()
# sign it, then run it with the returned signature
 
[run] ... hello

The server's authorization model is clear: signed safe capsule runs successfully.

5. Testing the Unsafe Opcode

The exploit target is opcode 0x7f. This is the dangerous record that eventually prints the flag. The server, however, refuses to sign it.

Newbie StateCommand / InputOutput / Expert Analysis
I tried to build the suspicious capsule manually. 
payload = rec_raw(0x7f) + rec_halt()

This is the target payload. If it runs, the challenge prints the flag. But first it needs a valid signature.
I tried the obvious path: ask the server to sign the unsafe capsule. 
c.send(FT_NEWCAP)
c.send(FT_APPEND, rec_raw(0x7f) + rec_halt())
c.send(FT_SIGN)
 
[sign] error / unsafe opcode

The server blocks the signing request. The check is at signing time, not only at run time.
I confirmed the binary knows about this rule. 
strings paper_lantern | grep -i unsafe
 
unsafe opcode

This is the main barrier: the server will not directly sign the capsule I want.

6. The Key Mental Shift: From Pwn to Signature Forgery

At this point, my thinking changed. The direct route was blocked: I could not make the server sign opcode 0x7f. But if I could recover the RSA private key, I would not need the server to sign it. 

Problem:
  unsafe capsule needs valid signature
  server refuses to sign unsafe capsule

New goal:
  recover RSA private key
  forge valid signature locally
  run unsafe capsule with forged signature
Important: this is why the challenge is both pwn and crypto. The pwn bug corrupts internal signing state, and the crypto math turns that corruption into a private key recovery.

7. Finding the Comment Braid Bug

Since the main capsule records were filtered, I looked at the secondary input surface: FT_COMMENT. Comments often look harmless, but here they have their own mini-language: literal writes, xor operations, and braid fills.

Newbie StateCommand / InputOutput / Expert Analysis
I noticed the exact value replay_buf = 72. Exact buffer sizes in CTFs are often hints. 
cat public_params.json

The comment decoder likely writes into a 72-byte replay/seam buffer. If a write can cross this boundary, nearby state may be corrupted.
I looked at the helper functions that generate comment bytecode. 
grep -n "comment_" capsule.py
 
comment_literal(...)
comment_xor(...)
comment_braid_fill(...)
comment_patch(...)

The comment parser is more complex than normal record parsing. Complex parsers are good attack surfaces.
I tested the key boundary calculation. 
comment_literal(b"\x00" * 60)
comment_braid_fill(18, 0)
 
60 + 18 = 78
replay_buf = 72
overflow = 6 bytes

This is the logic bug: a braid fill can continue past the intended seam and touch bytes after the replay buffer.

Beginner explanation

valid replay buffer: [0 ... 71]
write 60 bytes:       [0 ... 59]
braid 18 bytes:       [60 ... 77]
overflow bytes:       [72 ... 77]

This is not a classic overflow to overwrite a return address. Instead, it is a state corruption bug. The overwritten bytes influence how CRT-RSA signing behaves.

8. Producing a Faulty CRT-RSA Signature

CRT-RSA signs a message by computing two smaller signatures modulo p and modulo q, then recombining them. If one branch is corrupted, the final signature is wrong in a very useful way.

Newbie StateCommand / InputOutput / Expert Analysis
I already had a normal signature for a safe capsule. Now I needed a faulty signature for the same kind of safe capsule. 
comment = make_fault_comment(gate2)
fault_sig = get_safe_signature(comment)

The capsule itself remains safe. The comment corrupts signing state so the returned signature becomes faulty.
I did not know the exact gate byte that enables the useful fault. 
for gate2 in range(256):
    candidate = get_safe_signature(make_fault_comment(gate2))
    g = gcd(abs(good - candidate), n)
    if 1 < g < n:
        print("found")

A single byte has only 256 possibilities. The gcd check acts as an oracle: it tells me when the fault was useful.
The successful run printed the fault gate. 
python3 solve_paper_lantern.py 178.105.199.41 20000
 
[*] target 178.105.199.41:20000
[+] CRT fault triggered with gate2=0x84

This means the comment payload successfully corrupted the CRT signing path.

9. Factoring n with gcd

Once I had a correct signature and a faulty signature for the same message family, the RSA modulus could be factored with a greatest common divisor calculation.

Newbie StateCommand / InputOutput / Expert Analysis
I converted both signatures from bytes to integers. 
good_int = int.from_bytes(good_sig, "big")
fault_int = int.from_bytes(fault_sig, "big")

RSA signatures are numbers modulo n. To do math on them, I need integer form.
I used the core CRT fault equation. 
p = math.gcd(abs(good_int - fault_int), n)
q = n // p

If one CRT branch is wrong and the other is correct, the difference between the good and faulty signatures shares a factor with n.
The output showed real factors. 
[+] factored n
p = 0x9da033fd0799399257825aff0f7ca4b7866a4db13e250be52e50d7fc3d3eb295
q = 0xb6f7221baafa0048953633e0193e92928aab4bf4dba728b6f713e55135c0b6b5

This is the cryptographic break. Once n = p*q is known, the RSA private exponent can be computed.

Why gcd works

n = p * q

good_sig  is correct modulo p and q
fault_sig is correct modulo one branch, wrong modulo the other

Therefore:
  good_sig - fault_sig is divisible by p or q

So:
  gcd(good_sig - fault_sig, n) = p or q

10. Forging the Flag Capsule Signature

After factoring n, I computed the private exponent d. This gave me the ability to sign the unsafe capsule locally.

Newbie StateCommand / InputOutput / Expert Analysis
I had p and q, but needed the actual RSA signing exponent. 
phi = (p - 1) * (q - 1)
d = pow(e, -1, phi)

This is standard RSA private key recovery. d is the private exponent.
I needed to sign exactly what the server verifies, not just raw bytes. 
m = fdh_message(b"FH", n)

The server uses FDH: Full Domain Hash. The unsafe capsule serializes to b"FH", where F represents the flag opcode and H represents halt.
I created the forged RSA signature. 
forged_int = pow(m, d, n)
forged_sig = forged_int.to_bytes(64, "big")

This signature should verify under the public key, even though the server never agreed to sign the unsafe capsule.

12. Deep Dive for Beginners

What is CRT-RSA?

Normal RSA signing computes m^d mod n. CRT-RSA speeds this up by computing the signature separately modulo p and modulo q, then combining the results. This is faster, but it introduces a dangerous failure mode: if one branch is faulted, the faulty signature can reveal a factor of n. 

normal idea:
  signature = m^d mod n

CRT idea:
  sp = m^dp mod p
  sq = m^dq mod q
  signature = combine(sp, sq)

What is FDH?

FDH means Full Domain Hash. The server does not sign raw capsule bytes directly. It hashes the serialized capsule into a number modulo n, then signs that number. In this challenge, the unsafe capsule is represented as b"FH" before hashing.

What is state corruption?

In classic pwn, an overflow often aims to control the instruction pointer. In this challenge, the overflow is more subtle. The comment braid overflow changes hidden state near the replay buffer. That hidden state affects CRT-RSA signing, which is enough to break the authentication model.

Key lesson: pwn is not always about getting a shell. Sometimes the best exploit is to corrupt one security-critical variable and let the program's own crypto leak the key.

13. Hacker's Mindset

Why did I focus on FT_COMMENT?

The main record path was protected by an unsafe opcode check. When the obvious path is blocked, I look for secondary inputs: comments, metadata, replay buffers, debug fields, compression, or parsers. These are often less protected but still influence internal state.

Why use 60 + 18?

The JSON told me the replay buffer was 72 bytes. I needed a write slightly larger than 72, but not so large that the program would crash immediately. 60 + 18 = 78 crosses the boundary by 6 bytes, which is perfect for nearby state corruption.

Why brute-force gate2?

A one-byte gate has only 256 values. Instead of guessing the right value manually, I let the math decide. For each candidate faulty signature, I computed gcd(abs(good - candidate), n). If the result is between 1 and n, it is a real factor. 

for gate2 in range(256):
    candidate = sign_with_fault(gate2)
    g = gcd(abs(good - candidate), n)
    if 1 < g < n:
        print("CRT fault found")
