Compromised WordPress

A forensic investigation of a compromised WordPress site, focusing on Apache access logs, vulnerable plugins, attacker tooling, web shell activity and final attack reconstruction.

WordPress compromise
Investigation Goal Reconstruct how the attacker entered the environment, identify the vulnerable plugin, determine attacker tooling, locate the web shell and summarize the full attack lifecycle.

Challenge Overview

The scenario starts with a compromised WordPress site. The main hypothesis is that a vulnerable plugin allowed remote code execution and gave the attacker access to the underlying server.

The investigation focuses on Apache access logs and tries to reconstruct the attacker’s path from reconnaissance to successful web shell access.

Initial Log Review

I began by inspecting the first entries of access.log to understand normal traffic, source identity, timestamps and user-agent behavior. 

head -n 10 access.log
Key Findings
  • Activity starts at 12/Jan/2021:15:52:41 +0000.
  • Initial requests originate from 172.21.0.1, likely an internal Docker bridge or local proxy.
  • User-Agent indicates Linux Firefox 78: Mozilla/5.0 (X11; Linux x86_64; rv:78.0).
  • Initial traffic includes normal WordPress static assets such as CSS, JS and PHP resources.

Then I inspected the end of the log to identify later-stage activity and possible attacker success. 

tail -n 20 access.log

The tail end showed aggressive probing of WordPress plugin/theme paths such as contact-form-7, simple-file-list and kadence. This pattern is consistent with automated directory brute forcing or fuzzing.

Smoking Gun A successful request to /YouShouldSeeThis.txt returned 200 OK. The filename suggests a post-compromise marker or a key artifact left for the investigation.

Threat Actor Profiling

To identify suspicious sources, I counted unique IP addresses and sorted them by request volume. 

cat access.log | awk '{print $1}' | sort | uniq -c | sort -nr

The external IP 119.241.22.121 stood out as the most active non-internal source, suggesting automated scanning or brute-force behavior. The internal IP 172.21.0.1 remained dominant because the application was likely behind a gateway or proxy. 

cat access.log | awk '{print $1}' | sort | uniq -c | sort -nr > IPS.txt

User-Agent Analysis

I parsed the Apache User-Agent field to identify spoofed clients and automated security tools. 

cat access.log | awk -F'"' '{print $6}' | sort | uniq -c | sort -nr
Suspicious User-Agents
  • iPhone OS 5_1 — legacy device spoofing.
  • Opera/9.00 (Windows NT 5.1) — outdated Windows XP era browser spoofing.
  • python-requests/2.24.0 — custom Python exploitation script.
  • WPScan v3.8.10 — WordPress vulnerability scanner.
  • sqlmap/1.4.11 — SQL injection automation tool.
cat access.log | awk -F'"' '{print $6}' | sort | uniq -c | sort -nr > user-agents.txt

Initial Access via Successful POST Requests

To move from reconnaissance to confirmed compromise, I filtered for successful POST requests by excluding 403 Forbidden responses. 

grep "POST" access.log | grep -v "403"

The critical event was a successful POST to the WordPress plugin upload handler: 

POST /wp-admin/update.php?action=upload-plugin HTTP/1.1" 200 6434

In WordPress, a successful malicious plugin upload can provide Remote Code Execution by installing or dropping PHP code on the server. Repeated POST requests to /wp-admin/admin-ajax.php also suggested scripted interaction or persistence behavior.

Web Shell Discovery

Correlating successful POST requests by source IP and target URI revealed the web shell path. 

grep "POST" access.log | grep -v "403" | awk '{print $1 " " $7}' | cut -d'?' -f1 | sort | uniq -c | sort -nr
Confirmed Web Shell
  • Source IP: 103.69.55.212
  • Web shell: /wp-content/uploads/simple-file-list/fr34k.php
  • Count: 9 POST requests
  • Impact: Remote command execution through an uploaded PHP shell.

The path is highly suspicious because /uploads/ should contain media files, not executable PHP scripts. The filename fr34k.php is also characteristic of a web shell naming style. 

grep "POST" access.log | grep -v "403" | awk '{print $1}' | sort | uniq -c | sort -nr > POST_IP.txt
grep "POST" access.log | grep -v "403" | awk '{print $7}' | cut -d'?' -f1 | sort | uniq -c | sort -nr > POST_URIpath.txt

Reputation checks for 103.69.55.212 showed geolocation in Taipei, Taiwan, with no public blacklist hits. However, local log behavior confirmed malicious activity. 

grep "103.69.55.212" access.log > suspect_activity.txt
cat suspect_activity.txt
Execution Evidence At 06:27:04 UTC, the web shell returned 200 OK with a 1295-byte response, indicating the shell was active and returned interface/output data.

CVE and Vulnerable Plugin Analysis

The attacker probed multiple WordPress plugins. The two major vulnerable components were Simple File List and Contact Form 7.

CVE-2020-8380 — Simple File List RCE

This was the confirmed initial access vector. The attacker exploited ee-upload-engine.php in the Simple File List plugin to upload fr34k.php into the uploads directory. 

grep "simple-file-list" access.log | grep "POST" | grep -v "403"
Exploit Chain
  1. 119.241.22.121 sent a POST request to ee-upload-engine.php.
  2. User-Agent python-requests/2.24.0 indicated an automated Python exploit.
  3. The payload fr34k.php was dropped into the uploads directory.
  4. 103.69.55.212 connected to the shell 13 seconds later.

CVE-2020-35489 — Contact Form 7 Unrestricted File Upload

Contact Form 7 was also probed. The attacker targeted the REST API feedback endpoint, likely attempting unrestricted file upload exploitation. 

grep "contact-form-7" access.log | grep "POST" | grep -v "403"

The target endpoint was: 

POST /wp-json/contact-form-7/v1/contact-forms/7/feedback

Multiple requests were accepted with 200 OK over a short time window, indicating automated attempts to test or exploit the plugin upload workflow. This appears to be a secondary path or “Plan B”, while Simple File List became the successful entry point.

Final Attack Timeline

Time Event
Jan 12 15:56 Administrator activates simple-file-list, introducing the vulnerable plugin.
Jan 12 15:57 Administrator activates contact-form-7, another vulnerable plugin version.
Jan 14 05:42 Reconnaissance begins from 119.241.22.121 using spoofed User-Agents.
Jan 14 05:54 Brute-force attempts target /wp-login.php?itsec-hb-token=adminlogin, but return 403.
Jan 14 06:01 WPScan v3.8.10 is used for vulnerability scanning.
Jan 14 06:08 A second IP, 103.69.55.212, joins the operation.
Jan 14 06:26 Initial access achieved through Simple File List upload engine.
Jan 14 06:27 fr34k.php is accessed successfully and returns 200 OK.

Answers

1. Admin login panel URI /wp-login.php?itsec-hb-token=adminlogin
2. Two tools used by the attacker WPScan and sqlmap
3. Contact Form 7 CVE CVE-2020-35489
4. Plugin exploited to get access Simple File List
5. PHP web shell filename fr34k.php
6. Final web shell response code 200 OK

Lessons Learned

Key Takeaways
  • Successful POST requests are often the fastest path to isolate exploitation activity.
  • Legitimate-looking or clean public reputation IPs can still be malicious in local evidence.
  • Uploaded PHP files inside media directories are high-confidence web shell indicators.
  • WordPress plugin version fingerprinting is an important attacker reconnaissance step.
  • Defensive tools may block brute force but still fail if vulnerable plugins remain exposed.
