Scenario Overview

In this Sherlock, I worked with a provided NTFS $MFT artifact. The investigation was not about running the suspicious file. Instead, the goal was to use filesystem metadata to answer what happened: which ZIP was downloaded, where it came from, which extracted file was malicious, when it was created on disk, and what C2 address was embedded in the stager.

The main workflow was simple but powerful: parse $MFT with MFTECmd, open the CSV in Timeline Explorer, pivot around file paths and timestamps, then use HxD to recover the content of a small resident file from the raw MFT record.

Core Concepts I Learned

NTFS and MFT

NTFS is the Windows file system used to manage files, folders, timestamps, permissions and disk allocation. The Master File Table, or $MFT, is like the index book of an NTFS volume. It stores a record for files and directories, including file names, parent paths, timestamps, file size, in-use state and sometimes the file content itself.

Resident Files

Normally, an MFT record points to where file data lives elsewhere on disk. However, if a file is very small, NTFS can store the content directly inside the MFT record. This is called resident data. In this lab, the malicious invoice.bat file was small enough to recover directly from the MFT record.

Zone.Identifier / ADS

Zone.Identifier is an Alternate Data Stream created by Windows for files downloaded from external sources. It can contain ZoneId and HostUrl. For DFIR, this is useful because it can reveal the original download URL, which becomes an Indicator of Compromise.

0x10 vs 0x30 Timestamps

MFTECmd exposes timestamps from different MFT attributes. 0x10 refers to $STANDARD_INFORMATION, while 0x30 refers to $FILE_NAME. I used 0x10 for normal timeline reading, and 0x30 for cross-checking because it may preserve a different view of file creation metadata.

Tools Used

  • MFTECmd: Parse the raw $MFT artifact into a readable CSV file.
  • Timeline Explorer: Open, filter, sort and analyze the parsed MFT CSV timeline.
  • HxD Hex Editor: Jump to the MFT record offset and recover resident file content.

Parsing $MFT with MFTECmd

I first used MFTECmd to convert the raw $MFT into a CSV file. The raw MFT is not practical to read manually, so parsing it gives me structured fields such as file name, path, timestamps, size and ADS-related metadata.

MFTECmd.exe -f 'C:\Users\admin\Downloads\BFT\C\$MFT' --csv 'C:\Users\admin\Downloads\BFT\C' --csvf MFT_ANALYSIS.csv

The parser found 171,927 file records and exported the result to MFT_ANALYSIS.csv. This CSV became the main dataset for the investigation.

Reading the MFT Fields

After opening the CSV in Timeline Explorer, I focused on these fields:

Q1 — Identifying the Downloaded ZIP

The scenario said Simon Stark downloaded a ZIP file from a link in an email. I filtered for .zip files and focused on Simon's Downloads directory. The most relevant file was directly under .\Users\simon.stark\Downloads and its name contained the attack date 20240213.

Stage-20240213T093324-001.zip

Q2 — Extracting the HostUrl from Zone.Identifier

After identifying the initial ZIP, I examined the related Zone.Identifier ADS. The Zone ID Contents field revealed the HostUrl where the ZIP was downloaded from. This is valuable as an IOC because it directly links the file to its external source.

https://storage.googleapis.com/drive-bulk-export-anonymous/20240213T093324.039Z/4133399871716478688/a40aecd0-1cf3-4f88-b55a-e188d5c1c04f/1/c277a8b4-afa9-4d34-b8ca-e1eb5e5f983c?authuser

Q3 — Finding the Malicious Stager

Since the downloaded file was a ZIP, I pivoted into the extracted directory named after the archive. I filtered the Parent Path for Stage-20240213T093324Z-001. Inside the extracted content, invoice.bat stood out because batch scripts can execute Windows commands and are commonly abused in phishing-style attacks.

C:\Users\simon.stark\Downloads\Stage-20240213T093324Z-001\Stage\invoice\invoices\invoice.bat

Q4 — Created0x30 Timestamp

The question specifically asked for the Created0x30 value, so I used the timestamp from the $FILE_NAME attribute instead of Created0x10. This timestamp is useful for cross-checking file creation activity.

2024-02-13 16:38:39

Q5 — Calculating the MFT Record Offset

Each MFT record is typically 1024 bytes. The malicious file had Entry Number 23436. To find the start offset of that record inside the $MFT file, I multiplied the entry number by 1024.

23436 × 1024 = 23998464
23998464 decimal = 0x16E3000
0x16E3000

Q6 — Recovering Resident File Content and C2

I opened the raw $MFT in HxD, pressed Ctrl + G, and jumped to offset 16E3000. Because the batch file was small, its content was resident inside the MFT record. The decoded text revealed a PowerShell stager that attempted to run in a less visible way, bypass execution policy, download a remote script and execute it with iex. The script also attempted to delete itself after execution.

Analyst note: I treated the recovered command as evidence, not something to execute. The important IOC was the remote address embedded in the stager.
43.204.110.203:6666

Lessons Learned

This lab helped me understand why MFT forensics is powerful. Even if a user deletes a suspicious script, the MFT may still preserve metadata and sometimes even the file content. By combining path analysis, timestamps, ADS metadata, record offsets and resident data recovery, I reconstructed the attack chain from downloaded ZIP to malicious stager and C2 IOC.

  • NTFS Forensics: Learned how MFT records track file metadata.
  • Timeline Creation: Used Timeline Explorer to filter and reconstruct file activity.
  • File Recovery: Recovered resident batch content directly from the MFT.
  • Disk Forensics: Calculated MFT record offset using Entry Number × 1024.
  • Contextual Analysis: Combined file name, path, extension, timestamps, ADS and script content to identify malicious behavior.
Final Conclusion Simon Stark downloaded a ZIP archive from a Google Cloud Storage URL. After extraction, the malicious file was invoice.bat. Its resident MFT content revealed a PowerShell stager pointing to 43.204.110.203:6666.
