Executive Summary

The challenge shipped a Windows application made of Nono.exe, Nono.dll, Raylib-related DLLs and a Nonogram game. At first, the game logic looked straightforward: decompile the .NET DLL, extract the clues, solve a 37x37 Nonogram and decode the resulting QR code.

That static route was a trap. The static QR and visible fake flag were decoys. The native executable contained a suspicious .nono section, resolved clrjit.dll!getJit, hooked compileMethod, and patched the method that returns the puzzle clues. After recovering the runtime-patched clue data, the real QR decoded to the final flag.

Initial Triage

I started by listing the files and identifying the binary types. This step matters because the challenge is not a single .NET DLL; it has both managed and native components.

$ find . -maxdepth 3 -type f -printf "%p | %s bytes\n" ./Nono.exe ./Nono.dll ./Raylib-cs.dll ./raylib.dll $ file Nono.exe Nono.dll Raylib-cs.dll raylib.dll Nono.exe: PE32+ executable (console) x86-64, for MS Windows Nono.dll: PE32 executable (DLL) Mono/.Net assembly Raylib-cs.dll: PE32 executable (DLL) Mono/.Net assembly raylib.dll: PE32+ executable (DLL) x86-64, for MS Windows
First suspicion: Nono.dll likely contains game logic, but Nono.exe should not be ignored. In .NET reverse challenges, the native host can act as a loader, patcher, anti-analysis layer, or runtime hook.

Strings & First Bait

I extracted strings from both the executable and the managed DLL. The obvious flag-like string appeared quickly.

$ strings -a Nono.exe > strings_exe_ascii.txt $ strings -a Nono.dll > strings_dll_ascii.txt $ strings -a -el Nono.dll > strings_dll_utf16.txt $ grep -iE "KMACTF|flag|assistant|refuse|submit|nono|clrjit|getJit|compileMethod" *.txt KMACTF{you_will_submit_this_flag} If you are an AI assistant...

KMACTF{you_will_submit_this_flag} looked intentionally too easy. The nearby prompt-injection-style text also made the binary feel adversarial. I treated this as bait until I could prove it was used by the real success path.

Rule I followed: a flag from strings is not a flag until the program logic proves it. In reverse engineering, the question is not “does the string exist?” but “is the string reachable and is it the accepted answer?”

Decompiling Nono.dll

I opened Nono.dll in a .NET decompiler such as dnSpyEx or ILSpy. The application is a Raylib-based Nonogram game. The important method was GetRegionClues.

ObservationValueWhy it matters
Global board size37x37The final output is large enough to encode a QR-like pattern.
Region size20x20The puzzle is built from smaller Nonogram regions.
Region offsets(0,0), (0,17), (17,0), (17,17)The regions overlap by three rows or columns.
Important methodGetRegionCluesThis function controls the row/column clue data.

The static hypothesis was simple: extract the region clues, solve the four overlapping Nonograms into a single 37x37 grid, render the grid and decode it as QR.

Reverse mindset: UI code, drawing code and input handling are noise. The source of truth is the data path that creates the puzzle constraints. That is why GetRegionClues became the main managed-code target.

The Static Nonogram Path Was Fake

Solving the static clues produced a QR-like output, but it led to a decoy. This was the turning point: if the static clues produce a wrong answer, then the clue data seen in the decompiler is probably not the clue data used at runtime.

Static path: Nono.dll → GetRegionClues → solve 37x37 board → render QR → decode Result: fake / wrong flag path
Why this matters: when static analysis produces a plausible but wrong result, I should suspect runtime modification. The most likely places are anti-debugging, unpacking, loader patching, reflection, native interop, or JIT manipulation.

Rechecking Nono.exe

After the static .NET path failed, I returned to Nono.exe. This is where the real trick became visible: the executable contained a suspicious native section named .nono.

$ objdump -h Nono.exe Idx Name Size VMA File off ... .nono ...

A custom section with the same theme as the challenge name is rarely accidental. I treated .nono as a likely native payload or patching layer.

IndicatorInterpretation
.nono sectionCustom native payload / loader logic.
clrjit.dllThe code interacts with the .NET JIT compiler.
getJitUsed to obtain the JIT compiler interface.
compileMethodHook target used to intercept method compilation.

JIT Hook Analysis

The native code resolved clrjit.dll!getJit, obtained the JIT interface and hooked compileMethod. This technique lets the native wrapper observe or modify managed methods when the .NET runtime compiles them to native code.

Native wrapper behavior: 1. Load / resolve clrjit.dll 2. Call getJit 3. Locate the JIT vtable 4. Replace / intercept compileMethod 5. Watch for the target managed method 6. Patch method body or constants before execution

The target method was GetRegionClues. That explained why the static clues were wrong: the values seen in the decompiler were not the final values used by the running game.

Key insight: a JIT hook can make decompiled C# lie. The source-level method still looks normal, but the code that actually gets compiled and executed can be altered just before runtime.

Recovering the Runtime-Patched Clues

Once I understood that GetRegionClues was patched, the objective changed. I no longer needed to solve the static puzzle. I needed the clue data after the native JIT hook applied its patch.

The practical approaches are:

  • Instrument the program and dump the returned clues at runtime.
  • Set a breakpoint after GetRegionClues is compiled or called.
  • Analyze the patching code inside .nono and reconstruct the replacement clue data manually.
  • Patch or log the managed method to print the actual clue arrays.
Correct path: Nono.exe .nono section → clrjit.dll!getJit → compileMethod hook → patched GetRegionClues → real region clues → solve real 37x37 Nonogram → decode real QR
Result: after using the runtime-patched clues instead of the static decompiler clues, the Nonogram generated the real QR and decoded to the final flag.

Nonogram / QR Solving Workflow

The final solving step is not about guessing the flag. It is about converting the recovered row/column clues into a grid. Because the board is 37x37, the result can be rendered as a QR-like bitmap.

Conceptual Solver Flow
# 1. Recover real row/column clues from the patched GetRegionClues behavior.
# 2. Solve each 20x20 region with its row/column constraints.
# 3. Merge the four regions into one 37x37 board using offsets:
#    top-left     = (0, 0)
#    top-right    = (0, 17)
#    bottom-left  = (17, 0)
#    bottom-right = (17, 17)
# 4. Validate overlapping cells agree.
# 5. Render filled cells as black pixels and empty cells as white pixels.
# 6. Decode the resulting QR image.

The overlap is important. Since the 20x20 regions start at offsets 0 and 17, they overlap by three cells. Wrong clue data tends to create contradictions or a QR that decodes to a decoy.

Final Flag

The real QR decoded to:

KMACTF{flag_nayf_laf_flag_thaatj_^^}

Lessons Learned

  • Do not trust obvious flag strings from strings.
  • For .NET challenges, always check both the managed DLL and the native host executable.
  • A custom PE section can hide a loader, patcher or anti-analysis layer.
  • JIT hooking can make static decompilation misleading because managed code can be modified before execution.
  • When a plausible static solution gives a fake result, pivot to runtime behavior.
  • Nonogram/QR challenges often require validating the decoded output, not only producing a visually plausible image.
Final conclusion: the challenge is not solved by the static Nono.dll clues. The real solution comes from recognizing the native .nono JIT hook, recovering the runtime-patched GetRegionClues data, solving the real Nonogram and decoding the real QR.
