Boogeyman 2: Following the Evidence from Phishing Email to Persistence

This writeup documents the actual investigation flow I followed: what I saw in each screenshot or artifact, why it looked important, which command I ran next, and how each finding helped me move from a suspicious resume document to the final C2 and persistence evidence.

Boogeyman 2
Platform
TryHackMe
Focus
SOC / DFIR
Artifacts
EML, DOC, Memory Dump
Tools
olevba, Volatility, strings
Core idea
Correlate, do not guess
Investigation goal: prove how the HR workstation was compromised, identify what the Word document executed, confirm it in memory, find the binary and C2 connection, then recover how the attacker maintained access.

Scenario Overview

Maxine Beck, an HR employee at Quick Logistics LLC, received a resume application email. At first glance, that does not look strange because HR staff normally receive resumes from people outside the company. That is exactly why this lure was dangerous: it matched the victim's job role.

I treated the investigation like a small incident response case. I did not start by trying to answer the room questions directly. I first tried to reconstruct the chain in order:

Email → Word attachment → Macro → JavaScript stage → EXE payload → C2 callback → Scheduled task persistence

How I Approached the Case

For every artifact, I used the same simple process:

  1. Look at the evidence first. What is visible in the email, macro output, process tree or network output?
  2. Ask what it proves. Is this only a clue, or does it confirm execution?
  3. Choose the next command. The next command should answer a specific question, not just dump more data.
  4. Correlate before concluding. A URL in strings is not automatically an IOC. A process name alone is not enough. I only promoted evidence when it matched the execution chain.
Important distinction: olevba shows what the document is designed to do. Volatility shows what actually happened on Maxine's workstation.

1. Starting with the Email

The first thing I looked at was the email itself. I wanted to identify the sender, the victim, the subject, and whether an attachment was present.

From the preview, I could immediately extract the first part of the case:

From: westaylor23@outlook.com
To: maxine.beck@quicklogisticsorg.onmicrosoft.com
Subject: Resume - Application for Junior IT Analyst Role
Attachment: resume_WesleyTaylor.doc

At this point, I did not know yet whether the attachment was malicious. But the situation was suspicious enough to continue: an external Outlook address sent a resume-themed Word document to an HR employee. The pretext was believable, and that made it a good spear-phishing lure.

Next step I chose: do not open the Word document normally. First, identify the file type and hash it so I can treat it as evidence.

2. Checking the Attachment Without Opening It

I checked the file type first:

file Resume_WesleyTaylor.doc

The output showed a legacy Microsoft Word/OLE document. That mattered because older .doc files can contain VBA macro streams. A resume document should normally be passive content, but a macro-capable Office document in a phishing email needs closer inspection.

I also calculated hashes. The SHA256 was useful for reputation lookup, but the room specifically asked for MD5, so I ran:

md5sum Resume_WesleyTaylor.doc

The MD5 value was:

52c4384a0b9e248b95804352ebec6c5b

I then used VirusTotal and the community sandbox view to get context, not as final proof.

VirusTotal showed the file was widely detected, and the sandbox hinted at winword.exe → wscript.exe. That was useful because it gave me something concrete to verify later in memory.

Why I did not stop here: reputation results can guide the investigation, but they do not explain the attack. I still needed to read the macro and prove the process chain from the memory dump.

3. Reading the Word Macro with olevba

Because the file was a legacy Office document, the next command was:

olevba Resume_WesleyTaylor.doc

The output showed a macro stream called NewMacros.bas. The first important thing I noticed was Sub AutoOpen().

AutoOpen() changes the meaning of the document. This is no longer a file waiting for a user to click a macro button. It is designed to run code as soon as the victim opens the Word document.

Then I followed the macro line by line. It created a path under C:\ProgramData, created an HTTP object, downloaded a remote file, wrote it to disk, then executed it with Windows Script Host.

Download URL:
https://files.boogeymanisback.lol/aa2a9c53cbb80416d3b47d85538d9971/update.png

Saved locally as:
C:\ProgramData\update.js

Executed with:
wscript.exe C:\ProgramData\update.js

The file extension mismatch was the key detail. The remote file looked like update.png, but the macro saved it as update.js and executed it. So I treated update.png as the stage 2 JavaScript payload, not as an image.

Next question: did this actually run on Maxine's machine, or was it only code inside the document? To answer that, I moved to Volatility.

4. Validating Execution in Memory

The macro gave me a hypothesis: Word should spawn wscript.exe, and wscript.exe should run C:\ProgramData\update.js. I used the memory dump to check that.

vol -f WKSTN-2961.raw windows.pstree

The process tree matched the story perfectly:

OUTLOOK.EXE PID 1440
└── WINWORD.EXE PID 1124
    └── wscript.exe PID 4260
        └── updater.exe PID 6216

This was the first strong runtime confirmation. Outlook opened the email, Word opened the document, Word launched wscript.exe, and then a new suspicious executable appeared under it.

The process tree told me parent-child relationships, but I still needed the exact command lines. So I ran:

vol -f WKSTN-2961.raw windows.cmdline > cmdline.txt
cat cmdline.txt

The command-line output confirmed three important things:

1. Word opened the attachment from Outlook cache:
C:\Users\maxine.beck\AppData\Local\Microsoft\Windows\INetCache\Content.Outlook\WQHGCZFT\Resume_WesleyTaylor (002).doc

2. wscript.exe executed the stage 2 payload:
wscript.exe C:\ProgramData\update.js

3. updater.exe executed from:
C:\Windows\Tasks\updater.exe

At this point, the document was no longer just suspicious. It was proven to have triggered a real process chain on the victim workstation.

5. Finding What update.js Did Next

The child process updater.exe told me that update.js probably downloaded or launched another binary. To find file artifacts related to this, I used:

vol -f WKSTN-2961.raw windows.filescan > filescan.txt
grep -i "update" filescan.txt

This revealed update[1].png, update[1].exe, and \Windows\Tasks\updater.exe. That lined up with a staged flow: the macro downloaded a script, and the script downloaded an executable.

To recover the original URL for the executable, I searched memory strings for the attacker-controlled domain:

strings WKSTN-2961.raw | grep -i "boogeymanisback"

The important line was:

https://files.boogeymanisback.lol/aa2a9c53cbb80416d3b47d85538d9971/update.exe

I accepted this as the binary download URL because it matched the other evidence: update[1].exe existed in cache, and updater.exe was running from disk.

6. Proving the C2 Connection

Now that updater.exe was the suspicious process, I needed to see whether it communicated externally. I tried network plugins and used the output to focus on PID 6216:

vol -f WKSTN-2961.raw windows.netscan > netscan.txt
grep "6216" netscan.txt

The output showed repeated connections from updater.exe PID 6216 to:

128.199.95.189:8080

This was strong because the IP was not just a random string in memory. It was tied to the malicious process that came from the macro execution chain.

Why I was careful here: I also saw other suspicious-looking domains in raw strings, but I did not include them as confirmed IOCs because they were not tied to the observed process or network chain. 128.199.95.189:8080 was confirmed because updater.exe connected to it.

7. Recovering the Persistence Command

The room hinted that the attacker created a scheduled task after the C2 callback. My first instinct was to search cmdline.txt:

grep -i "schtasks" cmdline.txt
grep -i "tasks" cmdline.txt

That only showed C:\Windows\Tasks\updater.exe, not the full schtasks command. Instead of assuming there was no persistence, I changed the evidence source. Short-lived commands can exit before windows.cmdline recovers them, but their strings may still remain in memory.

So I created ASCII and Unicode strings:

strings -a -n 5 WKSTN-2961.raw > ascii_strings.txt
strings -a -el -n 5 WKSTN-2961.raw > unicode_strings.txt

grep -ni "schtasks" ascii_strings.txt
grep -ni "schtasks" unicode_strings.txt

The ASCII output gave noisy fragments, but the Unicode strings gave the clean command. That makes sense because Windows command lines and PowerShell-related strings often appear as UTF-16LE.

schtasks /Create /F /SC DAILY /ST 09:00 /TN Updater /TR "C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe -NonI -W hidden -c \"IEX ([Text.Encoding]::UNICODE.GetString([Convert]::FromBase64String((gp HKCU:\Software\Microsoft\Windows\CurrentVersion debug).debug)))\""

This created a scheduled task named Updater, configured to run daily at 09:00. The task launches PowerShell in hidden mode, reads an encoded payload from HKCU:\Software\Microsoft\Windows\CurrentVersion\debug, decodes it as Unicode, and executes it with IEX.

What this proved: the attacker did not only run a one-time payload. They also created a daily persistence mechanism using Scheduled Tasks, hidden PowerShell, and a registry-stored payload.

What I Learned

  • Do not jump straight to answers. Build the chain first.
  • olevba is useful for understanding what a malicious Office document wants to do.
  • Volatility process tree and command-line output are powerful because they prove what actually ran.
  • Raw strings are noisy; I should only trust strings that correlate with process, file, or network artifacts.
  • Persistence is not always a file on disk. It can be a scheduled task that executes payload stored in the registry.
