Masterminds: Threat Hunting & Network Traffic Analysis

A detailed network traffic analysis investigating three compromised machines in the Finance department, uncovering phishing payloads, RedLine Stealer, Phorpiex botnet, and C2 evasion techniques using Brim.

Goal: Investigate network traffic logs from three endpoints to identify indicators of compromise (IOCs), uncover the malware strains used, and determine the attacker's infrastructure.

1. Scenario

Three machines in the Finance department at Pfeffer PLC were compromised. We suspect the initial source of the compromise happened through a phishing attempt and by an infected USB drive. The Incident Response team managed to pull the network traffic logs from the endpoints. Use Brim to investigate the network traffic for any indicators of an attack and determine who stands behind the attacks.

2. Investigations: infection1.pcap

By filtering HTTP traffic, a serious anomaly was detected. The internal endpoint (192.168.75.249) initiated a plain text HTTP GET request to an untrusted external domain (hdmilg.xyz) to download a Windows executable file named catzx.exe. Immediately after downloading this malware, the infected server began sending signals to several other suspicious domains (e.g., cambiasuhistoria.growlab.es, gocphongthe.com), clearly indicating a malware strain establishing Command and Control (C2) communication channels.

Anomaly Detected: A highly suspicious HTTP transaction was identified. The target server returned a 404 Not Found status code; however, the associated response body size was unusually massive, clocking in at 46,456 bytes.

Analysis: Threat actors frequently utilize "Fake 404" responses to smuggle malicious payloads, configuration files, or secondary commands past traditional network defenses (like IDS or proxies) that might deprioritize inspecting traffic explicitly flagged as missing or erroneous.

DNS Traffic Analysis

After identifying the initial payload delivery, the investigation shifted to DNS traffic analysis to map out the broader Command and Control (C2) infrastructure.

The DNS resolution logs perfectly corroborated the anomalous HTTP traffic observed earlier. The infected endpoint systematically attempted to resolve a specific cluster of suspicious domains. Notably, CAB.MYKFN.COM and IE-BEST.NET recorded the highest query counts, strongly suggesting they act as primary C2 nodes for beaconing.

During analysis of outgoing HTTP connections, an unusual communication pattern was observed directed toward bhaktivrind.com. The infected endpoint made a GET request to the URI /cgi-bin/JBbb8/, resulting in a 500 Internal Server Error with a response body of 0 bytes.

While typically indicative of a standard server error, in the context of a malware infection, this behavior reveals crucial details about the attacker's infrastructure.

  • Hypothesis 1 (Third-party infrastructure compromised): The bhaktivrind.com domain may be a legitimate website, but compromised, being used as a proxy server or disposable C2 node.
  • Hypothesis 2 (C2 access control mechanism): Closer examination of the logs reveals an empty user_agent field. Sophisticated C2 servers often implement strict filtering. If the request lacks the specific hardcoded User-Agent string or authorization token that the malware expects, the C2 server may intentionally disconnect and return a 500 error to avoid detection by automated testing systems and security researchers.

www.letscompareonline.com was flagged as malicious by 7/91 security vendors.

cambiasuhistoria.growlab.es was flagged by 8/91 security vendors.

Verdict: True Positive - Compromised
Severity: High

Analyst Comments: Investigation confirms the endpoint is fully compromised. The host downloaded a cleartext malicious executable (catzx.exe) from hdmilg.xyz. Post-infection telemetry shows active Command & Control (C2) beaconing to multiple domains (notably cab.mykfn.com). The threat actor is utilizing defense evasion techniques, specifically smuggling 46KB payloads disguised as HTTP 404 Not Found errors, and returning 500 Internal Server Error to evade analysis.

Recommended Actions:

  • Immediately isolate the host from the corporate network.
  • Block all identified C2 domains and IPs at the perimeter firewall.
  • Perform full system reimage.

3. Investigations: infection2.pcap

For the second infected endpoint (192.168.75.146). The query results mapped out a clear, two-stage compromise process:

  • Stage 1 - Payload Delivery: The endpoint executed an HTTP GET request to the external domain hypercustom.top (resolved to IP 45.95.203.28), targeting the URI /jollion/apines.exe. This confirms the successful download of a secondary malicious executable.
  • Stage 2 - Anomalous C2 Activity: The endpoint subsequently initiated multiple HTTP POST requests to 5.181.156.252 over port 80. Crucially, the HTTP Host header contained a bare IP address rather than a resolved domain name.

To validate the anomalous HTTP traffic and identify the specific malware strain or delivery mechanism, I pivoted to analyzing the Suricata Intrusion Detection System (IDS).

The IDS alerts perfectly corroborated the prior HTTP logs, firing exactly at the timestamp of the payload delivery (45.95.203.28). Key triggered signatures included:

  • Suspicious Firesale gTLD EXE DL with no Referer: This confirms the automated, script-driven nature of the apines.exe download.
  • Suspicious User-Agent (AutoHotkey): Critical Finding. The HTTP request utilizing an AutoHotkey User-Agent indicates that the initial compromise vector (likely the suspected malicious USB drive mentioned in the initial incident briefing) contained an AutoHotkey script. Threat actors frequently weaponize AutoHotkey to compile loaders and droppers, effectively obfuscating their malicious intent to bypass standard endpoint AV detection.

The OSINT query yielded a definitive match. The payload is heavily flagged and associated with RedLine Stealer (often distributed alongside Cryptbot).

Impact Assessment: RedLine Stealer is a highly prolific Information Stealer designed to harvest saved browser credentials, session cookies, cryptocurrency wallets, and system inventory. This attribution perfectly contextualizes the anomalous HTTP POST requests observed targeting 5.181.156.252.

Verdict: True Positive - Critical Breach
Severity: Critical (Active Data Exfiltration)

Analyst Comments: Critical incident involving RedLine Stealer. Evidence indicates the initial compromise occurred via a weaponized AutoHotkey script, which acted as a downloader to fetch apines.exe from hypercustom.top. The malware successfully executed and established a C2 exfiltration channel via direct HTTP POST requests to a hardcoded IP (5.181.156.252). Given the nature of RedLine Stealer, it is highly probable that corporate credentials, session tokens, and financial data have been exfiltrated.

Recommended Actions:

  • Urgent host isolation.
  • Mandatory and immediate password rotation for the affected user across all corporate and financial platforms.
  • Revoke all active session tokens for the compromised user.

4. Investigations: infection3.pcap

Investigate the final compromised endpoint (192.168.75.232).

The endpoint exhibited classic automated dropper behavior supported by a resilient delivery infrastructure:

  • Sequential Payload Retrieval: The host systematically issued HTTP GET requests to download sequentially named executables (/s/1.exe through /s/5.exe).
  • Fallback Infrastructure: To ensure successful delivery, the malware cycled through multiple suspicious Russian TLDs acting as mirror servers: xfhoahegue.ru, afhoahegue.ru, and efhoahegue.ru.

The endpoint repeatedly requested the URI /s/VNEW=1 across these domains. This is highly indicative of a bot check-in mechanism communicating with the Command & Control server.

Following the identification of the dropper's payload delivery mechanism, the investigation expanded to DNS traffic to uncover potential lateral movement or internal network mapping attempts by the compromised endpoint.

While the log contained expected Windows broadcast noise (e.g., wpad.local, isatap.local), the endpoint generated a statistically significant volume of DNS queries for pseudo-randomly generated internal hostnames (e.g., hxbhaelzrcmf.local, vrahsvtwl.local, xcpthyukbep.local).

All automated GET requests to the malicious .ru domains utilized the exact same, hardcoded User-Agent string:

Mozilla/5.0 (Macintosh; Intel Mac OS X 10.9; rv:25.0) Gecko/20100101 Firefox/25.0

The User-Agent explicitly claims the client is running macOS. However, the requested payloads are Windows executables (.exe), a behavior inconsistent with legitimate macOS user activity.

The OSINT query successfully correlated the infrastructure to known malicious activity. Specifically, threat intelligence reports attribute the domain efhoahegue.ru as an active Command & Control (C2) node for the Phorpiex botnet (also known as Trik).

Verdict: True Positive - Compromised
Severity: High

Analyst Comments: Endpoint is infected by the Phorpiex Botnet, functioning as a multi-stage dropper. The host systematically downloaded sequential payloads (1.exe through 5.exe) utilizing a resilient fallback infrastructure composed of multiple .ru domains. The malware utilized a spoofed, outdated macOS User-Agent to bypass basic filtering. Furthermore, DNS logs indicate the malware is actively conducting internal network reconnaissance (querying randomized .local domains), presenting a high risk of lateral movement.

Recommended Actions:

  • Isolate the host immediately to prevent lateral movement.
  • Hunt for .local DNS queries across the broader network to identify if lateral movement was successful.
  • Reimage the machine.

5. Lessons Learned & Key Takeaways

Completing the "Masterminds" investigation provided invaluable hands-on experience in Network Traffic Analysis (NTA) and Incident Response.

  • Behavioral Anomaly Detection: This investigation highlighted that threat hunting is about spotting the unusual. I successfully identified multiple defense evasion techniques, including:
    • Massive HTTP payloads (46KB) masquerading as benign 404 Not Found errors.
    • Direct-to-IP Command & Control communications bypassing DNS resolution.
    • Spoofed, outdated macOS User-Agents requesting Windows .exe executables.
  • Telemetry Correlation: I practically applied the concept of event correlation by cross-referencing cleartext HTTP downloads with DNS beaconing frequencies and Suricata IDS alerts. This holistic approach was essential in reconstructing the complete kill chain for three distinct malware families (including RedLine Stealer and the Phorpiex Botnet).
