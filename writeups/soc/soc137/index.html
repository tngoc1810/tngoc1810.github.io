SOC137 - Malicious File Download Attempt

Investigation of a malicious macro-enabled document download attempt on host NicolasPRD, verifying whether the payload was successfully blocked or if the endpoint was compromised.

2. Initial Assessment & Hypothesis

Upon reviewing the alert details, several suspicious indicators were identified:

  • Suspicious Filename & Extension: The filename uses the keyword "Invoice," which is a common social engineering tactic designed to trigger urgency or curiosity in users. Furthermore, the .docm extension (Word Macro-Enabled Document) is frequently abused by attackers to embed malicious VBA scripts. Once opened, these scripts can execute silently to download secondary payloads to the victim's machine.
  • Defense Action: The device action is logged as Blocked, indicating that the perimeter defense system likely intercepted the download attempt.
Investigation Goal: Despite the "Blocked" status, a deep-dive investigation is necessary to verify two critical points:
  1. Is the file genuinely malicious (True Positive), or is this a false alarm?
  2. Was the download attempt truly and successfully blocked at the network level?

Threat Intelligence & Malware Analysis

To verify the malicious nature of the payload, the file and its associated hash were analyzed using OSINT platforms (VirusTotal and Joe Sandbox).

VirusTotal Analysis: The file hash was flagged as malicious by 40 out of 64 security vendors. The behavioral tags revealed that the document contains malicious macros, attempts to execute PowerShell, and employs evasion techniques.

Joe Sandbox Detonation: Sandbox analysis confirmed the file as MALICIOUS with high confidence. The behavioral signatures indicate that upon opening the document and enabling content, the embedded VBA macro executes obfuscated command lines and spawns suspicious child processes (PowerShell). This is a classic behavior of a dropper attempting to establish a connection and download secondary payloads.

Partial Conclusion: The payload is confirmed to be highly malicious (True Positive at the file level).

Log Management Analysis

We have proven that the weapon is real and highly dangerous. Now, the critical question remains: Did the host NicolasPRD successfully download and execute this malicious file?

A comprehensive search in the Log Management system was conducted using the Source IP (172.16.17.37), Hostname, and the specific timeframe. The search yielded no results regarding successful connections or file transfers.

The absence of network traffic logs (such as HTTP 200 OK) strongly corroborates the SIEM's "Blocked" action, proving the connection was terminated at the network perimeter before any data could be downloaded.

Endpoint Analysis & C2 Check

Further investigation on the Endpoint Detection and Response (EDR) platform for the host NicolasPRD revealed no traces of the malicious .docm file. There are no indicators of file creation, execution, or anomalous child processes. The endpoint remains clean and uncompromised.

C2 Communication Check: Even though the download was blocked and the EDR showed no signs of execution, I proactively searched the network logs for any suspicious outbound connections or beaconing activity from NicolasPRD to known Command and Control (C2) servers. The result was negative. The absence of any C2 traffic is the final proof that the endpoint is completely clean.

Analyst Note & Verdict

Analyzed alert SOC137 (EventID: 76) regarding a malicious .docm download attempt by host NicolasPRD. OSINT analysis confirmed the file is a malicious macro-enabled dropper. However, log management and EDR analysis verified that the download was successfully Blocked at the network perimeter. No malicious files were downloaded, and no C2 connections were established. The endpoint remains uncompromised.

Final Verdict: True Positive.
Action: Closing ticket. No further remediation is required.

Lessons Learned & Takeaways

  • This investigation reinforced the core SOC mindset. Although the SIEM device action reported the download as blocked, an analyst must never assume the system is perfectly safe. Verifying network logs and EDR telemetry is mandatory to confirm the actual impact.
  • The most critical step is determining whether the payload reached the endpoint and executed. In this case, proving the absence of network traffic and C2 communication was just as important as proving the file was malicious.
