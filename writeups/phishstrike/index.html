How I handled the case

I did not want to decide that the email was malicious only because the subject looked suspicious. I started with the sender identity, return path, authentication results and delivery chain. After that, I moved to the body URL, because in a phishing email the link usually tells the real story.

The sender domain itself did not look obviously malicious in reputation tools. That was a good reminder: a clean-looking domain does not make an email safe. The message still had SPF softfail, DKIM fail, and a body link that pointed directly to a Windows executable hosted on a raw IP address.

Working conclusion: the email was a phishing-to-malware-delivery case. The strongest evidence was not one single artifact; it was the combination of weak mail authentication, executable delivery URL, URLHaus history, sandbox behavior, persistence evidence and C2 indicators.

1. Email header analysis

I began with the visible identity of the email. The sender appeared as ERIKA JOHANA LOPEZ VALENTE <erikajohana.lopez@uptc.edu.co>. The recipient field used undisclosed-recipients, which made me think this could be a bulk phishing message rather than a normal one-to-one invoice email.

The authentication result was the first real technical problem. The email showed SPF softfail and DKIM fail. The sender IP tied to that authentication result was 18.208.22.104.

spf=softfail (sender IP is 18.208.22.104)
dkim=fail
smtp.mailfrom=uptc.edu.co
header.d=uptc.edu.co
dmarc=none
Why I cared: I did not use SPF/DKIM failure alone as the whole verdict, but it told me not to trust the visible sender. The next thing to check was the body link.

2. Malicious URL in the email body

The body pretended that an invoice or receipt was available for review. But the embedded link pointed directly to a Windows executable:

http://107.175.247.199/loader/install.exe

This changed the direction of the case. A real invoice is usually a PDF, document attachment, or a legitimate portal link. A raw IP address serving /loader/install.exe looked like malware staging, so I treated 107.175.247.199 as the main infrastructure to investigate.

3. Reputation and threat intelligence

I checked the sender domain and sender IP first, but I did not stop there. The sender domain uptc.edu.co was not broadly flagged, and 18.208.22.104 also looked mostly clean in reputation tools. That did not clear the email, because the body URL was still suspicious.

The payload hosting IP was different. 107.175.247.199 was tied directly to the executable download in the email body, so it became a stronger IOC than the sender domain.

URLHaus gave the strongest context. The same /loader/install.exe endpoint had been seen as a malware download URL and had delivered multiple payloads over time.

4. CoinMiner behavior

The cryptocurrency mining payload was tagged as CoinMiner. I did not just copy the family label; I checked the sandbox behavior around it because the lab also asked for a specific request made by the malware.

One process detail looked small but mattered: the sandbox showed a PowerShell encoded command. I decoded it in CyberChef and got a sleep command:

Start-Sleep -Seconds 50

Then I checked network artifacts. The CoinMiner sample reached out to staging infrastructure and made a request to a suspicious uploaded file on ripley.studio.

Reasoning: this is why the answer is not only “CoinMiner”. The family answers the mining question, while the HTTP request answers the follow-up question about what the malware requested.

5. BitRAT analysis and persistence

The BitRAT payload hash was bf7628695c2df7a3020034a065397592a1f8850e59f9a448b555bc1c8c639539. I followed this sample because the lab asked about autorun persistence, the downloaded file hash, the loader URL, a PowerShell delay and the C2 domain.

The strongest persistence clue was the registry startup entry. The sample added a dropped executable to a startup location:

C:\Users\RDhJOCNFevzX\AppData\Roaming\Ozndcoodb\Vwvix.exe

The same BitRAT analysis also tied the loader to http://107.175.247.199/loader/server.exe and identified the C2 domain as gh9st.mywire.org.

Reasoning: I kept these as separate answers because they represent different parts of the chain: persistence file name, payload hash, loader URL, sleep delay and C2 domain.

6. AsyncRAT and Telegram Bot API

The AsyncRAT payload was associated with SHA256 5ca468704e7ccb8e1b37c0f7595c54df4e2f4035345b6e442e8bd4e11c58f791. I checked it because the lab moved from the loader infrastructure to Telegram-based communication.

DNS: ripley.studio
GET: http://ripley.studio/loader/uploads/Zcpbmqlst.bmp
GET: http://107.175.247.199/loader/server.exe
DNS: gh9st.mywire.org
DNS: api.telegram.org
GET: https://api.telegram.org/bot5610920260:<redacted>/getUpdates

At this point, I had enough to separate the infrastructure roles: 107.175.247.199 hosted malware, ripley.studio served staged resources, gh9st.mywire.org was BitRAT C2, and api.telegram.org was used by AsyncRAT through a bot.

Answer: erikajohana.lopez@uptc.edu.coThis came from the Return-Path header.
Q3. What is the IP address of the server hosting the malicious file related to malware distribution?

Answer: 107.175.247.199The body link pointed to http://107.175.247.199/loader/install.exe.
Q4. Which malware family is responsible for cryptocurrency mining?

Answer: CoinMinerURLHaus and sandbox results tied the mining payload to CoinMiner.
Q5. Based on the cryptocurrency malware sample, what URL does this malware request?

Answer: http://ripley.studio/loader/uploads/Qanjttrbv.jpegThis was the specific HTTP request shown in the CoinMiner sandbox/network evidence.
Q6. Based on the BitRAT analysis, what is the executable's name in the first value added to the registry auto-run key?

Answer: Vwvix.exeVMRay showed the dropped executable path ending in Vwvix.exe under a startup persistence context.
Q7. Based on the BitRAT analysis, what is the SHA-256 hash of the file previously downloaded and added to the autorun keys?

Answer: bf7628695c2df7a3020034a065397592a1f8850e59f9a448b555bc1c8c639539This was the BitRAT payload hash tied to the autorun persistence evidence.
Q8. What is the URL in the HTTP request used by the loader to retrieve the BitRAT malware?

Q9. What is the delay in seconds caused by the PowerShell command according to the BitRAT analysis?

Q10. What is the C2 domain used by the BitRAT malware?

Answer: gh9st.mywire.orgThe BitRAT sandbox/community analysis identified this as the C2 domain.

Answer: bot5610920260The AsyncRAT network evidence showed Telegram Bot API requests using this bot identifier.
