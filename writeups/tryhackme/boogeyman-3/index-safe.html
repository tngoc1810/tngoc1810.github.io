TryHackMe SOC Investigation

Boogeyman 3 Incident Writeup

This writeup documents my investigation of the TryHackMe Boogeyman 3 room. I followed the evidence from a phishing email to endpoint execution, persistence, suspicious network traffic, credential activity, lateral movement, domain controller activity, and final impact staging.

Overview

The case started when the CEO of Quick Logistics LLC reported a suspicious financial email. The attachment looked like a normal PDF, but the investigation showed that it was actually a disguised payload delivered through a mounted image. I used Kibana to follow the process chain and build the full timeline.

Methodology

  1. Scoped the investigation time range to August 29 and August 30, 2023.
  2. Added useful columns in Kibana such as time, process name, parent process, command line, event ID, PID and parent PID.
  3. Searched for the suspicious attachment name and followed related process creation events.
  4. Pivoted from local execution to persistence, network activity, account activity, and remote execution.
  5. Mapped the behavior to MITRE ATT&CK and collected IOCs for the final report.

Step-by-step analysis

1. Phishing email

The email used urgency and financial wording. This made it more likely that the CEO would open the attachment. The sender appeared internal, which increased trust.

2. Disguised attachment

The downloaded file looked like a PDF, but Windows identified it as a mounted image. Inside it, the apparent PDF was actually an HTML Application. This explained why user interaction could lead to code execution.

3. Process chain

The first important event showed File Explorer launching the HTA handler. This was strong evidence of user-triggered execution. From there, the payload copied another file and executed it through a trusted Windows binary.

4. Persistence

The scheduled task named Review was created to rerun the staged payload. This showed that the attacker wanted continued access after the first click.

5. Network activity

After execution, one external HTTP endpoint appeared heavily in destination IP statistics. Because it was tied to suspicious processes, I treated it as a likely command-and-control endpoint.

6. Privilege and credential activity

Later events showed hidden PowerShell and a parent process pattern associated with UAC bypass. After gaining higher privileges, the attacker collected and reused credential material to move deeper into the environment.

7. Lateral movement

The attacker accessed a remote share, read an automation script, and then used discovered credentials to run remote commands on WKSTN-1327. On the target machine, the parent process was wsmprovhost.exe, which matched PowerShell Remoting behavior.

8. Domain controller and impact stage

The attacker eventually reached the domain controller and attempted directory credential access. Shortly after that, the attacker downloaded a final impact binary, showing the attack moving into the impact preparation stage.

Lessons learned

  • Always check the real file type, not only the visible filename.
  • Parent process relationships are very useful for rebuilding an attack chain.
  • Scheduled tasks, hidden PowerShell, and unusual trusted Windows binaries are high-value pivots.
  • Credential access and lateral movement should be investigated together.
  • Domain controller activity should be treated as critical.

Conclusion

This room was a good practice case for SOC investigation. It started with a simple phishing report but expanded into a full intrusion chain. By following evidence step by step, I was able to explain how the attacker moved from initial execution to persistence, network activity, credential abuse, lateral movement, domain compromise, and impact staging.
