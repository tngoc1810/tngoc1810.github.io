Misk Nocturne Engine

Reverse engineering a Windows PE64 checker where the visible flag is bait and the real validation is hidden behind eight state tables, a memory byte and a 42-byte target array.

Challenge hint: “The machine has only eight states, but it remembers every wrong transition.” I treated this as a direct hint that the checker would combine a small state value with another memory variable.

Executive Summary

The binary exposes the string 0xV01D{eight_states_are_not_enough}, but that string is a decoy. The real checker requires a 42-byte input and transforms each character through five rounds of table lookups. The selected table depends on an 8-state variable, while a separate memory byte is carried between characters.

After extracting the target array and transition table pointers from .rdata, I reimplemented the checker in Python and brute-forced the valid input one character at a time.

Initial Triage

$ unzip 3kfpga.zip misk_automaton.exe $ file misk_automaton.exe misk_automaton.exe: PE32+ executable for MS Windows 5.02 (console), x86-64

The file is a 64-bit Windows PE executable. That matters because the disassembly should be read with the Windows x64 calling convention in mind. More importantly, it means target arrays and table pointers must be extracted by converting virtual addresses to file offsets.

Reverse mindset: the challenge name and hint both point to state-machine logic. So I focused less on UI/output strings and more on tables, target bytes and state updates.

Fake Flag and Bait Strings

The first suspicious result came from strings.

$ strings -a misk_automaton.exe idle stalled awake operator note: the visible state names are bait 0xV01D{eight_states_are_not_enough}

The visible flag looks tempting, but the nearby operator note explicitly warns that visible state names are bait. That does not automatically prove the flag is fake, but it is enough reason to verify references in the disassembly.

Why I did not submit it: a hard reverse challenge often includes fake flags for string scanners. A flag-like string is only trustworthy if it is reachable through the normal success path or participates in the real check.

Target Data in .rdata

Checker Logic

The validation can be represented with this pseudocode:

if (strlen(input) != 42) { fail(); } state = 3; memory = 0x41; for (i = 0; i < 42; i++) { x = ((i * 9 + memory) & 0xff) ^ input[i]; for (j = 0; j < 5; j++) { table_id = (state + j) & 7; x = table[table_id][x]; memory = (19 * state + x + j + memory) & 0xff; state = (5 * state + x + i + j) & 7; } out = x ^ memory; if (out != target[i]) { fail(); } } success();
Key insight: the state space is small because state is masked with & 7. The memory byte is larger, but still deterministic. That means a solver can try characters, update state and memory exactly like the binary, and discard wrong branches early.

Solver

I wrote a Python solver that parses the PE sections, converts VAs to file offsets, extracts the target and tables, then searches printable ASCII characters.

View Solver Script
import struct

BIN = "misk_automaton.exe"
BASE = 0x140000000

data = open(BIN, "rb").read()

pe = struct.unpack_from("<I", data, 0x3c)[0]
num_sections = struct.unpack_from("<H", data, pe + 6)[0]
opt_size = struct.unpack_from("<H", data, pe + 20)[0]
sec_off = pe + 24 + opt_size

sections = []
for i in range(num_sections):
    off = sec_off + i * 40
    name = data[off:off + 8].rstrip(b"\x00").decode(errors="ignore")
    vsize, vaddr, rawsize, rawptr = struct.unpack_from("<IIII", data, off + 8)
    sections.append((name, vaddr, vsize, rawptr, rawsize))

def va_to_off(va):
    rva = va - BASE
    for name, vaddr, vsize, rawptr, rawsize in sections:
        size = max(vsize, rawsize)
        if vaddr <= rva < vaddr + size:
            return rawptr + (rva - vaddr)
    raise ValueError(f"Cannot convert VA: {hex(va)}")

TARGET_VA = 0x140005080
PTRS_VA = 0x1400050c0

target = data[va_to_off(TARGET_VA):va_to_off(TARGET_VA) + 42]

ptrs = []
for i in range(8):
    ptr = struct.unpack_from("<Q", data, va_to_off(PTRS_VA) + i * 8)[0]
    ptrs.append(ptr)

tables = []
for ptr in ptrs:
    table = data[va_to_off(ptr):va_to_off(ptr) + 256]
    tables.append(table)

def check_one_char(i, ch, state, memory):
    x = ((i * 9 + memory) & 0xff) ^ ch
    x &= 0xff

    for j in range(5):
        table_id = (state + j) & 7
        x = tables[table_id][x]
        memory = (19 * state + x + j + memory) & 0xff
        state = (5 * state + x + i + j) & 7

    out = x ^ memory
    return out & 0xff, state, memory

allowed = bytes(range(32, 127))
candidates = [(b"", 3, 0x41)]

for i, expected in enumerate(target):
    new_candidates = []
    for prefix, state, memory in candidates:
        for ch in allowed:
            out, new_state, new_memory = check_one_char(i, ch, state, memory)
            if out == expected:
                new_candidates.append((prefix + bytes([ch]), new_state, new_memory))
    candidates = new_candidates
    print(f"index={i}, candidates={len(candidates)}")

print("\nFinal candidates:")
for flag, state, memory in candidates:
    print(flag.decode(errors="replace"), "state =", state, "memory =", memory)
$ python3 solve.py Final candidates: 0xV01D{the_automaton_reads_between_states; state = 0 memory = 221 0xV01D{the_automaton_reads_between_states} state = 0 memory = 221

The first candidate ends with ;, which does not match the expected flag format. The second candidate is the valid flag.

Lessons Learned

  • Do not trust visible strings immediately in reverse engineering challenges.
  • Fake flags must be verified against the actual program logic.
  • Challenge hints often describe the algorithm, not just the story.
  • When lookup tables are involved, emulation is usually better than manual table analysis.
  • Small state spaces make brute forcing practical when the checker can be reproduced accurately.
Final conclusion: the checker is an eight-state automaton with a carried memory byte. By extracting the target and transition tables, then emulating the checker, the real flag becomes recoverable.
