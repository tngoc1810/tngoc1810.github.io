Shadow Trace: Static Malware Triage and Alert Correlation

A practical malware-analysis style walkthrough of a suspicious Windows updater. I used static triage, PE metadata, imports, FLOSS, capa, VirusTotal and alert decoding to identify IOCs and explain the behavior in plain English.

Goal: identify the suspicious binary, extract useful IOCs, understand its likely capabilities, and correlate the static findings with alert data.

Scenario Overview

The case begins with a file named windows-update.exe found on a user machine. At first glance, the name tries to look like a legitimate updater. My job was to avoid jumping to conclusions and instead build confidence from evidence: file metadata, hashes, PE structure, imported APIs, extracted strings, malware capabilities and EDR-style alerts.

Suspicious file → PE triage → Hashing → VirusTotal enrichment → Imports → FLOSS strings → capa capability mapping → Alert decoding → IOC list → Response actions

How I Approached the Case

  1. Identify the sample first. I collected file type, architecture, hashes and size before thinking about behavior.
  2. Do static analysis before execution. I inspected PE metadata, imports and strings without running the binary.
  3. Separate evidence from assumptions. A suspicious import is only a clue. It becomes stronger when it matches strings, capa output or alerts.
  4. Use tools for different questions. Detect It Easy and PE-bear explain the PE file. FLOSS reveals strings. capa summarizes capabilities. CyberChef decodes hidden values.
  5. Correlate alerts back to the malware evidence. Alert URLs and download behavior should be compared with the IOCs found during file analysis.

1. Initial File Triage

I started with Detect It Easy. My first question was simple: what kind of file is this? The tool identified the sample as a 64-bit Windows PE console executable compiled with Microsoft Visual C/C++.

The filename is also important. windows-update.exe sounds trustworthy, but the path was the user Desktop, not a normal Windows system directory. I treated that as masquerading: a legitimate-looking name used in an unusual location.

Analyst thought: architecture and compiler do not prove malware, but they tell me which analysis path to use. Here, PE64 / AMD64 means this is a 64-bit Windows executable.

2. VirusTotal Enrichment

After collecting the SHA-256 locally, I checked the hash on VirusTotal. I made sure the hash on VirusTotal matched my local sample before trusting the report.

The detection ratio was high: many vendors flagged the file as malicious. This is not a replacement for analysis, but it increases confidence that the sample deserves deeper review.

The community section showed a THOR / VALHALLA YARA match for suspicious command-line hacktool strings. I treated that as a generic warning, not a family name. It told me to pay attention to strings and command-like behavior.

3. PE Structure Review

Next, I used PE-bear to inspect the internal PE structure. I wanted to know whether the file looked packed or malformed. The section names were common for a Visual Studio PE file: .text, .rdata, .data, .pdata, .rsrc and .reloc.

The section layout did not show obvious packer markers like UPX0, UPX1, .vmp or random section names. The raw and virtual sizes also looked close enough that I expected normal static strings to be useful.

Analyst thought: normal-looking sections do not mean the file is safe. It only means the sample is probably not heavily packed, so imports and strings may be readable.

4. Import Analysis: What Can the Binary Do?

Import analysis is where the file started to look more like malware. I grouped APIs by capability instead of reading every function as an isolated item.

The KERNEL32 imports included process discovery and control functions such as CreateToolhelp32Snapshot, Process32First, Process32Next, OpenProcess and TerminateProcess. This suggests the program can inspect running processes and possibly terminate selected ones. I also noticed IsDebuggerPresent, which is a basic anti-debugging clue.

ShellExecuteA matters because it can open URLs, execute files or launch external resources through the Windows shell. SHGetKnownFolderPath can be used to resolve known user folders such as Desktop, Downloads or AppData.

WS2_32.dll is the Windows Sockets library. Seeing it imported tells me the program can use socket-based network communication. Functions such as getaddrinfo support resolving a domain name before connecting.

The WININET imports were even more direct: InternetOpenA, InternetConnectA, HttpOpenRequestA and HttpSendRequestA form a normal HTTP request chain. This supported the idea that the binary may contact a remote server over HTTP.

Capability hypothesis: the sample can perform HTTP communication, socket communication, process discovery, process termination, external execution and basic anti-analysis checks.

5. FLOSS String Extraction

After imports, I used FLOSS to extract strings. FLOSS is useful because it can find normal strings and sometimes strings built on the stack or decoded at runtime. In this sample, the important values were visible in static strings.

The strings revealed the main URL IOC:

http://tryhatme.com/update/security-update.exe

The same area also showed downloader-like messages such as Downloading to:, Download failed, Executed file, Deleting file and File deleted. This lined up perfectly with the WININET and ShellExecute imports.

Other strings looked like C2 commands or features: alive, sysinfo, exfiltrate, kill_proc, connectssh, miner, pdf_crypt and dw_exec. I also saw security-related process names such as MsMpEng.exe, Taskmgr.exe, avp.exe and avg.exe. Combined with the process termination APIs, this strongly suggests possible defense evasion behavior.

The PDB path was also useful:

C:\Users\User\source\repos\windows-update\x64\Release\windows-update.pdb

This is not an IOC I would block on, but it is useful context. It confirms a Visual Studio build path and project name.

One suspicious Base64-looking value decoded to the room flag:

VEhNe3lvdV9nMHRfc29tZV9JT0NzX2ZyaWVuZH0=
→ THM{you_g0t_some_IOCs_friend}

6. capa Capability Analysis

I then used capa to validate the behavior at a higher level. capa does not replace manual analysis, but it is excellent for summarizing what a binary is capable of doing.

capa identified communication and command-and-control style behaviors, including sending and receiving data, DNS resolution, HTTP request creation, URL download and socket communication. It also identified process enumeration, process termination, file reading, writing and deletion, hostname collection and runtime API linking.

Correlation: capa said the binary can download URLs; FLOSS showed security-update.exe; imports showed WININET; alerts later showed suspicious downloads. These independent clues all support the same behavior.

7. Alert Correlation

After file analysis, I moved to the alert table. I read alerts by process, rule, command and encoded data. The goal was not only to answer the task questions, but also to understand what the alert was really showing.

The first alert was triggered by powershell.exe. The command used System.Net.WebClient.DownloadString() and then piped the result into IEX. This is a classic PowerShell download cradle: download code from a remote URL and execute it directly in memory.

https://tryhatme.com/dev/main.exe

The second alert was triggered by chrome.exe. Instead of storing the URL as plain text, the JavaScript used decimal character codes and converted them back to a string with String.fromCharCode. After that, it used fetch(), created a Blob and forced a browser download.

https://reallysecureupdate.tryhatme.com/update.exe
Saved file: test.txt
Analyst thought: both alerts hide URLs with simple encoding. PowerShell used Base64. Browser JavaScript used decimal ASCII values. Encoding is not encryption, but it is enough to hide the URL from casual reading.

10. SOC Response Recommendations

  • Isolate the affected host while reviewing process and network activity.
  • Block or monitor the sample hash in EDR.
  • Search proxy, DNS and firewall logs for the extracted domains and URLs.
  • Hunt for security-update.exe, main.exe, update.exe and test.txt across endpoints.
  • Review PowerShell execution logs for DownloadString, IEX and Base64-decoded URLs.
  • Review browser download history and web cache for the Chrome-based download.
  • Check whether security process termination attempts occurred, especially around Defender or Task Manager process names.

Conclusion

This sample is best described as a suspicious downloader/backdoor-style Windows PE. The strongest evidence is the correlation between imports, extracted strings, capa findings and alerts. The binary imports HTTP and socket libraries, contains downloader and C2-like strings, references process termination targets, exposes network IOCs and matches critical alert behavior. The investigation shows why malware analysis should not rely on one tool: confidence comes from multiple evidence sources telling the same story.
