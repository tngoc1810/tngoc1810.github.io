Boogeyman 3: SOC Investigation Writeup

This writeup documents my investigation of the TryHackMe Boogeyman 3 room. I followed the evidence like a SOC analyst: start with the reported email, build a hypothesis, pivot through logs, connect events into a timeline, and explain why each finding mattered.

ElasticKibanaSysmonPowerShellWinRMDFIR

Overview

Quick Logistics LLC was targeted again by the Boogeyman threat group. The case began when the CEO reported a suspicious financial email. At first, the attachment looked like a normal PDF, but the evidence showed a disguised payload delivered through a mounted image. The investigation then expanded from the first workstation to a second workstation and finally to the domain controller.

The main SOC lesson in this room was that every event needs context. A process name alone is not enough. I had to compare the parent process, command line, timestamp, file path, user context, and network activity to understand the real attack chain.

Methodology

  1. Scope the time: I set the investigation window to August 29 and August 30, 2023.
  2. Prepare Kibana: I added fields for process name, parent process, command line, event ID, PID, parent PID, and PowerShell text.
  3. Start from the known clue: I searched for the fake financial document and followed the related process events.
  4. Follow the tree: I used parent-child relationships to rebuild the execution chain.
  5. Pivot carefully: I moved from file activity to process activity, then to network and account activity.
  6. Build the story: I combined the evidence into one timeline instead of treating logs as isolated events.

Step-by-step analysis

1. Phishing email

The first screenshot shows the phishing email. The subject used urgency, and the body asked the CEO to review an important financial document. This is a common social engineering pattern because it pushes the user to act quickly.

2. File type mismatch

The file in Downloads looked like a PDF, but Windows identified it as a disc image. This mismatch was the first strong technical clue. I treated the attachment as suspicious because the real file type did not match the visible name.

Inside the mounted image, the file still looked like a PDF, but its type was HTML Application. This explained how opening it could trigger script-based execution.

Analyst thought: At this point, I did not jump to the final conclusion. I only had a strong hypothesis: the PDF-looking file was a disguised execution payload.

3. Time range and fields

I scoped Kibana to the full suspected incident window. This prevented me from missing early or late activity around the compromise.

I also selected investigation-friendly columns. Parent process and command line fields were especially important because this case depended heavily on process relationships.

4. Initial execution chain

The first confirmed execution event showed File Explorer launching the Windows HTA handler. This mattered because File Explorer as the parent process suggested user interaction, while the child process showed how the disguised payload executed.

5. Stage 1 file copy and execution

The payload then copied a second file from the mounted image into the user's local temporary directory. This looked like staging: moving a payload from the delivery container into a writable local path.

Next, the staged file was executed through a trusted Windows binary. The file extension looked like ordinary data, but the behavior showed it was being treated like executable content.

6. Persistence

The attacker created a scheduled task named Review. The name blended with the business theme of document review, but the task behavior showed persistence.

The expanded event confirmed the task name and helped explain how the payload would run again later.

7. Network pivot

After execution and persistence, I pivoted to network fields. One external destination dominated the results. In a SOC workflow, this is where I ask: which process created this traffic, and did it happen after suspicious execution?

The detailed network view showed repeated HTTP traffic from the victim host to the same external endpoint. Because it was tied to suspicious execution, I treated it as likely attacker-controlled infrastructure.

8. Elevation and hidden PowerShell

Later, I found hidden PowerShell with a suspicious parent process. This pointed to an elevation attempt. The important thing here was not only PowerShell, but the full process relationship and the hidden execution flags.

I used quick research to validate why this parent process was suspicious in this context. This is normal in SOC work: Windows has many legitimate binaries, so context decides whether the behavior is suspicious.

9. Encoded PowerShell stage

The next PowerShell event used encoded content. Encoded PowerShell is not automatically malicious, but in this chain it followed a suspicious payload and elevation pattern. The decoded behavior showed remote stage retrieval and evasion-oriented logic.

10. Identity-focused activity

After the attacker gained stronger access, the logs showed tool download and identity abuse behavior. This was the point where the investigation moved from endpoint execution into account compromise and movement preparation.

The following events showed reuse of collected identity material. From a SOC perspective, this is important because it can explain why the attacker was able to move beyond the first machine.

11. Remote share access and lateral movement

The attacker accessed a remote share and read an automation script. Automation scripts are valuable during intrusions because they may reveal internal hosts, paths, or account material.

After reading the file, the attacker used newly discovered credentials to run a remote command on WKSTN-1327. This was the first clear lateral movement point.

On the second machine, the parent process was wsmprovhost.exe. That parent process is a strong clue for PowerShell Remoting / WinRM execution.

12. Second host activity

On the second host, the same intrusion pattern continued. The attacker used the new access to perform more identity-focused activity and prepare for deeper access.

13. Domain controller activity

The attacker eventually reached DC01.quicklogistics.org. Activity on a domain controller is always high severity because it can affect the whole environment, not only one workstation.

14. Final impact staging

After the domain controller activity, the attacker downloaded a final binary. This showed the intrusion moving into the impact preparation phase.

Lessons learned

  • Do not trust visible filenames. Check the real file type and execution behavior.
  • Parent-child process relationships are one of the fastest ways to rebuild an attack chain.
  • Suspicious PowerShell becomes much stronger evidence when connected to a suspicious parent process and timeline.
  • Remote share access can reveal the next step of lateral movement.
  • Domain controller activity should be treated as critical.

Conclusion

The Boogeyman 3 incident started with a single phishing email but expanded into a full intrusion chain. By following evidence step by step, I was able to connect the fake attachment, HTA execution, staged payload, persistence, network traffic, account activity, lateral movement, domain controller access, and final impact preparation into one clear timeline.

This room helped me practice the most important SOC habit: do not analyze logs in isolation. Connect them, validate each pivot, and explain why every event matters.
