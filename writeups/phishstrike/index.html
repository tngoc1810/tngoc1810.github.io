Executive Summary

The case started with a phishing email sent to an educational institution. The lure claimed that a commercial purchase worth $625,000 had been completed and invited the recipient to review an invoice. Header analysis showed weak authentication: SPF softfail, DKIM fail, and DMARC none. The email body contained a direct executable download URL instead of a real invoice: hxxp://107.175.247.199/loader/install.exe.

Threat intelligence confirmed that the link was a malware delivery endpoint. URLHaus showed historical payloads associated with CoinMiner, AsyncRAT, and BitRAT. Sandbox reports showed HTTP retrieval of staged resources from ripley.studio and 107.175.247.199, registry startup persistence for BitRAT, and AsyncRAT communication through Telegram Bot API.

Final assessment: this was a phishing-to-malware delivery chain using a purchase receipt lure, weak sender authentication, direct EXE download, staged payload retrieval, persistence, and C2 infrastructure.

Lab Scenario

As a cybersecurity analyst at an educational institution, I received an alert about a phishing email targeting faculty members. The message appeared to come from a trusted contact and claimed a high-value purchase. My goal was to investigate the raw email, identify authentication anomalies, inspect the embedded link, enrich suspicious infrastructure, and extract actionable IOCs.

1. Email Header Analysis

The visible sender was ERIKA JOHANA LOPEZ VALENTE <erikajohana.lopez@uptc.edu.co>. The message used undisclosed-recipients, which is common in bulk phishing campaigns because recipients are hidden from each other.

ArtifactValue
Fromerikajohana.lopez@uptc.edu.co
Return-Patherikajohana.lopez@uptc.edu.co
SubjectCOMMERCIAL PURCHASE RECEIPT ONLINE 27 NOV
Recipient Styleundisclosed-recipients
Message-ID Domainmail.gmail.com

The important authentication result was not that the sender domain looked suspicious in VirusTotal. The important point was that the email failed or weakened sender validation in context.

spf=softfail (sender IP is 18.208.22.104)
dkim=fail
smtp.mailfrom=uptc.edu.co
header.d=uptc.edu.co
dmarc=none

The sender IP associated with the SPF softfail and DKIM fail combination was 18.208.22.104. VirusTotal did not flag this IP, but a clean reputation score does not override the email authentication problem. The sender IP was not clearly authorized to send on behalf of uptc.edu.co.

SOC mindset: do not judge the email only by the reputation of the visible sender domain. Email investigation needs correlation between From, Return-Path, Received, SPF, DKIM, DMARC, and the body URL.

2. Malicious Link Discovery

After scrolling through the email body, the actual invoice link pointed to a raw IP address and downloaded a Windows executable:

http://107.175.247.199/loader/install.exe

This is a strong phishing indicator. A legitimate invoice should not be delivered as a direct .exe from a raw IP address. The path /loader/install.exe also suggests staged malware delivery rather than a normal business document.

3. Threat Intelligence Enrichment

The sender domain uptc.edu.co was not flagged as malicious by VirusTotal, and the SPF sender IP 18.208.22.104 also appeared clean. However, the payload-hosting IP 107.175.247.199 was flagged by multiple vendors and had a strong negative community reputation.

URLHaus confirmed that hxxp://107.175.247.199/loader/install.exe was a malware download URL. Historical records showed that the same URL delivered multiple executable payloads over time.

Key pivot: the URL was not tied to a single static file. It behaved like a loader endpoint that distributed CoinMiner, AsyncRAT, and BitRAT payloads at different times.

4. CoinMiner Sample Behavior

The CoinMiner payload hash was 453fb1c4b3b48361fa8a67dcedf1eaec39449cb5a146a7770c63d1dc0d7562f0. VirusTotal and Joe Sandbox confirmed malicious behavior. The sandbox process tree showed the sample launching PowerShell with an encoded command.

Encoded PowerShell:
UwB0AGEAcgB0AC0AUwBsAGUAZQBwACAALQBTAGUAYwBvAG4AZABzACAANQAwAA==

Decoded command:
Start-Sleep -Seconds 50

The delay is likely an anti-analysis technique. Malware often sleeps before executing its main behavior because sandboxes may only monitor for a short time. Joe Sandbox also observed HTTP traffic from the malware process to staged resources.

GET http://ripley.studio/loader/uploads/Qanjttrbv.jpeg
GET http://107.175.247.199/loader/server.exe

The .jpeg extension should not be assumed benign. In this case, the sandbox context suggests that it may be a disguised payload or configuration resource used by the loader.

5. BitRAT Analysis and Persistence

The BitRAT payload was the most useful sample for persistence analysis. The hash bf7628695c2df7a3020034a065397592a1f8850e59f9a448b555bc1c8c639539 was detected as malicious by many VirusTotal engines, and Joe Sandbox gave it a malicious verdict.

Sandbox signatures showed behavior consistent with a RAT or loader:

  • Encoded PowerShell command execution.
  • Dropped file detection.
  • Snort IDS alert for network traffic.
  • YARA detection for Costura Assembly Loader.
  • Potential .NET unpacking.
  • PE injection into another process.
  • Volume/environment checks.

VMRay confirmed persistence by showing that the malware added a dropped executable to Windows startup through the registry.

Persistence method: Windows startup via registry
Dropped startup file: C:\Users\RDhJOCNFevzX\AppData\Roaming\Ozndcoodb\Vwvix.exe
Executable name: Vwvix.exe

Community and sandbox intelligence also identified the BitRAT C2 domain:

gh9st.mywire.org
Impact: registry startup persistence means the malware can run again after user logon or system restart, preserving attacker access even if the first process is terminated.

6. AsyncRAT Network and Telegram Bot API

The AsyncRAT payload was associated with SHA256 5ca468704e7ccb8e1b37c0f7595c54df4e2f4035345b6e442e8bd4e11c58f791. Its network behavior overlapped with the same staging infrastructure and also revealed Telegram Bot API usage.

DNS: ripley.studio
GET: http://ripley.studio/loader/uploads/Zcpbmqlst.bmp
GET: http://107.175.247.199/loader/server.exe
DNS: gh9st.mywire.org
DNS: api.telegram.org
GET: https://api.telegram.org/bot5610920260:<redacted>/getUpdates

The observed Telegram bot identifier was bot5610920260. The endpoint /getUpdates is consistent with a bot polling Telegram for new messages or instructions. In malware analysis, this may indicate command retrieval, notification, or exfiltration-related communication through Telegram infrastructure.

Conclusion

This investigation shows why phishing triage should combine header analysis, body URL inspection, and historical threat intelligence. The sender domain reputation alone looked clean, but the authentication results were weak and the email body led to a direct executable download from known malicious infrastructure. URLHaus, VirusTotal, Joe Sandbox, VMRay, and CyberChef together revealed a full malware delivery chain involving CoinMiner, BitRAT, AsyncRAT, registry persistence, staged payload retrieval, a BitRAT C2 domain, and Telegram Bot API usage.
