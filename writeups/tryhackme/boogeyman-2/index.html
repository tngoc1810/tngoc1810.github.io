Boogeyman 2: Following the Evidence from Phishing Email to Persistence

This writeup documents the investigation flow: what I saw, what I thought it meant, which command I used next, and how each piece of evidence led to the final answers.

Room
Boogeyman 2
Platform
TryHackMe
Focus
SOC / DFIR
Artifacts
EML, DOC, Memory Dump
Tools
olevba, Volatility, strings
Core idea
Correlate, do not guess
Investigation goal: prove how the HR workstation was compromised, identify what the Word document executed, confirm it in memory, find the binary and C2 connection, then recover how the attacker maintained access.

Scenario Overview

Maxine Beck, an HR employee at Quick Logistics LLC, received a resume application email. A resume sent to HR is believable, so I treated the attachment carefully instead of assuming it was obviously malicious from the beginning.

Email → Word attachment → Macro → JavaScript stage → EXE payload → C2 callback → Scheduled task persistence

How I Approached the Case

  1. Start from the visible artifact. Identify sender, recipient, attachment and subject.
  2. Inspect the file safely. Use file, hashes and olevba instead of opening the document.
  3. Validate in memory. Use Volatility to prove the process chain actually executed.
  4. Correlate before concluding. A string alone is only a clue. A string plus process/file/network evidence becomes much stronger.

1. Email Triage

From: westaylor23@outlook.com
To: maxine.beck@quicklogisticsorg.onmicrosoft.com
Subject: Resume - Application for Junior IT Analyst Role
Attachment: resume_WesleyTaylor.doc

I took the sender and victim directly from the email. The attachment was the obvious next artifact because it was the object the victim would open.

2. Attachment Triage

file Resume_WesleyTaylor.doc
sha256sum Resume_WesleyTaylor.doc
md5sum Resume_WesleyTaylor.doc

The document was a legacy Word/OLE file, which can contain VBA macro streams. The MD5 used later in the answer summary was:

52c4384a0b9e248b95804352ebec6c5b

The reputation results and sandbox behavior were useful hints, especially the winword.exe → wscript.exe chain, but I still needed primary evidence from the document and memory dump.

3. Macro Analysis

olevba Resume_WesleyTaylor.doc

The macro used AutoOpen(), so it was designed to run when the document opened. It downloaded update.png, saved the response body as C:\ProgramData\update.js, then executed it with wscript.exe.

Stage 2 URL:
https://files.boogeymanisback.lol/aa2a9c53cbb80416d3b47d85538d9971/update.png

Saved payload:
C:\ProgramData\update.js

Executed by:
wscript.exe

The extension mismatch mattered: the remote payload looked like an image, but the macro saved and executed it as JavaScript.

4. Memory Validation

vol -f WKSTN-2961.raw windows.pstree
OUTLOOK.EXE PID 1440
└── WINWORD.EXE PID 1124
    └── wscript.exe PID 4260
        └── updater.exe PID 6216

This confirmed the macro behavior at runtime. Word launched wscript.exe, and the script led to updater.exe.

vol -f WKSTN-2961.raw windows.cmdline > cmdline.txt
cat cmdline.txt
Attachment runtime path:
C:\Users\maxine.beck\AppData\Local\Microsoft\Windows\INetCache\Content.Outlook\WQHGCZFT\Resume_WesleyTaylor (002).doc

Stage 2 command:
wscript.exe C:\ProgramData\update.js

Malicious binary path:
C:\Windows\Tasks\updater.exe

5. Stage 2 Binary

vol -f WKSTN-2961.raw windows.filescan > filescan.txt
grep -i "update" filescan.txt

filescan showed update[1].png, update[1].exe and \Windows\Tasks\updater.exe. That suggested the JavaScript stage downloaded an executable.

strings WKSTN-2961.raw | grep -i "boogeymanisback"
https://files.boogeymanisback.lol/aa2a9c53cbb80416d3b47d85538d9971/update.exe

6. C2 Analysis

vol -f WKSTN-2961.raw windows.netscan > netscan.txt
grep "6216" netscan.txt
Process: updater.exe
PID: 6216
Remote endpoint: 128.199.95.189:8080

I treated this as confirmed C2 because the endpoint was connected to the malicious process, not because of a reputation score alone.

7. Persistence

windows.cmdline did not show the full task creation command, so I moved to raw strings. On Windows, Unicode strings are often more useful for command-line artifacts.

strings -a -n 5 WKSTN-2961.raw > ascii_strings.txt
strings -a -el -n 5 WKSTN-2961.raw > unicode_strings.txt

grep -ni "schtasks" ascii_strings.txt
grep -ni "schtasks" unicode_strings.txt
schtasks /Create /F /SC DAILY /ST 09:00 /TN Updater /TR "C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe -NonI -W hidden -c \"IEX ([Text.Encoding]::UNICODE.GetString([Convert]::FromBase64String((gp HKCU:\Software\Microsoft\Windows\CurrentVersion debug).debug)))\""

This creates a task named Updater, runs it every day at 09:00, starts hidden PowerShell, reads an encoded payload from the registry and executes it with IEX.

How the Answers Were Derived

Email answers

The sender and victim came from the phishing email header/preview. The attachment name came from the email and the local artifact.

Sender: westaylor23@outlook.com
Victim: maxine.beck@quicklogisticsorg.onmicrosoft.com
Attachment: resume_WesleyTaylor.doc

Document and macro answers

The MD5 came from md5sum. The stage 2 URL, local path and executor came from olevba.

md5sum Resume_WesleyTaylor.doc
olevba Resume_WesleyTaylor.doc

Memory answers

windows.pstree gave the process chain and PID/PPID. windows.cmdline gave the exact runtime paths.

vol -f WKSTN-2961.raw windows.pstree
vol -f WKSTN-2961.raw windows.cmdline

Binary and C2 answers

filescan and strings exposed the executable download. netscan tied updater.exe PID 6216 to 128.199.95.189:8080.

vol -f WKSTN-2961.raw windows.filescan
strings WKSTN-2961.raw | grep -i "boogeymanisback"
vol -f WKSTN-2961.raw windows.netscan

Persistence answer

The scheduled task command was recovered from Unicode strings because the original schtasks.exe process likely exited before the memory capture.

strings -a -el -n 5 WKSTN-2961.raw > unicode_strings.txt
grep -ni "schtasks" unicode_strings.txt

What I Learned

  • Build the attack chain first, then answer the questions.
  • Macro output explains intent; memory output proves execution.
  • Parent-child process relationships are strong evidence for malicious document execution.
  • Raw strings are useful but noisy. Only promote strings to confirmed IOCs when they correlate with files, processes or network evidence.
  • Scheduled task persistence can hide the real payload in the registry and execute it through hidden PowerShell.
