Paper Lantern — CRT-RSA Fault Attack

A beginner-friendly technical thought journal for a pwn challenge where memory/state corruption is used to break CRT-RSA signing, recover the private key, forge an unsafe capsule signature, and print the flag.

Category: Pwn / Crypto Bug: Comment braid overflow Crypto: CRT-RSA FDH Result: Private key recovery

Table of Contents

1. Summary 2. Initial Recon 3. Understanding the Capsule Protocol 4. Discovering the Unsafe Opcode Barrier 5. Finding the Comment Braid Bug 6. Turning Memory Corruption into a CRT-RSA Fault 7. Factoring n with gcd 8. Forging the Flag Capsule Signature 9. Final Output 10. Lessons Learned

1. Summary

At first glance, this looked like a normal pwn challenge. However, the real exploitation path was not a classic stack overflow or ROP chain. The binary used a custom capsule protocol, and every capsule had to be signed before it could run.

The target flag logic was hidden behind an unsafe opcode, 0x7f. The server refused to sign capsules containing this opcode. Therefore, the real goal became: recover the RSA private key, forge a valid signature for the unsafe capsule, and run it.

Core idea: comment parser bug → corrupt CRT signing state → obtain one correct and one faulty RSA signature → factor n using gcd → recover private key → forge signature for opcode 0x7f → flag.

2. Initial Recon

Newbie StateCommand / InputExpert Analysis
I started with a zip file. I did not know whether this was a normal binary exploitation task, a protocol task, or a crypto-assisted pwn challenge. 
unzip "dist(5).zip" -d paper
cd paper/dist
ls -la
 The directory contained paper_lantern, capsule.py, and public_params.json. The Python helper and JSON parameters hinted that the challenge used a custom protocol and cryptographic verification.
I checked the binary type because this was still marked as pwn. 
file paper_lantern
 The file was an ELF 64-bit binary. But instead of immediately jumping into ROP, I focused on the helper script because custom protocols often hide the real attack surface.
The JSON file looked suspicious. Why would a pwn challenge give me public parameters? 
cat public_params.json
 The important fields were scheme = crt-rsa-fdh, e = 65537, signature_size = 64, and replay_buf = 72. This strongly suggested that RSA signatures and a 72-byte replay/comment buffer were part of the vulnerability.

Important observed data:

{
  "scheme": "crt-rsa-fdh",
  "e": 65537,
  "signature_size": 64,
  "replay_buf": 72
}

The word crt was the biggest hint. CRT-RSA is faster than normal RSA, but if the signing process is faulted in only one branch, a single faulty signature can reveal a factor of the RSA modulus.

3. Understanding the Capsule Protocol

Newbie StateCommand / InputExpert Analysis
I opened capsule.py and saw many constants. At first, it looked like random protocol numbers. 
sed -n '1,220p' capsule.py
 The constants described a frame protocol: HELLO, NEWCAP, APPEND, SIGN, COMMENT, and RUN.
I wanted to know the normal path before attacking anything. 
FT_NEWCAP  = 0x20
FT_APPEND  = 0x21
FT_SIGN    = 0x22
FT_COMMENT = 0x23
FT_RUN     = 0x24
 The server requires a capsule to be created, filled with records, signed, and then executed. This is an authorization model: no valid signature, no execution.
I saw normal records like text and halt, plus a raw record helper. 
payload = rec_text(b"A") + rec_halt()
 This was a safe capsule. It allowed me to test whether I understood the protocol and whether the server would return a valid signature.

Normal test output:

[*] getting normal signature
[+] good signature = <64-byte hex signature>

This output confirmed that the server signs safe capsules. The signature later became the good signature used in the CRT fault attack.

4. Discovering the Unsafe Opcode Barrier

Newbie StateCommand / InputExpert Analysis
I noticed rec_raw(kind), so I suspected there might be hidden opcodes. 
payload = rec_raw(0x7f) + rec_halt()
 Opcode 0x7f was the unsafe opcode that eventually prints the flag. The goal was to run this capsule.
I tried to ask the server to sign the unsafe capsule directly. 
c.send(FT_APPEND, rec_raw(0x7f) + rec_halt())
c.send(FT_SIGN)
 The server rejected the signing request because the capsule contained an unsafe opcode. This was the main access-control barrier.
I wanted to confirm that the binary explicitly knew about unsafe opcodes. 
strings paper_lantern | grep -i unsafe
 The string unsafe opcode confirmed that the server had a signing-time filter. The exploit needed to bypass signing, not merely craft a clever opcode.
Key mental shift: I stopped trying to make the server sign opcode 0x7f. Instead, I aimed to recover the private key and sign it myself.

5. Finding the Comment Braid Bug

Since the main capsule records were checked, I turned to the secondary surface: FT_COMMENT. Comment parsers are often treated as metadata, but they can still corrupt internal state.

Newbie StateCommand / InputExpert Analysis
I saw replay_buf = 72 and did not know why that exact number mattered. 
cat public_params.json
 A fixed buffer size in challenge metadata is usually intentional. It hints that the parser has a bounded memory area.
I inspected comment helper functions. 
comment_literal(b"\x00" * 60)
comment_braid_fill(18, 0)
 The math is the bug: 60 + 18 = 78, but the replay buffer is only 72 bytes. This creates a 6-byte overwrite past the replay seam.
I checked strings related to braid/replay to validate the hypothesis. 
strings paper_lantern | grep -i braid
strings paper_lantern | grep -i replay
 Strings such as bad replay braid, braid exceeds seam, and replay sealed showed that comment replay logic was complex and security-relevant.

Deep Dive: What is comment_braid_fill?

For beginners, imagine the replay buffer as 72 boxes numbered 0 to 71. A literal comment fills some boxes normally. Then braid_fill continues writing a span of bytes. If we already filled 60 boxes and ask braid to fill 18 more, it writes boxes 60 to 77. Boxes 72 to 77 are outside the buffer. 

valid replay buffer: [0 ... 71]
write 60 bytes:       [0 ... 59]
braid 18 bytes:       [60 ... 77]
overflow bytes:       [72 ... 77]

This is not a classic return-address overwrite. Instead, the overflow changes hidden state near the replay buffer. That hidden state controls whether CRT signing becomes faulty.

6. Turning Memory Corruption into a CRT-RSA Fault

The exploit uses the overflow to make the server produce a faulty RSA signature for the same safe capsule. One correct signature plus one faulty signature is enough to factor n if the fault affects only one CRT branch.

Newbie StateCommand / InputExpert Analysis
I had a safe capsule signature. Now I needed a faulty signature for the same capsule. 
comment = make_fault_comment(0x84)
fault_sig = get_safe_signature(comment)
 The comment does not change the capsule message. It changes the signing state, causing CRT-RSA to compute an incorrect signature.
I did not know the exact gate byte at first. 
for gate2 in range(256):
    comment = make_fault_comment(gate2)
    candidate = get_safe_signature(comment)
 Bruteforcing one byte is cheap. Each attempt is checked mathematically using gcd. If a real factor appears, the gate value was correct.
The exploit printed that the fault was triggered. 
[+] CRT fault triggered with gate2 = 0x84
 This means the server returned a structured faulty signature. The word structured matters: it is not random corruption; it leaks information about one prime factor.

Deep Dive: Why CRT-RSA faults are dangerous

RSA modulus is n = p * q. CRT-RSA signs separately modulo p and modulo q, then combines the two results. If one side is wrong and the other side is right, the difference between the good and faulty signatures becomes divisible by one of the primes. 

good_sig  = correct CRT result
fault_sig = one CRT branch is wrong

factor = gcd(abs(good_sig - fault_sig), n)

7. Factoring n with gcd

Newbie StateCommand / InputExpert Analysis
I converted both signatures from bytes to integers. 
good_int = int.from_bytes(good_sig, "big")
fault_int = int.from_bytes(fault_sig, "big")
 RSA arithmetic happens over integers modulo n. Bytes must be converted into integers before applying number theory.
I applied the magic-looking gcd formula. 
p = math.gcd(abs(good_int - fault_int), n)
q = n // p
 If 1 < p < n, then p is a non-trivial factor of n. This breaks the RSA key.

Observed terminal output:

[+] factored n
    p = 9da033fd0799399257825aff0f7ca4b7866a4db13e250be52e50d7fc3d3eb295
    q = b6f7221baafa0048953633e0193e92928aab4bf4dba728b6f713e55135c0b6b5

This was the turning point. Before this, I only had the public key. After this, I could compute the private exponent and sign anything the server would verify.

8. Forging the Flag Capsule Signature

Newbie StateCommand / InputExpert Analysis
I had p and q. I needed the RSA private exponent d. 
phi = (p - 1) * (q - 1)
d = pow(e, -1, phi)
 This is standard RSA private key recovery. Since e = 65537 and phi is known, d can be computed with modular inverse.
I needed to sign the exact message that the server verifies for the unsafe capsule. 
m = fdh_message(b"FH", n)
 The server uses FDH, so it does not sign raw record bytes directly. The unsafe capsule is serialized as b"FH": flag opcode plus halt.
I created the forged RSA signature. 
forged_int = pow(m, d, n)
forged_sig = forged_int.to_bytes(64, "big")
 This is the same mathematical operation the server would perform if it agreed to sign the unsafe capsule.
I sent the unsafe capsule with my forged signature. 
payload = rec_raw(0x7f) + rec_halt()
c.send(FT_APPEND, payload)
c.send(FT_RUN, forged_sig)
 The server verifies the signature using the public key. Since the forged signature is mathematically valid, it executes opcode 0x7f.

10. Lessons Learned

What made this pwn challenge different?

This was not a typical control-flow hijack. There was no need to overwrite RIP or build a ROP chain. The exploit corrupted internal signing state and used mathematics to turn that corruption into a private key recovery. 

comment parser bug
→ replay seam overflow
→ CRT-RSA signing fault
→ good/fault signature pair
→ gcd leaks p or q
→ RSA private key recovered
→ forge unsafe opcode signature
→ flag

Beginner takeaway

When a pwn challenge includes cryptographic signing, do not only look for shellcode or ROP. Ask whether the program signs, verifies, serializes, hashes, or stores metadata incorrectly. Sometimes the best exploit is not to control the instruction pointer, but to corrupt the trust boundary.

Recognition pattern for future CTFs

  • If you see crt-rsa, think about CRT fault attacks.
  • If you can obtain a correct and faulty signature for the same message, try gcd(good - fault, n).
  • If a parser has a fixed buffer size and mini-operations like copy, repeat, xor, or fill, check for boundary mistakes.
  • If the server refuses to sign a dangerous payload, consider recovering or forging the signing key instead of bypassing the filter directly.
