HawkEyeKeyloggerExfiltration

PCAP investigation of an invoice-themed malware infection. The important part was not only finding the malware download, but proving that the same host later sent Base64-encoded credential logs through SMTP.

PlatformCyberDefenders
CategoryNetwork Forensics
Evidencestealer.pcap
ToolsWireshark · CyberChef · VirusTotal

How I approached it

I did not start by searching for “HawkEye” directly. At the beginning, I only had a packet capture and a scenario about an accountant opening an invoice link. So my first question was simple: which internal host became noisy after the suspected download?

Once I found 10.4.10.132, I followed that host across protocols. HTTP explained the initial payload delivery. SMTP explained what happened after infection. The flow became clear only after those two parts were connected together.

Working conclusion: 10.4.10.132 / BEIJING-5CD1-PC downloaded tkraw_Protected99.exe from proforma-invoices.com. Shortly after that, the same host sent repeated SMTP messages to 23.229.162.69:587. Decoding the Base64 subject and body showed HawkEye Keylogger - Reborn v9 password logs.

Then I checked Protocol Hierarchy. HTTP had the largest data volume, so I expected a file download. SMTP was also present. That immediately got my attention because keyloggers often use email to send stolen logs.

What I thought here: HTTP probably explains how the malware arrived. SMTP probably explains how data left. I kept both protocols in scope instead of focusing only on the biggest traffic volume.

3. HTTP payload delivery

After identifying the likely victim, I filtered HTTP traffic. The suspicious request was not a PDF or a normal invoice page. It was a direct executable download from an invoice-looking domain.

GET /proforma/tkraw_Protected99.exe HTTP/1.1
Host: proforma-invoices.com
User-Agent: Mozilla/4.0 (compatible; MSIE 7.0; Windows NT 6.1; WOW64; Trident/7.0)

The server response confirmed that the content was a Windows executable download:

HTTP/1.1 200 OK
Content-Type: application/x-msdownload
Content-Length: 202472
Server: LiteSpeed

Following the HTTP stream showed MZ, PE, and This program cannot be run in DOS mode. That was enough to say the response body was a PE executable, not an invoice document.

What I thought here: The domain name was trying to look like invoice infrastructure, but the file type did not match the business story. An accountant clicking an invoice link should not receive a PE executable.

Lessons learned

This lab is a good reminder that the largest protocol by bytes is not always the only important one. HTTP showed the malware arriving, but SMTP showed the actual data loss.

The second lesson is that Base64 is not encryption. In this capture, the username, password, subject and body were easy to decode once I followed the SMTP stream.

Finally, repeated SMTP sessions mattered more than a single decoded message. They showed behavior: HawkEye was configured to periodically send collected password logs, which is exactly the kind of pattern worth putting into a timeline and detection idea.
