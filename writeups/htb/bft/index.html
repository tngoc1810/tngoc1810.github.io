A Windows filesystem forensics investigation using a raw NTFS $MFT artifact. I focused on how to reason from metadata: finding the downloaded ZIP, using Zone.Identifier to prove origin, locating the extracted batch stager, calculating its MFT record offset and recovering its resident content without running the file.

Investigation goal: Reconstruct how Simon Stark received and downloaded a suspicious ZIP, identify the malicious stager inside the extracted archive, determine creation metadata, recover the file content from the MFT and extract the C2 IP and port.

Scenario Overview

The lab provides a Windows NTFS $MFT artifact. That changes the way I investigate. I do not have a full disk image, memory dump, packet capture or event logs. I only have filesystem metadata, so every conclusion needs to come from names, paths, timestamps, ADS records and resident file content.

At a high level, the attacker targeted Simon Stark on February 13. Simon downloaded a ZIP file from a link received in an email. The extracted content contained a batch script that acted as a stager. My job was to prove that chain from the MFT instead of guessing from the scenario text.

What I thought first: If this started from an email link, the earliest useful artifact is probably a downloaded archive in Simon's profile. After that, I should look for extracted files with executable extensions, then check whether Windows preserved the original download URL in Zone.Identifier.

Investigation Mindset

When facing a filesystem-only artifact like $MFT, I think in pivots:

  1. Start from scenario keywords: ZIP, email link, Simon Stark and February 13.
  2. Translate them into filters: extension .zip, path simon.stark, folder Downloads and date-like strings such as 20240213.
  3. Identify the first-stage artifact: the downloaded ZIP.
  4. Pivot from archive to extraction folder: files created under a similarly named folder are likely extracted contents.
  5. Look for executable/script files: .bat, .cmd, .ps1, .js, .vbs, .exe.
  6. Check origin metadata: Zone.Identifier can reveal where the file came from.
  7. If the suspicious file is small: try resident recovery from the MFT record.

This is why the lab is useful: it teaches a repeatable forensic workflow, not only a list of answers.

Core Concepts I Learned

NTFS and the Master File Table

NTFS is the Windows filesystem that tracks file names, folder structure, permissions, timestamps and where data lives on disk. Its key metadata structure is the Master File Table, usually seen as $MFT.

I picture $MFT as the index book of an NTFS volume. For every file or folder, NTFS keeps a record containing the filename, parent directory, file size, record status, timestamps, alternate data streams and sometimes the actual file content if the file is tiny.

This matters because deleting a file does not instantly erase all forensic traces. A user may remove a suspicious file from Explorer, but the MFT record can remain long enough to reconstruct the activity.

Alternate Data Streams and Zone.Identifier

Alternate Data Streams are extra hidden streams attached to a file on NTFS. A common one is Zone.Identifier, created by Windows for downloaded files. It can contain:

[ZoneTransfer]
ZoneId=3
HostUrl=https://example.com/file.zip

ZoneId=3 generally indicates the file came from the Internet Zone. The HostUrl field is valuable because it can show the exact source URL. In a SOC investigation, that URL becomes an IOC for proxy searches, EDR scoping and blocklisting.

Why I cared: the file name tells me what landed on disk, but Zone.Identifier can tell me where it came from. That is stronger evidence than just saying “a ZIP existed in Downloads.”

0x10 vs 0x30 Timestamps

MFTECmd exposes multiple timestamp sets because NTFS stores time values in different attributes:

AttributeNameHow I used it
0x10$STANDARD_INFORMATIONUseful for normal timeline reading, but more commonly affected by normal filesystem operations and timestomping.
0x30$FILE_NAMEUseful for cross-checking because it is tied to filename and parent-directory metadata. In this lab, the answer specifically came from Created0x30.

I do not treat 0x30 as impossible to manipulate. The safer wording is: 0x30 is useful for cross-checking and can preserve a different metadata view than 0x10.

Resident Data

Each MFT record is commonly 1024 bytes. If a file is small enough, NTFS can store the content directly inside the MFT record instead of allocating separate disk clusters. This is called resident data.

The easiest analogy: normally the MFT says “the file content is stored somewhere else.” But if the file is tiny, NTFS can keep the content inside the record itself. That is why small scripts such as .bat, .cmd or .txt files can sometimes be recovered directly from $MFT.

Tools Used and Why

  • MFTECmd: Converts raw $MFT records into a structured CSV. Without it, reading the raw MFT manually would be slow and error-prone.
  • Timeline Explorer: Opens the CSV in a forensic-friendly table. It lets me filter by extension, path, filename and timestamp without destroying formatting.
  • HxD Hex Editor: Opens the raw $MFT as bytes. I used it when I needed to jump to a specific record offset and inspect resident data.

The workflow was:

$MFT raw artifact
  -> MFTECmd parses it
  -> MFT_ANALYSIS.csv
  -> Timeline Explorer filters and pivots
  -> HxD recovers raw resident content

Step 1 — Parsing $MFT with MFTECmd

I first parsed the provided $MFT file into CSV format. The command tells MFTECmd where the MFT is located, where to write the CSV output and what to name the CSV file.

MFTECmd.exe -f 'C:\Users\admin\Downloads\BFT\C\$MFT' --csv 'C:\Users\admin\Downloads\BFT\C' --csvf MFT_ANALYSIS.csv

The output showed that MFTECmd recognized the input as an MFT file and processed 171,927 file records. The resulting MFT_ANALYSIS.csv became my main evidence table.

What I checked here: I wanted to confirm the parser completed cleanly before trusting the CSV. If the parser failed or only parsed a small number of records, every later pivot would be risky.

Q1 — What ZIP File Was Downloaded?

The question gave three strong hints: Simon Stark, February 13 and ZIP file. My first filter was the Extension column for ZIP files. Then I checked whether the file lived under Simon's user profile, especially Downloads.

Reasoning: A file downloaded from a browser or email link commonly lands in C:\Users\<user>\Downloads. A ZIP directly in Downloads is more likely to be the original download than a ZIP buried inside WindowsApps or another unrelated program folder.

The matching file was:

Stage-20240213T093324-001.zip

The 20240213 in the filename matches February 13, which aligns with the scenario.

Q2 — What HostUrl Did Zone.Identifier Reveal?

After finding the ZIP, I pivoted to its Zone.Identifier ADS. This was important because it could prove where the ZIP came from, not just where it was stored locally.

The field contained a [ZoneTransfer] block with ZoneId=3 and this full HostUrl:

https://storage.googleapis.com/drive-bulk-export-anonymous/20240213T093324.039Z/4133399871716478688/a40aecd0-1cf3-4f88-b55a-e188d5c1c04f/1/c277a8b4-afa9-4d34-b8ca-e1eb5e5f983c?authuser

This URL is a strong IOC because it identifies the remote source used to deliver the initial ZIP.

Q3 — Which File Executed Malicious Code and Connected to C2?

Once the initial ZIP was identified, I assumed the next stage would be inside the extracted folder. I filtered Parent Path for Stage-20240213T093324Z-001, which looked like the extraction directory for the downloaded archive.

Reasoning: Attackers often hide scripts under innocent names such as invoice, document or report. A .bat file inside an invoice-themed folder is suspicious because batch files execute commands on Windows. It is not malicious by extension alone, but the path and later recovered content made it stand out.

The malicious stager was:

C:\Users\simon.stark\Downloads\Stage-20240213T093324Z-001\Stage\invoice\invoices\invoice.bat

I treated this as a stager because a batch file is often used as a small launcher that calls PowerShell, downloads a second-stage payload or performs simple execution logic.

Q4 — When Was the File Created on Disk?

The question specifically asked for the Created0x30 timestamp of the file from Q3. I read the value from the Created0x30 column, not from Created0x10.

Why 0x30 matters: Created0x30 comes from the $FILE_NAME attribute. It is useful for cross-checking against $STANDARD_INFORMATION timestamps and can reveal inconsistencies caused by copy, move, rename or timestomping behavior.

The file creation time on disk was:

2024-02-13 16:38:39

Q5 — Finding the Hex Offset of the MFT Record

The malicious file had Entry Number = 23436. Since each MFT record is 1024 bytes, the offset of the record inside the $MFT file can be calculated like this:

Entry Number × MFT record size
23436 × 1024 = 23998464
23998464 decimal = 0x16E3000
Precision note: This is the offset inside the $MFT file, not necessarily an absolute physical disk offset for the whole volume.

The hex offset was:

0x16E3000

Q6 — Recovering the Resident Stager and Finding the C2

Because invoice.bat was small, I checked whether its content was stored as resident data in the MFT record. I opened the raw $MFT in HxD, pressed Ctrl + G and jumped to offset 16E3000.

The recovered batch content showed a PowerShell stager. I did not need to execute it. Reading the command was enough to understand the behavior:

  • @echo off: hides command echoing in the console.
  • start /b: starts a process in the background.
  • powershell.exe: uses PowerShell as the execution engine.
  • -nop: avoids loading the user's PowerShell profile.
  • -ep bypass: bypasses PowerShell execution policy.
  • DownloadString: retrieves remote script content.
  • iex: executes the downloaded content immediately.
  • del "%~f0": attempts to delete the batch file after running.
Analyst interpretation: This is classic stager behavior. The batch file itself is small and simple. Its job is to launch PowerShell, retrieve a remote payload and reduce visible evidence by deleting itself. The actual IOC I needed was the remote IP and port embedded in the command.

The C2 address and port recovered from the resident content were:

43.204.110.203:6666

Lessons Learned

This lab taught me that Windows filesystem metadata can tell a complete story even when I do not have full logs or an executable sample. The MFT showed the downloaded ZIP, extraction path, suspicious batch file, timestamps and enough raw resident content to recover the C2 address.
