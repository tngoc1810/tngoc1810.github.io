How I read the case

I did not want to call the email malicious just because the subject looked suspicious. I started from the headers, checked whether the sender identity held up, then looked at the body to see where the user was being pushed.

The turning point was the link in the email body. A purchase receipt should not send the user to a raw-IP-hosted install.exe. That made the payload URL the center of the investigation.

My conclusion: this was a phishing email that used a fake purchase/invoice lure to deliver malware. The sender domain reputation alone was not enough; the failed authentication results, executable URL, URLHaus history, and sandbox behavior had to be read together.

1. Email headers

I first checked the visible sender, return path, recipient pattern and authentication results. The sender appeared to be from uptc.edu.co, but the authentication results did not look healthy.

The authentication result was the first technical warning sign:

spf=softfail
_dkim=fail
dmarc=none
sender IP: 18.208.22.104

2. Delivery URL

After the header check, I moved to the body. The email pretended to provide an invoice or receipt, but the link pointed straight to an executable:

http://107.175.247.199/loader/install.exe

I kept the sender domain and sender IP as context, but the payload host became the main thing to investigate.

3. Reputation check

I checked reputation next, but I was careful not to overtrust it. Clean reputation for the sender domain did not make the email safe. The question was whether the payload URL had a malware history.

4. Malware behavior from sandbox results

The threat intel results pointed to several malware families over time, including CoinMiner, BitRAT and AsyncRAT. I treated those as behavior leads rather than random labels. Then I checked the sandbox artifacts for what the samples actually did.

The useful lesson from this part was simple: one URL can serve different payloads across time. I should not assume a single family name explains the whole infrastructure.
