Misk Automaton

Reverse engineering a Windows PE64 checker that hides the real validation logic behind a fake flag, target bytes, eight transition tables and a small state machine.

Overview

The challenge archive contained one main artifact: misk_automaton.exe. The binary looked small, but it had enough structure to hide a stateful checker. The goal was to recover the real input accepted by the validation function, not to trust the first flag-looking string.

Initial Triage

$ unzip 9znj0v.zip Archive: 9znj0v.zip inflating: misk_automaton.exe $ file misk_automaton.exe misk_automaton.exe: PE32+ executable for MS Windows 5.02 (console), x86-64 (stripped to external PDB), 11 sections

The file type matters because this is a Windows x64 binary. While reading assembly, I kept the Windows x64 calling convention in mind: rcx, rdx, r8 and r9 are the first arguments.

Reverse mindset: a small checker is often data-driven. Instead of only reading branches, I looked for target arrays, lookup tables and references from the checker into .rdata.

Fake Flag in Strings

The first trap appeared immediately in the strings output.

$ strings -a misk_automaton.exe | grep -E '0xV01D|bait|operator|Correct|Wrong|usage' operator note: the visible state names are bait 0xV01D{eight_states_are_not_enough}

The string 0xV01D{eight_states_are_not_enough} looks like a flag, but it is placed next to a warning that the visible state names are bait. I needed to check where this string was referenced before trusting it.

Why I rejected it: the fake flag branch is reached only when argc == 31337. Under normal execution, the real checker requires argc == 2 and an input length of 0x2a bytes.

.rdata Analysis

In checker binaries, .rdata often contains the real validation material. Dumping the section revealed state-name bait, a 42-byte target array and eight pointers that looked like transition tables.

Checker Function

The checker around 0x140003220 handled argument checks, fake-flag logic and real validation. The key length check was 0x2a, meaning the correct input must be 42 characters.

$ objdump -d -M intel misk_automaton.exe > disasm.txt 140003232: 81 fb 69 7a 00 00 cmp ebx,0x7a69 140003238: 0f 84 c1 00 00 00 je 0x1400032ff 14000323e: 83 fb 02 cmp ebx,0x2 140003247: 48 8b 5e 08 mov rbx,QWORD PTR [rsi+0x8] 140003253: 48 83 f8 2a cmp rax,0x2a

The checker initializes two states and points into the target and table pointer array:

i = 0 state1 = 0x41 state2 = 3 table_ptrs = 0x1400050c0 target = 0x140005080

Automaton Logic

The program processes one input byte at a time. First it mixes the byte with the index and state1:

x = ((i * 9 + state1) & 0xff) ^ input[i]

Then it runs five transition rounds. The transition table depends on state2 and the round index.

for (j = 0; j < 5; j++) { table_index = (state2 + j) & 7; x = table[table_index][x]; state1 = (state2 * 19 + x + j + state1) & 0xff; state2 = (state2 * 5 + x + i + j) & 7; }

After the five rounds, the output byte is compared to the target:

if ((x ^ state1) != target[i]) { wrong(); }
Key insight: the checker is stateful, but the state is fully deterministic. If I know i, state1 and state2, I can brute force the next character and carry the new state forward.

Solver

The solver extracted the target bytes and eight transition tables from the PE file, then used DFS over a limited flag charset. The important part was converting virtual addresses to file offsets correctly.

$ python3 solve.py [+] target length: 42 [+] target: b575c490fc585248c3b2bf9071376aa0ee870aaab00363ced48379cd5fd077a7173e83e0ed2736b21ea7 [+] table pointers: 0x140005800 0x140005700 0x140005600 0x140005500 0x140005400 0x140005300 0x140005200 0x140005100 [+] extracted 8 tables [+] flag: 0xV01D{the_automaton_reads_between_states}
Solver idea: try each candidate character, run the same transition logic as the binary, compare against target[i], then recurse only when the byte matches. This avoids brute forcing the whole 42-character string at once.

Final Flag

The recovered flag is 42 characters long, matching the binary condition strlen(input) == 0x2a.

0xV01D{the_automaton_reads_between_states}

Lessons Learned

  • Never trust an obvious flag from strings in a hard reverse challenge.
  • Check where suspicious strings are referenced before submitting them.
  • PE virtual addresses must be converted to file offsets before extracting binary data in Python.
  • Lookup-table checkers can often be solved by modeling the exact state transition.
  • A stateful checker can still be solved character-by-character if the state update is deterministic.
Final conclusion: the real checker uses two evolving states, eight transition tables, a 42-byte target and five transition rounds per character. Once the transition is modeled accurately, the true flag is recovered.
